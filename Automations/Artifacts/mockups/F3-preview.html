Phase 3 — Canvas Editor

Visual workflow builder on top of P2's multi-node engine. P3 is gated on user demand per PRD §5.3 — meaning if no one explicitly asks for canvas editing, it might never ship. This mockup is the answer to "but what if we skip it?"

4. Canvas-driven create (F3-9)

Today the only way to create a workflow is chat. Canvas-driven create lets the user drag nodes onto a blank canvas and save — useful for visual-first thinkers who struggle to articulate the workflow in prose.

  • Start on `/workflows/new/canvas`. Empty canvas + palette.
  • Drag a trigger → drop on canvas. Right drawer fills in defaults.
  • Drag steps → drop, wire edges between them.
  • "Save" button validates the DAG, hits `workflows_create` with the constructed payload.
  • The same `WorkflowProposal` shape — just authored visually instead of via drafter LLM.

What you'd still have without P3

  • Chat-driven workflow creation (the F-16 path — already polished).
  • The expanded WorkflowCard detail view we just shipped (description, agent prompt, connections, settings).
  • All of P2's multi-node engine + new node kinds, just edited via chat or hand-rolled JSON.

What you'd miss

  • Visual editing — for users who think in graphs, not prose.
  • Live-run highlighting — watching a workflow execute step-by-step on the canvas.
  • Drag-drop creation — alternative entry point for "I don't know what I want, let me build it visually."
  • Three additional node kinds (transform / await_human_approval / fan_out) — though some of these could ship in P2 if needed.

Per the PRD's §5.3, P3 is meant to be gated on user demand precisely because chat-driven creation might already be enough. The mockup above is what you GAIN if you commit to it; skipping it costs you the visual surface but keeps the rest of the system intact.
