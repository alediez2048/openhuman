Phase 2 — Execution Depth

What you'd be missing if you defer P2 to do P4 first. Each section is one of the capabilities P2 ships that doesn't exist in Phase 1 today.

Note: P2 is also the hard prerequisite for P4's `BrowserAction` workflow node (multi-node infrastructure lives in F2-1). Skipping P2 isn't really an option; the choice is P2-then-P3-then-P4 vs P2-then-P4-then-skip-P3.

1. Multi-node workflows

Phase 1 today: exactly ONE `agent_prompt` node per workflow. Phase 2 adds chains.

⚡

New Stripe payment → Notion + Slack

• high

When a Stripe webhook fires, create a Notion page, wait 5 minutes, then notify Slack — unless the payment is below $10 (then just log it).

Trigger
Webhook · POST /workflows/stripe-payments

2a tool_call composio_execute · NOTION_CREATE_PAGE

↑ New: 4 node kinds (condition, tool_call, delay, channel_message, http_request), if/else branching, webhook trigger. Today's Phase 1 collapses all of this into one big prompt the agent has to interpret.

2. New node kinds catalog

Phase 2 ships 5 new node kinds beyond `agent_prompt`. The drafter learns each one + the canvas editor (P3) can drop them into workflows.

tool_call
Invoke any registered tool directly (composio_execute, memory_*, http_request) without an LLM step. Cheaper + deterministic.
http_request
First-class HTTP step with URL templating, secrets, JSON-path response extraction.
channel_message
Send to Slack / Telegram / Discord / WhatsApp via the F2-5 outbound layer. Replaces today's "Phase 2 deferred" stub.
condition
If/else branching on a jq-style expression over the previous step's output.
delay
Pause N seconds/minutes/hours between steps (e.g. wait an hour before follow-up).
await_human_approval
Pause the run + push a confirm prompt to the user. (Also in P3.)

3. await_human_approval — mid-run intervention

Workflows can pause and ask. Today Phase 1 runs are all-or-nothing.

5. New triggers

Phase 1 supports only cron + manual. Phase 2 adds three event-driven triggers.

webhook F2-9
"When my Stripe sends a payment.succeeded webhook, do X." Workflows fire on inbound HTTP. Built-in URL + auth + replay protection.
composio_event F2-10
"When a new Gmail arrives matching label X, do Y." Native Composio trigger subscriptions — no polling.
channel_message F2-11
"When I get a Telegram DM with 'help', escalate." Inbound from any connected channel becomes a trigger.

What you'd still have without P2

  • Single-step (`agent_prompt` only) cron + manual workflows. Today's capability.
  • Chat-driven create flow (F-16 fix is in).
  • Composio-routed everything (Slack, Gmail, Calendar, etc.).

What you'd genuinely miss

  • Workflows that chain > 1 step deterministically (today's agent has to "reason" through chains, which is slow + flaky).
  • Real outbound for Telegram-Bot / WhatsApp / LinkedIn / etc. (today stubs out).
  • If/else branching, delays, await_human_approval — all of which the user research said matters.
  • Event-driven workflows (webhook / composio_event / channel_message triggers).
  • Retries with backoff on flaky 3rd-party API calls.
