# F2-5 Primer: `channel_message` Node Kind

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F2-4 shipped `http_request`. Phase 0 shipped the unified channel mechanism.

---

## What Is This Ticket?

Send a message to a connected chat channel (Slack / Discord / Telegram / WhatsApp / iMessage). Reuses the existing channel-send tools registered in `tools::registry`.

### Why It Matters

- The most common Phase 2 chain shape: agent_prompt → channel_message ("summarise this, send to Slack").

---

## What This Ticket Must Accomplish

### Goal

- `ChannelMessageConfig`:
    ```rust
    pub struct ChannelMessageConfig {
        pub channel_ref: ConnectionRef,   // ConnectionRef::Channel { provider, channel_id }
        pub body_template: String,
    }
    ```
- Executor body:
    1. Substitute `body_template` against `NodeContext`.
    2. Look up the existing channel-send tool (e.g. `channel_send`) and invoke it with `{ ref: channel_ref, body: substituted_body }`.
    3. Return `NodeOutput { text: "sent" }`.
- Validator: `channel_ref` must be present + live in the `ConnectionsSnapshot`.

### Deliverables Checklist

#### A. Implementation

- [ ] `dispatch_node` arm calling the existing channel-send tool.
- [ ] Validator arm reusing the F-3 `ConnectionsSnapshot::is_connected` walk.

#### B. Tests

- [ ] Round-trip against a stub channel-send tool.
- [ ] Templating substitutes node output into the body.
- [ ] Missing channel connection → validation error at create time + health = NeedsConnections.
