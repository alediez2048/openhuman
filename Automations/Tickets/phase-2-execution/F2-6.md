# F2-6 Primer: `condition` Node Kind

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F2-2 shipped multi-node execution with `topological_sort`.

---

## What Is This Ticket?

Branch the workflow's execution path. The `condition` node evaluates a text-match predicate against its `NodeContext`; the executor takes one of two edges based on the result. **No UI for branching this ticket** — that's Phase 3 canvas. Phase 2 ships the executor + validator + JSON shape; the LLM emits these via the drafter when the user asks for "if X, do Y, else Z."

### Why It Matters

- The minimum branching primitive. Lets the drafting agent express "if subject contains URGENT, send to Slack; else queue for tomorrow."
- Phase 3 canvas inherits the same edge semantics — `condition` nodes become visible branches on the DAG renderer.

## What This Ticket Must Accomplish

### Goal

- `ConditionConfig`:
    ```rust
    pub struct ConditionConfig {
        pub left: String,          // template, substituted via NodeContext
        pub op: CompareOp,         // Eq / NotEq / Contains / Matches(regex)
        pub right: String,         // literal or template
        pub then_node_id: NodeId,
        pub else_node_id: Option<NodeId>,  // optional; missing = halt-on-false
    }
    ```
- Executor body:
    1. Substitute `left` + `right` against `NodeContext`.
    2. Evaluate the predicate per `op`.
    3. Emit a `NodeOutput { text: "true" / "false", routing_decision: NodeId }`.
- `execute_inner` (F2-2) honors the routing_decision: instead of advancing to the next topologically-sorted node, it jumps to `then_node_id` or `else_node_id`.
- Validator: `then_node_id` + `else_node_id` must reference nodes in the same workflow.
- DAG validation: with `condition` nodes, the graph can have multiple outbound edges per node. `topological_sort` permits multiple edges out; cycles still rejected.

### Deliverables Checklist

#### A. Implementation

- [ ] `CompareOp` enum + `ConditionConfig`.
- [ ] `dispatch_node` arm — returns a `NodeOutput` variant the executor recognises as a routing decision.
- [ ] `execute_inner` honors routing decisions.
- [ ] Validator: NodeId references resolve; cycles still rejected.

#### B. Tests

- [ ] `condition` with `op = Contains` and a templated `left` routes to `then_node_id`.
- [ ] `condition` with `op = Eq` on substituted values routes correctly per truthiness.
- [ ] Regex op: `Matches("^URGENT")` against a substituted body.
- [ ] Workflow with `else_node_id = None` halts cleanly on false (status = Succeeded; remaining nodes skipped).
- [ ] Dangling `then_node_id` → validator error.
