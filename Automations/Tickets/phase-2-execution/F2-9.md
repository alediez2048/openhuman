# F2-9 Primer: `webhook` Trigger — Inbound Bridge to External Platforms

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F2-2..F2-8 ship multi-node execution. `webhooks/` tunnel + `TunnelRegistration` enum already exist.

---

## What Is This Ticket?

The **inbound bridge to n8n / Zapier / IFTTT / Make** per ADR-005. Extends `webhooks::TunnelRegistration` with a `Workflow { workflow_id }` variant. HMAC-verified inbound POST → `executor::dispatch_run` with `TriggerSource::Webhook { payload }`.

### Why It Matters

- One trigger type unlocks the entire external-automation ecosystem inbound to OpenHuman. The user can wire any service that can POST a webhook to fire a workflow.
- The drafting agent's webhook escape hatch (ADR-013) emits these whenever a user describes a trigger source OpenHuman doesn't natively integrate with.

---

## What This Ticket Must Accomplish

### Goal

- `Trigger::Webhook { tunnel_uuid, target_path }` already declared in F-1.
- On `workflows_create` / `_enable` for a `Webhook` workflow:
    1. Register `TunnelRegistration::Workflow { workflow_id }` via `webhooks::tunnel::register`.
    2. Capture the generated tunnel URL + HMAC secret.
    3. Persist `target_path` + `tunnel_uuid` on the trigger config.
- On inbound POST:
    1. `webhooks/` HMAC-verifies the request.
    2. Looks up the matching `Workflow { workflow_id }` registration.
    3. Calls `executor::dispatch_run(config, workflow_id, TriggerSource::Webhook { payload })`.
    4. The webhook body becomes `NodeContext.trigger_payload` (F2-2 — already wired).
- On `workflows_delete` / `_disable`: deregister the tunnel.
- Boot-time reconcile: walk all enabled webhook workflows + ensure tunnels exist (matches F-7's scheduler reconcile pattern).
- The proposal preview's `setup_instructions` field carries the tunnel URL when the agent emits a webhook proposal — so the user knows what to paste into n8n / Zapier / IFTTT / Make.

### Deliverables Checklist

#### A. Implementation

- [ ] `webhooks/types.rs`: add `TunnelRegistration::Workflow { workflow_id }`.
- [ ] `webhooks/dispatch.rs`: route to `workflows::executor::dispatch_run` on a `Workflow`-registered tunnel.
- [ ] `workflows/ops.rs`: register/deregister tunnels on create/update/delete/enable/disable when trigger is Webhook.
- [ ] `workflows/scheduler.rs::reconcile_at_startup`: walk webhook workflows + restore registrations.
- [ ] Validator: `target_path` non-empty + matches `^[a-zA-Z0-9/_-]+$`.
- [ ] Proposer: when generating a Webhook trigger proposal, populate `setup_instructions` with the live tunnel URL + an n8n / Zapier / IFTTT / Make how-to paragraph.

#### B. Tests

- [ ] Round-trip: register webhook workflow → POST to tunnel URL with valid HMAC → run starts.
- [ ] HMAC failure → run does NOT start; tunnel returns 401.
- [ ] Deregister on delete.
- [ ] Boot reconcile restores webhook tunnels after a workspace reload.
- [ ] Proposal carries `setup_instructions` with the tunnel URL.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| HMAC required (NFR-2.2.4) | Phase 0 webhook precedent | One drop-in trigger covers every external platform inbound |
| Setup instructions embedded in proposal | ADR-013 webhook escape hatch | User sees the URL inline; no separate config screen |
| Tunnel lifecycle bound to workflow enable/disable | This ticket | Disabling stops accepting inbound POSTs immediately |
