## What Is This Ticket?

Pause execution for a wall-clock duration. Persists run state so a core restart resumes the delay correctly rather than losing the in-flight run.

### Why It Matters

- Unlocks scheduled-multi-step patterns: "draft now, wait 1 hour, send if no reply." Without persistent resume, a restart during the wait would lose the run â€” unacceptable for any real automation.

## What This Ticket Must Accomplish

### Goal

- `DelayConfig { duration_secs: u32 }` â€” capped at 24h.
- Executor body:
    1. Persist `workflow_runs.delayed_until = now + duration_secs` and `workflow_runs.delayed_step_id = current step id`.
    2. Release the in-flight slot so other workflows can run.
    3. Return to the run-loop's poll.
- New boot-time + periodic sweep `resume_delayed_runs`:
    - SELECT runs where `delayed_until <= now`.
    - For each: re-acquire the in-flight slot, advance past the delay step, continue from the next node.
    - Conflict (slot taken by a new dispatch) â†’ keep waiting; surface as a `WorkflowRunSkipped` with reason `delay_blocked`.
- F-9's `orphan_recovery_sweep` already catches dead runs â€” it must SKIP delayed runs (status = Delayed, not Running).
- New `RunStatus::Delayed` variant â€” recompute every prior status filter.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| Release in-flight slot during delay | F-9 single-flight | A long delay shouldn't lock the workflow out of new triggers |
| Persistent resume via DB row, not tokio::time::sleep | Forward-compat | Survives crashes; matches the "single source of truth = DB" pattern |
| Hard cap at 24h | This ticket | Avoids accidentally creating zombie workflows that wait forever |
