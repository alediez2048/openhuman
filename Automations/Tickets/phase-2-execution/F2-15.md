# F2-15 Primer: `active_hours` Enforcement on Cron Triggers

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F-7 declared `Trigger::Cron { active_hours: Option<ActiveHours> }` but ignored the field. Phase 1 deferred enforcement.

---

## What Is This Ticket?

The `ActiveHours { start, end }` field already exists on `Trigger::Cron`. F2-15 makes the scheduler actually respect it.

### Why It Matters

- The user expects "every 30 minutes during business hours" to mean weekdays 9–5, not 4 AM. Without enforcement, cron fires at every cron tick regardless of the time-of-day clamp.

---

## What This Ticket Must Accomplish

### Goal

- `ActiveHours { start: "HH:MM", end: "HH:MM" }` already declared in F-1.
- Scheduler tick (F-7) honors the field:
    - When firing a cron tick, evaluate `now.time()` in the trigger's `tz`.
    - If outside `[start, end]`, drop the tick + publish `WorkflowRunSkipped { reason: outside_active_hours }`.
- `next_run_for_schedule` (in `cron/schedule.rs`) is unchanged — it still returns the cron's next time. The drop check is at dispatch time, not at next-run computation.
- Validator: `start` and `end` are `HH:MM` 24-hour strings; `start < end` (no wraparound — for "10 PM through 2 AM" the user makes two workflows).
- Drafter prompt update (F2-13): tell the LLM the `active_hours` shape + when to use it.
- `WorkflowRunSkipped` already accepts a `reason_json` — add the new `outside_active_hours` discriminator to `SkippedReason`.

### Deliverables Checklist

#### A. Implementation

- [ ] Scheduler-tick dispatch arm checks `active_hours`.
- [ ] `SkippedReason::OutsideActiveHours { start, end }` variant.
- [ ] Validator arms.
- [ ] Update `phase_constraints_block(2)` to mention `active_hours`.

#### B. Tests

- [ ] Cron fires inside hours → run dispatches.
- [ ] Cron fires outside hours → run skipped + Skipped event published.
- [ ] Validator rejects `start >= end`.
- [ ] Validator rejects malformed time strings.
- [ ] Timezone handling: a `tz = "America/Los_Angeles"` workflow fires at 9 AM PT, not 9 AM UTC.
