# F2-2 Primer: Multi-Node Execution — Linear Chain Walk + Inter-Node Context

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F2-1 unlocked Phase 2 `NodeKind` variants in the validator + added `executor::dispatch_node` stubs.

---

## What Is This Ticket?

Extend Phase 1's single-node `execute_inner` (F-8) to walk multi-node linear chains. Lands the inter-node data-passing model that every Phase 2 node kind (F2-3..F2-7) consumes.

### Why It Matters

- F-8's executor hard-coded `workflow.nodes[0]` and the validator rejected `nodes.len() != 1` for Phase 1. F2-2 removes both constraints.
- Locks the inter-node context shape — `tool_call` (F2-3), `http_request` (F2-4), `channel_message` (F2-5), `condition` (F2-6), `delay` (F2-7) all consume it.

---

## What This Ticket Must Accomplish

### Goal

- Walk `workflow.nodes` in **edge order** (topological — Phase 2 is linear so this is `edges[0].from → edges[0].to → edges[1].to → ...`).
- For each node, build a `NodeContext` containing:
    - The trigger's payload (webhook body, composio event, channel message body).
    - Every prior node's `output`, keyed by `node.id`.
- Pass the `NodeContext` to `dispatch_node`. Per-node executors (F2-3..F2-7) use it for templating (OQ-7) — substitute `{{node.<id>.output.<jsonpath>}}` and `{{trigger.<jsonpath>}}` before invoking the underlying API.
- Persist per-node `output_json` (truncated to 64 KiB per NFR-2.3.5).
- Honor F2-8's `on_error` policy: `Halt` stops the run, `Continue` records the failed step + moves to the next node.

### Deliverables Checklist

#### A. Implementation

- [ ] `NodeContext` type in `executor.rs`:
    ```rust
    pub struct NodeContext {
        pub trigger_payload: serde_json::Value,
        pub outputs: HashMap<NodeId, serde_json::Value>,
    }
    ```
- [ ] `topological_sort(workflow.nodes, workflow.edges) -> Result<Vec<NodeId>>` — returns the execution order. Phase 2 is linear; the helper rejects cycles.
- [ ] `execute_inner` (rewrite) — walks the topologically-sorted node list, builds `NodeContext`, calls `dispatch_node`, persists per-step rows, advances or halts per `on_error`.
- [ ] `templating::substitute(context, raw_string) -> String` — small handlebars-style substitution for `{{trigger.<path>}}` and `{{node.<id>.output.<path>}}`. JSONPath via `serde_json_path` or a hand-rolled walker.
- [ ] Make F-15's `execute_inner` test stub work multi-node: install one stub per node kind during the test setup.

#### B. Tests

- [ ] `executor_tests`: 2-node chain — first node emits "hello"; second node's prompt templates `{{node.n1.output}}` and the substituted prompt makes it into the stub.
- [ ] `executor_tests`: 3-node chain with `on_error = Continue` on node 2 — node 2 fails, node 3 still runs, run status = `Failed` but all three step rows persist.
- [ ] `executor_tests`: cyclic edges rejected by `topological_sort` with a structured error.
