## What Is This Ticket?

Make the browser agent a first-class workflow node kind. Today a workflow has one `NodeKind::AgentPrompt` (Phase 1) or several nodes including `tool_call`/`http_request` etc. (Phase 2). F4-4 adds `NodeKind::BrowserAction`.

A `BrowserAction` node:
- Declares the target URL, an optional `BrowserProfile`, and the natural-language goal.
- The executor opens a `CdpSession` for the run, hands it to a sub-agent with the F4-3 tools in scope, and runs until the goal is complete or the iteration cap hits.
- Returns the structured output the next node consumes (e.g., the extracted balance flows into the downstream Slack-send node).

### Types added (Rust core)

```rust
// In src/openhuman/workflows/types.rs (extending the existing NodeKind enum):
pub enum NodeKind {
    AgentPrompt,
    BrowserAction,  // ← new
    // ... other Phase 2 variants
}

pub enum NodeConfig {
    AgentPrompt(AgentPromptConfig),
    BrowserAction(BrowserActionConfig),  // ← new
    // ...
}

pub struct BrowserActionConfig {
    /// Natural-language description of the goal the browser-agent must
    /// accomplish. E.g. "Log into example.com, navigate to /portfolio,
    /// extract the total balance, return it as a number."
    pub goal: String,

    /// Starting URL the session navigates to before the agent takes over.
    /// Optional — when None, the agent starts on `about:blank` and
    /// navigates as part of the goal.
    pub start_url: Option<String>,

    /// Which browser profile to use. Defaults to `EphemeralIsolated`
    /// for safety; workflows that need a persistent login set
    /// `NamedPersistent { name }`.
    pub profile: BrowserProfile,

    /// Maximum number of agent iterations. Defaults to 25 (higher than
    /// agent_prompt's default because browser tasks need more
    /// observe/act cycles).
    pub iteration_cap: u32,

    /// Domain whitelist. The agent's `browser_act("navigate to ...")`
    /// is rejected for hosts not in this list. Empty list = no
    /// restriction (use with care). F4-6's safety preamble layers
    /// on top.
    pub allowed_hosts: Vec<String>,

    /// Optional output schema. When set, the node's terminal
    /// step uses `browser_extract` with this schema as the final
    /// action, and the schema-validated value becomes the node's
    /// output. When None, the agent's text summary is the output.
    pub output_schema: Option<serde_json::Value>,
}
```

### Why It Matters

- Without F4-4, the browser tools from F4-3 can only be added to an `agent_prompt` node's allowlist via hand-rolled config — works for power users but not part of the drafter's vocabulary.
- A first-class node lets the workflow_propose_create drafter produce browser-agent workflows from chat input ("every Monday log into my brokerage..." → drafter emits a BrowserAction node).
- The Phase 3 canvas editor can render BrowserAction nodes distinctly (different icon, different config form) — natural extension of the canvas work.

#### C. Executor extensions

- [ ] `src/openhuman/workflows/executor.rs::execute_node` dispatches `NodeKind::BrowserAction` to a new `run_browser_action_node()` function.
- [ ] `run_browser_action_node(config, run_id, browser_action_config)`:
    1. Resolve / open the CdpSession via `SessionRegistry::open_or_attach`.
    2. If `start_url` is set, navigate the session there.
    3. Build a workflow_node agent (the same F-16 pattern) with `allowed_tools` extended by `browser_observe`, `browser_act`, `browser_extract`. Iteration cap = `browser_action_config.iteration_cap`.
    4. Set the agent's event context to `workflow:<run_id>` so F-16's tool-failure counter still applies.
    5. Inject the browser_action_config (goal + allowed_hosts + output_schema) into the agent's user-facing prompt with a structural preamble: "Your goal: <goal>. You must only navigate to hosts in: <list>. When done, your final response must be a JSON object matching this schema: <schema>."
    6. Call `agent.run_single(...)`. Capture the agent's final text response.
    7. If `output_schema` is set, parse + validate the final response as JSON.
    8. Close the session (or let `SessionRegistry`'s RAII handle it on run termination).
    9. Return `NodeOutput { text, tool_failure_count }` so F-16's honest-status gate still applies.

- [ ] Per-user concurrent-session cap (default 5). If a workflow tries to open a 6th session for the same user, queue or reject (default reject; configurable).

## Architectural Decisions

### Why a new node kind, not just an `agent_prompt` with browser tools?

Three reasons:

1. **Drafter clarity.** A dedicated `BrowserAction` kind tells the drafter "use this for UI-driven tasks" without entangling it with `agent_prompt`'s much broader semantics.
2. **Canvas editor.** Phase 3 canvas wants distinct node types with distinct icons + config forms. Cramming everything into `agent_prompt` makes Phase 3 painful.
3. **Safety.** `BrowserAction` carries `allowed_hosts` + `output_schema` as first-class fields — they aren't natural fits for a generic `AgentPromptConfig`. The validator can enforce browser-specific invariants cleanly.

### Why ephemeral-isolated as the default profile?

Safety. A workflow that fails to specify a profile shouldn't accidentally use the user's authenticated GitHub session. Default-ephemeral forces workflows that need auth to opt in explicitly + lets the validator cross-check against `allowed_connections`.

### Why `iteration_cap` defaults to 25, not 12?

Empirical: browser-agent tasks average 8–15 iterations per "real" workflow. 12 is too tight (F4-3's architectural rationale walks through a 10-step login-then-extract example). 25 gives comfortable headroom; the F-16 cost-bounded executor + F4-6 cost cap ensure runaway tasks still terminate.

### Why does the executor inject the goal/allowed_hosts/schema as a USER-message preamble, not extend the workflow_node SYSTEM prompt?

The workflow_node TOML system prompt is per-archetype and stable. Per-run goals are user-message content. Mixing them risks prompt-cache misses + makes the system prompt harder to reason about. The "preamble in user-message" pattern matches how the proposer.rs drafter packs instructions today.

### Where does the F-16 honest-status tool-failure counter apply?

Unchanged. The browser_* tools fire `ToolExecutionCompleted` events with `success: true|false` per call. A run that has a `browser_act("click submit")` returning `{ status: "failed", reason: "element not found" }` (success=true at the tool level but the inner status says failure) won't trigger F-16's counter. We need F4-6's audit layer to surface inner-status failures distinctly — that's why F4-6 also touches step-status accounting.
