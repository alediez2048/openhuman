## What Is This Ticket?

The browser agent has high blast radius — it can do anything the user can do in their authenticated browser. F4-6 is the layer that keeps that from going wrong:

1. **Safety preamble.** A system-prompt section the workflow_node ALWAYS sees when browser tools are in its allowlist. Codifies what the agent will and will not do.
2. **Dry-run mode.** Agent describes every action without executing it. Per-workflow setting.
3. **Cost caps.** Hard per-session token + LLM-call budgets. Trip → run terminates.
4. **Per-action confirmation thresholds.** Some action+domain combos auto-require user confirmation (e.g., form submit on `.gov`, post on social media).
5. **Audit log.** Every tool call persisted with timestamp + args + outcome. Replayable.
6. **Redaction policy.** Screenshots + DOM content sent to F4-5 / persisted by audit log run through a redaction pass.

### Why It Matters

Without F4-6 the browser agent is a footgun:

- LLM might fill credentials (instead of using the user's already-authenticated session).
- LLM might post to social media impersonating the user without confirmation.
- LLM might burn a $20 per-run token budget on a confused 50-iteration loop.
- LLM might persist a screenshot containing the user's SSN in plain text.

F4-6 is what makes Phase 4.1 safe to enable for non-power-users. Without it, F4 stays internal-only.

## What Was Already Done

- F-16's `workflow_node` agent has the `omit_safety_preamble = false` flag — the global safety preamble already applies. F4-6 ADDS to it, browser-specifically.
- F-16's event-bus tap counts `ToolExecutionCompleted{success:false}`. F4-6 extends this so inner-status failures (tool returns success but `{ status: "failed" }` in payload) also count.
- The cron + scheduler infrastructure has per-run cost tracking via the existing `tokenjuice` module. F4-6 adds browser-specific cost gates on top.
- The persistence layer (F-1) has `workflow_run_steps` with `output_json`. F4-6 adds a separate `browser_agent_audit_log` table keyed by run_id.

## What This Ticket Must Accomplish

### Goal

Browser-agent workflows can be enabled by non-power-users with confidence. Every concerning action requires explicit user confirmation; cost overruns terminate cleanly; audit log captures everything for post-hoc review.

### Deliverables Checklist

#### A. Safety preamble — appended to workflow_node prompt when browser tools are in scope

- [ ] `src/openhuman/agent/agents/workflow_node/browser_safety.md` — new prompt fragment.
- [ ] `workflow_node/prompt.rs` detects when the rendered prompt context includes any `browser_*` tool and appends `browser_safety.md` to the system prompt.

Content covers:

- **Never type credentials.** "If you see a login page, the user is already authenticated via the workflow's configured profile. If you find yourself NOT authenticated, STOP and emit a final answer of `{status: 'session_expired', message: 'User must re-authenticate before this workflow can run'}`. Do not attempt to type a username/password — there are no credentials available to you."
- **Never solve CAPTCHAs.** "If a CAPTCHA appears, STOP and report it."
- **Never post / submit / pay without explicit confirmation.** "Form submits on auth-sensitive pages (`signin`/`login`/`account`/`billing` URL segments) require the user to confirm via the live-preview panel before you proceed. The `browser_act` tool will return `{status: 'requires_confirmation'}` for these — wait for the user's confirmation result, don't retry."
- **Stay within allowed_hosts.** "Only navigate to hosts listed in the workflow's `allowed_hosts`. Reject the goal if it requires navigating elsewhere."
- **Single task per run.** "You are running ONE workflow node. Don't browse for fun; don't explore. Focus on the declared goal."

#### B. Dry-run mode

- [ ] `BrowserActionConfig.dry_run: bool` field added (default `false`).
- [ ] When `dry_run = true`, the browser_* tools change behavior:
    - `browser_observe` still returns real snapshots (read-only is fine).
    - `browser_act` does NOT dispatch the CDP primitive. Returns `{ status: "dry_run", would_have: <action_description> }`.
    - `browser_extract` runs normally (read-only).
- [ ] Workflow drafter (workflow_propose_create) gets a hint: workflows being created with `BrowserAction` nodes default `dry_run = true` for their first save. The user must edit the workflow to set `dry_run = false` after they've verified the dry-run output. This is the **F4-style training-wheels default**.
- [ ] WorkflowCard / WorkflowProposalPreview surfaces the dry-run state prominently — a "Dry run" pill on the workflow card so the user is never confused about what mode it's in.

#### C. Cost caps

- [ ] Per-session hard limits in `BrowserActionConfig`:
    - `max_iterations: u32` (already exists from F4-4)
    - `max_llm_tokens: u32` (new; default 50_000)
    - `max_screenshots: u32` (new; default 30)
    - `max_session_wall_clock_secs: u32` (new; default 600 — 10 minutes)
- [ ] `SessionRegistry` tracks these counters per session. F4-3's tool implementations increment.
- [ ] When any counter exceeds its cap, the session enters `terminated_by_cost_cap` state. The next tool call returns `{ status: "cost_cap_exceeded", which: "max_llm_tokens" }`. The workflow_node agent's safety preamble instructs it to terminate cleanly here.
- [ ] Counters are reset per workflow run (not cumulative across runs).

#### D. Per-action confirmation policy

- [ ] `src/openhuman/browser_agent/safety/confirmation.rs` — `ConfirmationPolicy` evaluator.
- [ ] Default policy (Phase 4.1):
    - Form submits on URL paths containing `signin|login|register|password|billing|payment|checkout|account|profile|settings` require confirmation.
    - Any POST/PUT/DELETE-like action (form submit, button-with-action) on hosts NOT in a user's explicit "trusted" list requires confirmation.
    - "Trusted hosts" are managed in user prefs (initially empty — F4-6 ships the data model; UI to manage lands later).
- [ ] When confirmation is required:
    - `browser_act` returns `{ status: "requires_confirmation", action: <description>, screenshot: <png_b64>, timeout: 120s }`.
    - F4-5's live preview shows a "Confirm" / "Reject" prompt with the captured screenshot.
    - User responds within 120s or the action is rejected.
    - The user's response feeds back via a new `DomainEvent::BrowserActionConfirmation { run_id, confirmed: bool }`.

## Architectural Decisions

### Why TWO layers of safety (preamble + tool-level)?

Belt + suspenders. The preamble shapes the LLM's intent — most violations get caught by the LLM itself refusing to type a password. The tool-level enforcement is the backstop for when the LLM ignores the preamble (which happens — model behavior is a probability distribution).

### Why default dry-run to TRUE for newly-drafted workflows?

Phase 4.1 is brand-new capability. Users won't know what to expect. Training-wheels default means the first run shows them what the agent WOULD do — they review, then disable dry-run. This is the same pattern shipping `cron_add` used (manual confirmation before first auto-run).

### Why a SEPARATE audit-log table, not just extend `workflow_run_steps.output_json`?

`workflow_run_steps` is one row per node; a browser run can have 20+ tool calls within a single node. Cramming them into output_json makes the schema messy and breaks the run-detail UI's structured rendering. A purpose-built audit table is the right shape — it's also queryable for support / debugging ("show me all browser actions that targeted .gov hosts in the last 30 days").

### Why redact passwords if "the user is already authenticated"?

Defense in depth. The preamble says don't type passwords; the redaction is for when the agent somehow ends up looking at one (a half-loaded session, a 2FA challenge with the prior password still in the box, etc.). Screenshots persisted on disk MUST NOT contain passwords — full stop.

### Why per-host confirmation policy as a hardcoded URL-pattern list rather than user-configurable from day one?

User-configurable from day one means most users never configure it and rely on a default they can't see. Hardcoded patterns are visible in code + auditable + can be overridden via "trusted hosts" (the opt-out, not opt-in, is the right default-bias). Per-host fine-grained policy is a Phase 4.2 thing if demand materializes.

### How does F4-6 interact with F-16's `tool_failure_count`?

F-16 increments on `ToolExecutionCompleted{success:false}` at the agent harness level. F4-6's tool implementations return `success: true` even on inner statuses like `{ status: "cost_cap_exceeded" }` — because the tool itself ran cleanly. To bridge: F4-6 ships a small extension to the workflow executor's step-status logic — also Failed when ANY browser_* tool result has an `inner_status` of `failed | cost_cap_exceeded | requires_confirmation_timeout | dry_run_summary`. This is a single ~20-line addition to `execute_agent_prompt_node`.
