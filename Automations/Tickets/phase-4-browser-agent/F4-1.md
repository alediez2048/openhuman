## What Is This Ticket?

Create a generic, non-provider-specific Rust module that wraps Chrome DevTools Protocol (CDP) calls into a stable internal API the rest of F4 builds on. Lives at `src/openhuman/browser_agent/cdp/`.

Today, OpenHuman has CDP code scattered across per-provider scanner modules (`app/src-tauri/src/telegram_scanner/`, `whatsapp_scanner/`, etc.). Each one re-implements click, type, navigation against its own session. F4-1 extracts the common shape into a reusable API.

### Surface this ticket exposes

```rust
pub struct CdpSession {
    target_id: TargetId,
    transport: Arc<dyn CdpTransport>,
}

impl CdpSession {
    pub async fn navigate(&self, url: &str) -> Result<()>;
    pub async fn wait_for_load(&self, opts: WaitOptions) -> Result<()>;
    pub async fn screenshot(&self, opts: ScreenshotOptions) -> Result<Vec<u8>>;
    pub async fn click_at(&self, x: f64, y: f64, button: MouseButton) -> Result<()>;
    pub async fn type_text(&self, text: &str, opts: TypeOptions) -> Result<()>;
    pub async fn press_key(&self, key: Key, modifiers: KeyModifiers) -> Result<()>;
    pub async fn scroll(&self, dx: f64, dy: f64) -> Result<()>;
    pub async fn evaluate(&self, expr: &str) -> Result<serde_json::Value>;
    pub async fn url(&self) -> Result<String>;
    pub async fn title(&self) -> Result<String>;
    pub async fn cookies(&self) -> Result<Vec<Cookie>>;
    pub async fn close(self) -> Result<()>;
}
```

Plus session-construction surface:

```rust
pub async fn open_session_for_user(
    user_id: &UserId,
    profile: BrowserProfile,
) -> Result<CdpSession>;
pub async fn attach_to_existing_target(target_id: TargetId) -> Result<CdpSession>;
```

`BrowserProfile` enum captures whether to reuse an existing authenticated session, isolate to a fresh ephemeral one, or attach to a specific webview (`webview_account_*`).

### Why It Matters

- F4-2 needs CDP screenshots + DOM-snapshot calls. F4-2 should not re-invent the transport.
- F4-3's `browser_act` translates LLM intent → low-level primitives. Without F4-1, F4-3 has to call `chrome-devtools-protocol` directly and that surface is huge and unstable.
- The per-provider scanners (gmail, slack, telegram, etc.) can eventually migrate to this layer as cleanup — but **NOT in F4-1**. F4-1 ships a new surface; existing scanners stay as-is. Cleanup tickets land later.

## What This Ticket Must Accomplish

### Goal

A reusable `browser_agent::cdp::CdpSession` type that can drive any CEF target with the listed surface. Stable enough that F4-2 and F4-3 can be written against it.

### Deliverables Checklist

#### A. Module scaffolding

- [ ] `src/openhuman/browser_agent/mod.rs` with module documentation explaining the F4 architecture (point at F4-overview).
- [ ] `src/openhuman/browser_agent/cdp/mod.rs` with the public types.
- [ ] `src/openhuman/browser_agent/cdp/session.rs` — `CdpSession` impl.
- [ ] `src/openhuman/browser_agent/cdp/transport.rs` — `CdpTransport` trait + an impl backed by the existing CDP client crate.
- [ ] `src/openhuman/browser_agent/cdp/types.rs` — `BrowserProfile`, `MouseButton`, `Key`, `KeyModifiers`, `WaitOptions`, `ScreenshotOptions`, `TypeOptions`, `Cookie`.
- [ ] `src/openhuman/browser_agent/cdp/errors.rs` — `CdpError` enum (NavigationFailed / Timeout / TargetClosed / EvaluationError / TransportClosed / Other).
- [ ] Lib entry — `src/lib.rs` or wherever — adds `pub mod browser_agent;` so the rest of the codebase can use it.

#### B. Primitives (each with at least one unit test against a mock transport)

- [ ] `navigate(url)` — fires `Page.navigate`; resolves once `Page.frameStoppedLoading` arrives.
- [ ] `wait_for_load(opts)` — opts: `until: DomContentLoaded | NetworkIdle | Custom(predicate_fn)`, `timeout: Duration`. Default `NetworkIdle`, 30s.
- [ ] `screenshot(opts)` — opts: `format: Png | Jpeg`, `quality`, `clip: Option<Rect>`, `full_page: bool`. Returns raw bytes.
- [ ] `click_at(x, y, button)` — `Input.dispatchMouseEvent` `mousePressed` + `mouseReleased` pair, with proper `clickCount`. Default `MouseButton::Left`.
- [ ] `type_text(text, opts)` — `Input.insertText` for the easy case + per-character `Input.dispatchKeyEvent` when `opts.humanized_delay_ms > 0`. Default humanized 30–80ms per char (Phase 4.1 anti-bot mitigation).
- [ ] `press_key(key, mods)` — `Input.dispatchKeyEvent` with proper keyCode / code / modifiers. Cover at minimum: Enter, Tab, Escape, ArrowKeys, Backspace, plus letters with modifiers (Cmd+A, Cmd+C).
- [ ] `scroll(dx, dy)` — `Input.dispatchMouseEvent` with `type=mouseWheel`, deltaX/deltaY.
- [ ] `evaluate(expr)` — `Runtime.evaluate` with `awaitPromise=true` and `returnByValue=true`. **Caveat:** this is the only primitive that runs JS in the page. Per CLAUDE.md's CEF JS-injection ban, this is allowed for **agent-driven** evaluation (specific scoped queries), NOT for persistent injection. The tool docstring + safety preamble must reinforce this distinction.
- [ ] `url()`, `title()`, `cookies()` — straightforward `Page.getNavigationHistory` / `Network.getCookies`.
- [ ] `close()` — closes the target. RAII Drop impl as a safety net.

#### C. Session construction

- [ ] `open_session_for_user(user_id, profile)`:
    - `BrowserProfile::ReuseAuthenticated { provider }` → attaches to the user's existing `webview_account_<provider>` target if one exists; errors with `NoAuthenticatedSession { provider }` otherwise.
    - `BrowserProfile::EphemeralIsolated` → spawns a fresh CEF target with a temporary profile, disposed on `close`.
    - `BrowserProfile::NamedPersistent { name }` → reuses or creates a named profile that persists across runs (for workflows that need login on first run + remember it after).
- [ ] `attach_to_existing_target(target_id)` for low-level escape-hatch tests.
- [ ] Per-user target isolation: a session opened for `user_id_A` cannot be attached-to by an agent running for `user_id_B`. Enforced at construction.

#### D. Logging + observability

- [ ] Every primitive call logs at `tracing::debug!(target: "browser-agent-cdp")` with the target_id + arguments.
- [ ] Errors log at `warn!` with the CDP error code + stage.
- [ ] A counter histogram (via the existing `observability` module if available) tracks per-primitive latency. Used by F4-6's cost-cap layer.

#### E. Tests

- [ ] Unit: every primitive against a mock `CdpTransport` (records the CDP messages sent; the test asserts the message shape).
- [ ] Integration: open a real CEF target pointed at a local static HTML test page (`tests/fixtures/browser_agent/basic.html`), navigate, click a button, type into an input, screenshot, verify pixel hash of a known region.
- [ ] Regression: `wait_for_load(NetworkIdle)` actually waits and times out cleanly; doesn't return early on a stuck load.
- [ ] Regression: opening a session for user A then trying to attach from user B fails with `PermissionDenied`.

## Architectural Decisions

### Why a fresh module, not reuse `webview_accounts`?

`webview_accounts` is provider-aware (knows about Telegram / WhatsApp / etc.) and login-detection-focused. F4-1 is provider-agnostic and action-focused. Cleaner to layer rather than merge — F4-1 can call into `webview_accounts::cef_target_for_provider(...)` to find the right target, but the CDP wrapping lives in F4-1.

### Why `CdpTransport` trait?

Two reasons. First, testability — the mock impl is how we unit-test every primitive without a real Chromium. Second, future flexibility — Phase 4.2's Playwright runtime is just a different `CdpTransport` impl. F4-1 doesn't need to ship the Playwright impl, but the architecture must be ready.

### Why humanized typing by default?

Anti-bot fingerprinting is a real problem. Many sites detect instant text insertion as bot-like and serve CAPTCHAs. 30–80ms per char is the well-documented sweet spot from browser-use's research. Costs us a few seconds per form but avoids being flagged. **NOT a CAPTCHA-defeat measure** — just enough realism for honest UI flows.

### Why expose `evaluate` at all if CLAUDE.md bans JS injection?

CLAUDE.md's ban is on **persistent** injection (`addScriptToEvaluateOnNewDocument`, init scripts that run on every page load). One-off `Runtime.evaluate` queries scoped to a specific agent task are a different thing — they're how the agent introspects state (e.g., "what's the value of this React-controlled input?"). The safety preamble in F4-6 will codify that `evaluate` is for read-only introspection, not for installing handlers. The tool docstring on F4-3's `browser_act` will hide this primitive — agents call high-level tools, not raw `evaluate`.
