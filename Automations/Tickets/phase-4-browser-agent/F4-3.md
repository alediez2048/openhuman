## What Is This Ticket?

Wrap F4-1 + F4-2 in three LLM-facing tools modeled after Stagehand's `act` / `extract` / `observe` API. These are the tools the workflow_node sub-agent calls; everything below them is internal plumbing.

### The three tools (registered in `src/openhuman/tools/impl/browser_agent/`)

```rust
// "What's on the page right now?" — returns a PageSnapshot rendered for the LLM.
pub struct BrowserObserveTool;
impl Tool for BrowserObserveTool {
    fn name() -> &str { "browser_observe" }
    fn parameters_schema() -> Value {
        json!({
            "type": "object",
            "properties": {
                "detail": { "enum": ["compact", "standard", "verbose"], "default": "standard" }
            }
        })
    }
    // Returns the rendered text snapshot in markdown_formatted.
}

// "Do this thing on the page." — natural-language action, translated to CDP primitives.
pub struct BrowserActTool;
impl Tool for BrowserActTool {
    fn name() -> &str { "browser_act" }
    fn parameters_schema() -> Value {
        json!({
            "type": "object",
            "properties": {
                "action": {
                    "type": "string",
                    "description": "Natural-language action like \"click the Save button\", \"type 'hello' into the search field\", \"scroll down 500px\", \"go to https://example.com\"."
                },
                "element_id": {
                    "type": "integer",
                    "description": "Optional: target element id from the most recent browser_observe snapshot. Use when you already know which element."
                }
            },
            "required": ["action"]
        })
    }
    // Returns: { status, observed_change: { url, title, summary } } so the LLM can chain.
}

// "Pull these specific values out of the page."
pub struct BrowserExtractTool;
impl Tool for BrowserExtractTool {
    fn name() -> &str { "browser_extract" }
    fn parameters_schema() -> Value {
        json!({
            "type": "object",
            "properties": {
                "instruction": {
                    "type": "string",
                    "description": "Natural-language extraction instruction, e.g. \"get the total balance as a number\" or \"list all event titles from the calendar table\"."
                },
                "schema": {
                    "type": "object",
                    "description": "Optional JSON schema the extracted value must conform to. When omitted, returns a free-form string."
                }
            },
            "required": ["instruction"]
        })
    }
    // Returns: { extracted: <typed value>, evidence: <which snapshot elements were used> }
}
```

### Why It Matters

- F4-1 and F4-2 are infrastructure. F4-3 is the surface workflows and the chat agent see.
- The Stagehand 3-primitive API is well-validated — it's the smallest surface that covers 90% of browser-agent use cases. Smaller surfaces = easier for LLMs to use correctly, easier to write safety constraints around.
- After F4-3 lands, a power user could already use the browser agent inside a workflow's `agent_prompt` node — no UI yet, no safety guards, but the core capability works.

## What Was Already Done

- [`F4-1`](./F4-1.md) provides `CdpSession` with `click_at`, `type_text`, `scroll`, `navigate`, `evaluate`, etc.
- [`F4-2`](./F4-2.md) provides `snapshot(session)` returning a `PageSnapshot` with `ActionableElement` ids.
- The workflow_node sub-agent (F-16) accepts per-instance `allowed_tools` overrides. F4-3's tools register in `tools::ops::all_tools_with_runtime` so they're available; they get added to `build_node_agent_definition`'s output when a workflow declares a browser-capable connection. F4-4 handles the connection-type wiring.
- Per-call session lifecycle: each tool resolves the **current session for this workflow run** from a per-run registry (`SessionRegistry`). New ticket-specific detail: F4-3 ships the `SessionRegistry` because nothing earlier needed it.

## What This Ticket Must Accomplish

### Goal

Three working tools with deterministic NL→primitive translation. End-to-end: a workflow whose `agent_prompt.prompt` says "Go to example.com, click Login, type my username, take a screenshot of the dashboard, and return the total balance" should complete in <60 seconds with all four tool calls succeeding.

### Deliverables Checklist

#### A. SessionRegistry — session lifecycle per workflow run

- [ ] `src/openhuman/browser_agent/registry.rs` — `SessionRegistry` keyed by `(user_id, run_id)`.
- [ ] `SessionRegistry::open_or_attach(user_id, run_id, profile) -> Arc<CdpSession>` — opens a fresh session OR returns the cached one for this `(user_id, run_id)`. The session lives until the workflow run terminates.
- [ ] RAII teardown: the `RunId` going terminal triggers `close()` on the session. Lives in the workflow executor — F4-4's executor integration wires this.
- [ ] Thread-safe (sessions are mutated across tool calls).

#### B. BrowserObserveTool

- [ ] `src/openhuman/tools/impl/browser_agent/observe.rs`.
- [ ] On execute: resolve session from registry (using event_context tag for the workflow run), call `snapshot()`, render with the requested `detail` tier, return `success_with_markdown(json_payload, snapshot.to_llm_text(tier))`.
- [ ] If no session exists yet (first call in the run), open one with `BrowserProfile::EphemeralIsolated` by default. The workflow's declared connections (F4-4) override this.
- [ ] `supports_markdown = true` so the snapshot text reaches the LLM as the primary payload.

#### C. BrowserActTool — the natural-language → CDP translator

- [ ] `src/openhuman/tools/impl/browser_agent/act.rs`.
- [ ] On execute:
    1. Resolve session.
    2. If a snapshot from this run isn't cached (or is >30s old, or the URL changed since the last snapshot), take a fresh one.
    3. Translate `action` + optional `element_id` into a primitive call. Translation logic:
        - **`element_id` provided** → fast path: look up the element in the cached snapshot, dispatch the implied verb (click/type/select based on `role`).
        - **No `element_id`, natural language only** → call a small "intent" LLM (the same `chat_with_system` path the workflow drafter uses, model = "fast" tier) with a tight system prompt: `"Given this page snapshot and the user's action request, output exactly one JSON tool call from this set: [click, type, scroll, navigate, press_key]. Be conservative. If ambiguous, return { error: 'ambiguous', candidates: [N, M] }."` Pass the action + the snapshot's compact tier as context. Parse the structured response.
    4. Dispatch the resolved primitive via F4-1's `CdpSession`.
    5. After the action: wait for network-idle OR 2s timeout, then take a fresh snapshot. Return: `{ status: "ok", observed_change: { url_changed: bool, title: string, summary: "Page changed from X to Y" } }`.
- [ ] On translation failure: return `success_with_markdown({ status: "ambiguous", message: "Couldn't resolve action — was it 'click [1] (Login button)' or 'click [4] (Login link)'?" }, ...)`. The agent can then call again with an explicit `element_id`.
- [ ] On dispatch failure (CDP error): return `success_with_markdown({ status: "failed", reason: "..." }, ...)`. The tool result `success` flag stays true (the tool ran cleanly); the harness's `ToolExecutionCompleted` event will reflect the inner-status via F4-6's audit layer.

#### D. BrowserExtractTool

- [ ] `src/openhuman/tools/impl/browser_agent/extract.rs`.
- [ ] On execute:
    1. Resolve session, snapshot.
    2. Render the snapshot's `text_content` + relevant element labels.
    3. Call the same "fast" LLM with: `"Given this page content and the extraction instruction, return a JSON value matching the schema (or a free-form string if no schema)."` Use OpenHuman's structured-output mechanism (same pattern as the workflow drafter's fenced-JSON pattern).
    4. Validate against `schema` if provided.
    5. Return: `{ extracted: <value>, evidence: [<element_ids_used>] }`.
- [ ] Same `success_with_markdown` pattern as `act`.

#### E. Tool registration

- [ ] All three tools register in `src/openhuman/tools/ops.rs::all_tools_with_runtime`.
- [ ] **Crucially:** they are NOT added to the orchestrator's `[tools].named` allowlist by default — they're workflow-only in Phase 4.1 (chat-tier exposure lands in Phase 4.2 with a feature flag). The `workflow_node` archetype's `build_node_agent_definition` adds them when a `ConnectionRef::Browser` is in `allowed_connections` (F4-4 wires this).

#### F. Tests

- [ ] Unit: BrowserObserveTool returns the rendered snapshot text for the right detail tier. Mock the session.
- [ ] Unit: BrowserActTool with explicit `element_id` dispatches the correct primitive. Mock session + assert primitive call.
- [ ] Unit: BrowserActTool with NL action calls the intent-LLM exactly once and dispatches the parsed result. Mock both session and LLM call.
- [ ] Unit: BrowserExtractTool with a JSON schema returns a validated value.
- [ ] Integration: full loop — register tools, build a workflow_node agent with browser_* in allowed_tools (use F-16's `from_config_for_agent_with_tool_override`), prompt it to "go to file:///tests/fixtures/browser_agent/forms.html and extract the value of the input named 'email'". Assert the agent: (a) called browser_act("navigate to ..."), (b) called browser_extract, (c) returned the right string.

## Architectural Decisions

### Why a separate "intent LLM" inside `browser_act`?

The workflow_node agent's LLM emits high-level natural language ("click the Save button"). Translating that to a specific CDP click takes a tight, scoped LLM call with the snapshot as context. Doing the translation inside the orchestrating LLM directly would force it to emit JSON-shaped tool calls referencing element ids, which:

1. Couples the workflow_node prompt to F4-2's snapshot format.
2. Bloats the workflow_node's context window with the entire snapshot per turn.
3. Loses the Stagehand-style ergonomics (the whole point of these tools).

The intent LLM is a 200–500-token call to the "fast" model tier per action. Cost-bounded. Test-mockable.

### Why not just have `browser_act("click button N from the snapshot")` and skip the intent LLM?

That's actually the **fast path** (when `element_id` is provided). The intent LLM only runs when the LLM emits NL without an id. We support both because:

- Fast path is cheaper and more deterministic — preferred when the LLM already saw the snapshot.
- NL path is more forgiving — the agent doesn't need to memorize snapshot ids across turns; can say "click the login button" and trust the tool to figure it out.

### Why no `browser_navigate` tool?

`browser_act("navigate to https://...")` covers it via the NL translator. Adding a separate tool would clutter the LLM's tool list with no benefit; the action-routing is what `browser_act` exists for.

### What about `browser_close_session`?

Not exposed as an LLM tool. Sessions auto-close when the workflow run terminates (F4-4 wires the executor hook). Exposing close to the LLM would create footguns (agent closes the session mid-task, can't recover).

### How does this interact with workflow_node's `iteration_cap`?

Each tool call is one iteration. A typical browser-agent task uses 5–15 iterations:

1. `browser_observe` — see the landing page
2. `browser_act("click login")`
3. `browser_observe` — see the login form
4. `browser_act("type email")`
5. `browser_act("type password")` — F4-6 NEVER allows this; agent stops here unless session already authenticated
6. `browser_act("click submit")`
7. `browser_observe` — verify logged in
8. `browser_act("navigate to /portfolio")`
9. `browser_extract("balance as number")`
10. Final summary.

The F-16 fix that made `iteration_cap` actually load-bearing matters here. The default cap of 12 is tight for browser tasks. F4-4 should set the default iteration_cap for `BrowserAction` nodes to **25** to give breathing room — the workflow_node TOML's max_iterations stays at 20 as the floor, but specific browser-action nodes opt for higher.

### Why `success_with_markdown` for all three tools?

Same reason as F-16's propose tools: markdown_formatted is forwarded to the LLM when `prefer_markdown_tool_output=true`, but the JSON payload is what the LLM sees when it's off (the default). Putting the structured result in both ensures the LLM gets the info regardless of config. Avoids the "tool returns empty to the LLM" bug F-16 hit.

## Acceptance criteria

- `cargo test --lib browser_agent::tools` passes.
- `cargo test --test browser_agent_e2e` passes — full workflow_node loop driving a local HTML test page.
- Tools appear in the global registry via `ops::all_tools_with_runtime` but NOT in the orchestrator's `[tools].named`.
- A workflow with a manually-edited `allowed_tools = ["browser_observe", "browser_act", "browser_extract"]` runs end-to-end against a real local page in <30 seconds.
- Tracing logs show the intent-LLM round-trip distinctly from the F4-1 CDP primitive call.
