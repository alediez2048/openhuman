## What Is This Ticket?

Build the layer that turns a live page into an LLM-readable structured representation. Output is a `PageSnapshot` — compact enough to fit in a system prompt, rich enough that the LLM can pick which element to click without screenshots in 90%+ of cases.

Inspired by `browser-use/browser-use`'s DOM-extraction algorithm. Phase 4.1 is DOM + accessibility-tree only; vision-grounded fallback lives in [`F4-7`](./F4-7.md).

### Surface this ticket exposes

```rust
pub struct PageSnapshot {
    pub url: String,
    pub title: String,
    pub viewport: Viewport,
    pub elements: Vec<ActionableElement>,
    pub text_content: String,             // <main>-like region, truncated
    pub timestamp: chrono::DateTime<Utc>,
    pub snapshot_token_estimate: usize,   // pre-computed budget signal
}

pub struct ActionableElement {
    pub id: u32,                          // stable within this snapshot only
    pub role: ElementRole,                // button, link, input, select, ...
    pub label: String,                    // accessible name + visible text
    pub state: ElementState,              // enabled/disabled, checked, expanded, …
    pub bounds: Rect,                     // for vision fallback / click coords
    pub xpath: String,                    // for re-finding after redraws
    pub attributes: HashMap<String, String>, // narrow set: href, value, placeholder, aria-*
}

pub async fn snapshot(session: &CdpSession, opts: SnapshotOptions) -> Result<PageSnapshot>;
```

LLM-facing format (rendered by F4-3) looks like:

```text
URL: https://app.brokerage.com/portfolio
Title: Portfolio · Example Brokerage

[1] link "Dashboard" — /dashboard
[2] link "Portfolio" (current page) — /portfolio
[3] button "Refresh"
[4] heading "Total balance"
[5] text "$142,381.22"  (the balance number)
[6] table "Holdings" — 24 rows
[7] button "Add holding"
...
```

The LLM addresses elements by `[N]`. `browser_act("click [3]")` resolves to a CDP click on element id 3's bounds.

### Why It Matters

- The single biggest determinant of browser-agent quality is grounding: how well the LLM can pick the right element to interact with. Bad grounding = browser-use throws hands; good grounding = clean execution.
- Tokens-per-snapshot is the cost driver. A naive `document.outerHTML` snapshot for a real-world page is 50k–500k tokens and useless. The skill is reducing to ~200–2000 tokens of structured, scannable summary.
- F4-3's `browser_observe` tool returns a `PageSnapshot`. F4-3's `browser_act("click the Save button")` resolves "Save button" by searching the same snapshot's elements. Without F4-2, both tools are impossible.

## What Was Already Done

- F4-1 CDP primitives (`navigate`, `screenshot`, `evaluate`, `click_at`, etc.). F4-2 uses `evaluate` heavily to extract DOM state in one shot rather than dozens of CDP round-trips.
- `chrome-devtools-protocol` provides `DOM.getDocument`, `DOM.getBoxModel`, `Accessibility.getFullAXTree`, `DOM.querySelectorAll`. The grounding work is mostly composition + filtering of these.

### Reference reading before starting

- [`browser-use/browser-use/src/dom/`](https://github.com/browser-use/browser-use/tree/main/browser_use/dom) — the canonical DOM-extraction algorithm. Read `_view.py` and `service.py`. Port the **algorithm**, not the code; it's Python.
- [`browserbase/stagehand`](https://github.com/browserbase/stagehand)'s extract API for the structured-extraction shape (used by F4-3, but it's natural to think about it here).
- Chrome's accessibility-tree docs — what `role` and `name` mean in WAI-ARIA terms.

## What This Ticket Must Accomplish

### Goal

A `snapshot(session)` call that produces a `PageSnapshot` in <2 seconds for a typical page, fits in <3000 tokens of LLM context for 90%+ of pages, and is stable enough that calling `snapshot()` twice in a row on an unchanging page produces structurally-equivalent results.

### Deliverables Checklist

#### A. Module structure

- [ ] `src/openhuman/browser_agent/perceive/mod.rs` — public surface.
- [ ] `src/openhuman/browser_agent/perceive/snapshot.rs` — main `snapshot()` entry + `PageSnapshot` type.
- [ ] `src/openhuman/browser_agent/perceive/elements.rs` — `ActionableElement` extraction + filtering.
- [ ] `src/openhuman/browser_agent/perceive/render.rs` — `PageSnapshot::to_llm_text()` (the `[N] role "label" — attrs` format).
- [ ] `src/openhuman/browser_agent/perceive/tests.rs`.

#### B. DOM extraction (the meat of the ticket)

- [ ] `extract_actionable_elements()` runs a single `evaluate()` of an embedded JS snippet that returns the actionable subset of the DOM as JSON. The snippet:
    - Walks the document with `document.querySelectorAll('a, button, input, select, textarea, [role], [onclick], [tabindex]:not([tabindex="-1"])')`.
    - For each match, computes: `boundingClientRect`, computed `display !== "none"` and `visibility !== "hidden"`, `disabled`/`aria-disabled`, `aria-label || textContent || placeholder || value`, `xpath` (own helper).
    - Filters: elements off-screen, elements with zero area, elements occluded by ancestors with `pointer-events: none`.
    - Returns JSON array with the extracted attrs.
- [ ] The embedded JS snippet lives in `perceive/dom_extractor.js` and is `include_str!`'d. Size budget: <8 KB minified.
- [ ] Rust side parses the JSON, builds `ActionableElement` Vec, assigns sequential `id`s.

#### C. Accessibility-tree augmentation

- [ ] `merge_a11y_tree()` calls `Accessibility.getFullAXTree`, walks it, and:
    - Promotes `role` to the more-accurate WAI-ARIA value when the DOM extractor's role guess was generic ("div" → "button" via `role="button"`).
    - Pulls in `aria-label` / `aria-describedby` text the DOM walker missed.
    - Flags elements as `ElementState::Disabled` based on `aria-disabled` even if no native `disabled` attr.
- [ ] If `getFullAXTree` fails (Chrome version mismatch, internal page), fall back to DOM-only and log a warn.

#### D. Token-aware budget

- [ ] `PageSnapshot::to_llm_text(budget_tokens: usize)` renders the snapshot with three quality tiers:
    - **Compact** (<500 tokens) — top 30 actionable elements + page title + URL only. Used when the snapshot fits comfortably.
    - **Standard** (500–2000 tokens) — all actionable elements + a short `text_content` excerpt. Default tier.
    - **Verbose** (2000–5000 tokens) — adds `attributes` per element + larger `text_content`. Used when LLM explicitly asks for more detail via `browser_observe(detail="verbose")`.
- [ ] Token estimation uses a simple `chars / 4` heuristic — accurate enough for budget enforcement.
- [ ] Hard cap: if the standard tier exceeds 5000 tokens after filtering, drop low-priority elements (form inputs deep in collapsed sections, links in footer/header navs). Log a warn so we know the truncation happened.

#### E. Stability + caching

- [ ] `snapshot()` does NOT cache by default — the LLM may snapshot multiple times during a task and expect fresh state.
- [ ] But: `ActionableElement.xpath` MUST be stable across snapshots of an unchanging page (same id in snapshot N and N+1 → same xpath). This is what lets F4-3 re-find an element after a redraw.
- [ ] Element `id`s are NOT stable across snapshots — they're sequential per snapshot. The LLM must use them within the response it generated for that snapshot.

#### F. Tests

- [ ] Unit: feed in a synthetic JSON payload from the JS extractor, assert `PageSnapshot` shape (10+ cases covering buttons / inputs / disabled / hidden / nested).
- [ ] Unit: token-budget enforcement — assert `to_llm_text(500)` drops elements correctly.
- [ ] Integration: snapshot a real `tests/fixtures/browser_agent/forms.html` page (covers form inputs, checkboxes, radios, select, textarea), assert the structured output matches a snapshot file.
- [ ] Integration: snapshot a deliberately complex SPA-like page (`tests/fixtures/browser_agent/spa_dashboard.html` — uses React-style nested divs + ARIA roles), assert critical elements (5 buttons, 1 table heading) are extracted.
- [ ] Regression: snapshot before + after a JS-driven re-render (button click adds a row to a table); assert the new row's button is in the new snapshot with a stable xpath suffix.

## Architectural Decisions

### Why JS injection via `evaluate`, not pure CDP?

CDP's DOM API (`DOM.getDocument` + `DOM.querySelectorAll`) requires dozens of round-trips for a real page. A single `Runtime.evaluate` call returning JSON is 10–50× faster. CLAUDE.md's CEF JS-injection ban applies to **persistent** initialization scripts on third-party origins — one-off agent-driven `evaluate` is allowed (see F4-1's rationale). The extractor JS is small, scoped to the active session, and never persists.

### Why DOM + a11y tree both, not one or the other?

- DOM alone misses semantics added by ARIA (`role="button"` on a `<div>`).
- A11y tree alone misses raw DOM attributes (`href`, `value`) that are useful for the LLM's reasoning.
- Stagehand and browser-use both converged on hybrid for the same reasons.

### Why xpath, not CSS selectors?

- xpath uniquely identifies a node by structural position. CSS selectors break the moment a sibling is added.
- F4-3 re-finds elements after page changes by xpath; CSS selectors would force re-snapshot more often.
- xpath generation is a 20-line JS helper. Tiny cost.

### Why pre-render `text_content` even though the LLM could ask?

The LLM often needs page text for context ("what is the headline?"). Including a trimmed `text_content` saves a round-trip for the common case. Budget is bounded by the token-tier system.

### What about iframes?

- Phase 4.1: iframes are NOT recursed into. The extractor stops at iframe boundaries and flags them as `ElementRole::IframePresent { src }` so the LLM knows they exist.
- Phase 4.2 or later: cross-frame snapshotting. Punted because (a) cross-origin iframes have CSP / CDP target issues, (b) most user-visible flows don't depend on iframe interaction, and (c) handling it well is several days of work on its own.
