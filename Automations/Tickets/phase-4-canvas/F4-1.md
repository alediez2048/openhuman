# F4-1 Primer: Canvas Scaffold — @xyflow/react Integration + Read-Only Render

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** Phase 1 + Phase 2 complete.

---

## What Is This Ticket?

Stand up `@xyflow/react` and ship a read-only visual render of an existing workflow at `/workflows/<id>/canvas`. No edit affordances; F4-2 + F4-3 add palette + drawer.

### Why It Matters

- Locks the canvas library choice (@xyflow/react per `techstack.md`) and the route shape. Every subsequent F4-N ticket builds on this foundation.

---

## What This Ticket Must Accomplish

### Goal

- Add `@xyflow/react` to `app/package.json` deps.
- New route `/workflows/:workflowId/canvas` in `AppRoutes.tsx`.
- New page `app/src/pages/Workflows/WorkflowCanvas.tsx`:
    - Fetches the workflow via `workflowsApi.get(id)`.
    - Maps `workflow.nodes` → xyflow `Node[]` (one per workflow node, positioned via `node.position` if present, else auto-layout).
    - Maps `workflow.edges` → xyflow `Edge[]`.
    - Renders the graph at full-page height.
    - "Back to /workflows" link in the header.
- Custom node component per `NodeKind` showing the icon + name + summary. (Per-kind detail is F4-3's drawer.)
- Read-only: no drag, no edge dragging, no palette.

### Deliverables Checklist

#### A. Implementation

- [ ] `@xyflow/react` dep added.
- [ ] Route added + page component.
- [ ] Custom node renderer.
- [ ] Auto-layout (dagre or elkjs) when `node.position` is absent.

#### B. Tests

- [ ] Vitest snapshot of the canvas page given a fixture workflow.
- [ ] Route navigates from a `/workflows` card click on the workflow name.
