# F4-8 Primer: `fan_out` Node Kind + Parallel-Children Executor

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F2-2's executor walks DAGs sequentially. F4-8 unlocks per-fan-out parallelism.

---

## What Is This Ticket?

`fan_out` runs its child branches concurrently. Per OQ-18, default concurrency 5 with per-node override up to 20. The "draft one email per attendee" pattern (RU-9 in `prd.md`).

### Why It Matters

- The trickiest Phase 4 pattern. Without it, the only way to parallelize is to manually clone N copies of a sub-chain — unusable for any list-driven workflow.

---

## What This Ticket Must Accomplish

### Goal

- `FanOutConfig`:
    ```rust
    pub struct FanOutConfig {
        pub items_source: String,        // template producing a JSON array
        pub concurrency: u32,            // default 5, max 20
        pub aggregate: AggregateSpec,    // Collect | Reduce { spec: TransformSpec }
    }
    ```
- Executor body:
    1. Substitute `items_source` against `NodeContext`.
    2. Parse as `Vec<serde_json::Value>`. Reject if not array.
    3. For each item, run the child sub-DAG with `NodeContext.outputs["item"] = item` injected. Use `futures::stream` + `buffer_unordered(concurrency)`.
    4. Collect results per `aggregate`:
        - `Collect`: emit `[output_1, output_2, ...]`.
        - `Reduce { spec }`: pipe collected through the same transform spec as F4-6.
    5. Persist one `workflow_run_steps` row per child execution (gives the canvas N step rows to highlight in parallel during F4-5's live render).
- Cancel propagates: cancelling the run drops every in-flight child future.
- Per-node retry from F2-8 applies per-child, not for the whole fan-out.

### Deliverables Checklist

#### A. Implementation

- [ ] `FanOutConfig`, `AggregateSpec`.
- [ ] Executor arm using `futures::stream::buffer_unordered`.
- [ ] `dispatch_node` extension to accept an `injected_outputs` overlay (the per-child `item`).
- [ ] Live-render compatibility: each child publishes its own `WorkflowRunStepStarted` / `WorkflowRunStepCompleted` with a `parent_node_id`.
- [ ] Canvas form: items_source editor + concurrency slider + aggregate spec dropdown.

#### B. Tests

- [ ] 5 items × 1-node child chain runs in parallel; aggregate is `[output_1..output_5]`.
- [ ] Concurrency cap honored: 20-item fan-out at concurrency 5 completes in ~4× the slowest child's duration.
- [ ] Cancel mid-flight drops in-progress children.
- [ ] `items_source` not an array → run Failed with structured error.
