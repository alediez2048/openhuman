## What Is This Ticket?

Build the actual unified Connections Hub UI. Replace the contents of `app/src/pages/Connections.tsx` (renamed from Skills in P0-4) with the full 6-section layout. The page calls `connections_list` (P0-2 RPC) on mount, distributes the unified `Vec<ConnectionView>` to six section components, and adds search + filter chips above the section list.

The six sections, in display order:

1. **`<ComposioSection>`** ŌĆö relocated Composio UI (existing components from `app/src/components/composio/` re-mounted here).
2. **`<ChannelsSection>`** ŌĆö pulled in from the old `/channels` page; surfaces native chat channels (Slack, Telegram, Discord, ŌĆ”).
3. **`<WebviewAccountsSection>`** ŌĆö formerly buried in the `/chat` right-rail account picker. CEF-hosted webview accounts (LinkedIn, Twitter, WhatsApp, ŌĆ”).
4. **`<BuiltinIntegrationsSection>`** ŌĆö **NEW** (stubbed here, fully wired in P0-6). Lists Twilio / Apify / Google Places / Parallel / Seltz / Stock Prices.
5. **`<McpServersSection>`** ŌĆö **NEW** (stubbed here, fully wired in P0-6). Lists the registered MCP servers.
6. **`<GenericHttpSection>`** ŌĆö **NEW**. Uses P0-3's `connections_generic_http_create/update/delete/test` RPCs.

Search + filter chips per `requirements.md ┬¦1.9` apply across all sections ŌĆö chips select one or more `ConnectionKind` filters, search filters across `ConnectionView.label`.

Each card deep-links to the existing per-mechanism setup flow (`navigate('/onboarding/composio/<toolkit>')`, etc.) so we don't rewrite any OAuth or account-add flows.

### Why It Matters

- This is the **visible product** of Phase 0 ŌĆö the user opens `/connections` after this PR and finally sees every connected service in one place.
- The aggregator (P0-2) and the CRUD (P0-3) are dormant without this UI. P0-5 lights them up.
- The `<BuiltinIntegrationsSection>` and `<McpServersSection>` stubs introduced here establish the slot in the layout for P0-6 to fill ŌĆö the two PRs are sized as separable reviews.

## Suggested Implementation Pattern

**`app/src/services/api/connectionsApi.ts` (extract):**

```ts
import { coreRpcClient } from '../coreRpcClient';
import type { ConnectionView, ConnectionKind } from '../../types/connections';

export interface ListRequest { kind_filter?: ConnectionKind[]; search?: string; }
export interface ListResponse { connections: ConnectionView[]; generated_at: string; }

export const connectionsApi = {
  list: (req: ListRequest = {}): Promise<ListResponse> =>
    coreRpcClient.call('connections_list', req),
  createGenericHttp: (req: CreateGenericHttpRequest): Promise<GenericHttpConnection> =>
    coreRpcClient.call('connections_generic_http_create', req),
  updateGenericHttp: (id: string, req: UpdateGenericHttpRequest): Promise<GenericHttpConnection> =>
    coreRpcClient.call('connections_generic_http_update', { id, ...req }),
  deleteGenericHttp: (id: string): Promise<void> =>
    coreRpcClient.call('connections_generic_http_delete', { id }),
  test: (id: string): Promise<TestProbeResult> =>
    coreRpcClient.call('connections_test', { id }),
};
```

**`app/src/components/connections/ConnectionsHub.tsx` (extract):**

```tsx
export function ConnectionsHub() {
  const t = useT();
  const dispatch = useAppDispatch();
  const { connections, loadStatus } = useAppSelector(s => s.connections);
  const [searchParams] = useSearchParams();

  useEffect(() => { dispatch(fetchConnections()); }, [dispatch]);

  const search = searchParams.get('search') ?? '';
  const kindFilter = searchParams.getAll('kind') as ConnectionKind[];
  const filtered = useMemo(
    () => filterConnections(connections, { search, kindFilter }),
    [connections, search, kindFilter],
  );

  const byKind = useMemo(() => groupBy(filtered, c => c.kind), [filtered]);

  return (
    <div data-testid="connections-page-root">
      <header>...</header>
      <ConnectionsSearchBar />
      <ConnectionsFilterChips active={kindFilter} />
      {loadStatus === 'loading' && <SectionSkeleton count={6} />}
      {loadStatus === 'success' && (
        <>
          <ComposioSection         items={byKind.composio ?? []} />
          <ChannelsSection         items={byKind.channel ?? []} />
          <WebviewAccountsSection  items={byKind.webview ?? []} />
          <BuiltinIntegrationsSection items={byKind.builtin ?? []} />
          <McpServersSection       items={byKind.mcp ?? []} />
          <GenericHttpSection      items={byKind.generic_http ?? []} />
        </>
      )}
    </div>
  );
}
```

## Edge Cases to Handle

1. **Empty workspace.** `connections_list` returns `[]`. Render the 6 section headers with empty-state "Add your first X" CTAs.
2. **`connections_list` errors.** Show the error banner with a "Retry" button that re-dispatches `fetchConnections()`.
3. **A single connection in `Error` state.** Show with a red status badge; deep-link to the per-mechanism re-auth flow.
4. **URL hash `#channels` on first render.** `useEffect` scrolls the `<ChannelsSection>` into view after the section renders (use a ref + `scrollIntoView({ behavior: 'smooth' })`).
5. **Filter chip + search applied together.** Both are AND-combined; the section may render with zero cards. Show a "No matches" inline message.
6. **`<GenericHttpEditModal>` close mid-submit.** Disable the close button while `isSubmitting`; on submit error, surface the message and keep the modal open.
7. **`connections_test` returns slowly (close to 10s).** Show a spinner inside the "Test" button; disable other test buttons on the same card.
8. **A built-in integration is disabled in config.** Show with status badge `Disabled`, action button is "Enable" (disabled in P0-5, wired in P0-6).
9. **Long base URLs in cards.** Truncate with `text-ellipsis` and a tooltip on hover.
10. **Browser refresh while modal is open.** Acceptable to lose modal state ŌĆö no persisting needed.
