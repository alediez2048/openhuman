# P0-6 Primer: Surface native integrations + MCP servers as first-class connections

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Date:** 2026-05-19
**Previous work:** [`P0-5.md`](./P0-5.md) — Unified Connections Hub page with 6 sections; `<BuiltinIntegrationsSection>` and `<McpServersSection>` are structural stubs marked `// P0-6`.

---

## What Is This Ticket?

Light up the two sections P0-5 left as stubs. Both surface mechanisms that today exist in the codebase but are buried (built-in integrations live in `src/openhuman/integrations/` with config-driven toggles; MCP servers live in the MCP transport layer with no first-class settings UI).

**`<BuiltinIntegrationsSection>`** lists the six built-in integrations defined under `src/openhuman/integrations/`: **Twilio**, **Apify**, **Google Places**, **Parallel**, **Seltz**, **Stock Prices**. Each card shows the integration name, current enabled state (read from config), and (where applicable) scope/credential controls. The user can toggle each integration on/off and update credentials inline.

**`<McpServersSection>`** lists the MCP servers registered in the existing MCP server registry. The user can see server status (running/disabled), restart, and disable a server. "Add MCP server" deep-links to the existing settings flow (no inline add form in Phase 0).

This is a primarily frontend ticket, but it includes a small backend RPC pair (`integrations_list` + `integrations_set_enabled`) if the existing config surface isn't already exposed via a controller. Audit first; reuse what exists.

### Why It Matters

- Surfaces previously-hidden connection mechanisms — without P0-6, users don't know they have Twilio/Apify/etc. available.
- MCP servers are how OpenHuman extends its tool surface; first-class management makes the system feel less like a black box.
- Completes the 6-section coverage promised by `requirements.md §1.9` and `prd.md §5 Phase 0`.

## What This Ticket Must Accomplish

### Goal

Make both stub sections fully interactive. After P0-6, the user can toggle every built-in integration on/off from the Connections Hub, update credentials inline (where applicable), restart or disable an MCP server, and see all six integrations + every registered MCP server as live cards driven by `connections_list` (with refresh on mutation). If a backend RPC for the integration enabled-flag doesn't exist, add it under the controller-only exposure pattern.

## Suggested Implementation Pattern

**`app/src/components/connections/sections/BuiltinIntegrationsSection.tsx` (extract):**

```tsx
export function BuiltinIntegrationsSection({ items }: { items: ConnectionView[] }) {
  const t = useT();
  const dispatch = useAppDispatch();
  const [configuring, setConfiguring] = useState<string | null>(null);

  const handleToggle = async (id: string, enabled: boolean) => {
    await connectionsApi.setIntegrationEnabled(id, enabled);
    dispatch(fetchConnections());
  };

  return (
    <section data-testid="builtin-integrations-section">
      <SectionHeader
        title={t('connections.section.builtin')}
        count={items.length}
      />
      <div className="grid gap-3">
        {items.map(item => (
          <ConnectionCard
            key={item.ref.integration}
            title={t(`connections.builtin.${item.ref.integration}Name`)}
            description={t(`connections.builtin.${item.ref.integration}Description`)}
            status={item.status}
            actions={
              <>
                <Toggle
                  checked={item.status === 'Connected'}
                  onChange={enabled => handleToggle(item.ref.integration, enabled)}
                />
                <Button onClick={() => setConfiguring(item.ref.integration)}>
                  {t('connections.builtin.configureButton')}
                </Button>
              </>
            }
          />
        ))}
      </div>
      {configuring && (
        <BuiltinIntegrationConfigModal
          integration={configuring}
          onClose={() => setConfiguring(null)}
        />
      )}
    </section>
  );
}
```

**Backend RPC if added — `src/openhuman/integrations/rpc.rs` (extract):**

```rust
pub async fn handle_integrations_set_enabled(
    ctx: &Ctx,
    req: SetEnabledRequest,
) -> RpcOutcome<IntegrationView> {
    let updated = ctx.integrations().set_enabled(&req.id, req.enabled)?;
    publish_global(DomainEvent::ConnectionUpdated {
        r#ref: ConnectionRef::Builtin { integration: req.id.clone() },
    });
    tracing::info!(target: "connections", "[connections] integration {} enabled={}", req.id, req.enabled);
    RpcOutcome::Ok(updated)
}
```

## Edge Cases to Handle

1. **An integration is enabled in config but has no credential.** Status badge `MissingCredentials`. Toggle is on but the card shows a yellow warning. Configure modal opens with empty credential fields.
2. **A credential rotate via the modal.** Reuse the per-integration setter (same secret-rotation discipline as P0-3 Generic HTTP): write new secret, swap reference, delete old.
3. **MCP server in `Error` state.** Show the error message in a tooltip. Restart button is enabled.
4. **MCP server restart fails.** Surface the error toast; do NOT mark the card disabled.
5. **Concurrent toggles for the same integration.** Disable the toggle while the RPC is in flight; show a spinner.
6. **A built-in integration was deprecated upstream.** If the aggregator still lists it but the toggle RPC returns `Err`, show the error and offer "Remove" (out of scope for P0-6; log a follow-up issue).
7. **MCP server "Add" CTA in P0-6.** Just deep-links to existing settings; no in-modal add form. If existing settings don't have an MCP-add flow, surface that as a follow-up issue.
8. **Toggle a builtin off while a workflow uses it.** P0-6 doesn't validate downstream usage — that's Phase 1's workflow-health subscriber. The toggle succeeds; Phase 1's health recomputation will flip the affected workflows to `NeedsConnections`.
9. **i18n missing key fallback.** Configure modal renders the integration name from the metadata if the `connections.builtin.<id>Name` key is missing.
10. **A built-in integration is config-only (no credential).** The configure modal renders just scopes/options, no credential field.
