## What Is This Ticket?

DOM grounding (F3-2) fails when a page is rendered in canvas, SVG, or heavy-WebGL — there's no actionable element tree to extract. Examples: Figma-style UIs, some banking dashboards with canvas-drawn charts, gaming UIs, mapping interfaces.

F3-7 adds a vision-grounded fallback: when DOM grounding returns "no actionable elements" OR the LLM's `browser_act` translation fails twice in a row, the system takes a screenshot and calls a vision-capable model (Claude Sonnet/Opus with vision, or GPT-4o vision) with the screenshot + the natural-language action. The model returns pixel coordinates; F3-1 dispatches `click_at(x, y)` against those coordinates.

Modeled on Anthropic's computer-use Claude — same prompting pattern, same coordinate-grounded output.

### Why It Matters

- Without F3-7, browser-agent workflows fail silently or visibly on the ~10% of pages where DOM grounding doesn't work.
- F3-7 is the "ceiling-raising" feature — the cost is significant per call (vision models are 10–100× the tokens of text), but the capability expansion is real.
- Critically, F3-7 is OPT-IN per workflow. Default-off; users with canvas-heavy targets enable it. Cost discipline.

### Why It Matters Less Than F3-1..F3-6

- DOM grounding covers the vast majority of real-world targets. Most banking sites are HTML-form-based; most SaaS dashboards expose semantic HTML; most newsletters / portals / etc. are accessible.
- Vision grounding's failure modes are also harder to debug — when the model picks the wrong coordinates, the failure is opaque ("model said click here, page redrew").
- If F3-7 has to slip out of Phase 3.1 due to time pressure, the rest of the initiative still ships value. F3-7 is the "stretch" ticket.

## What This Ticket Must Accomplish

### Goal

When DOM grounding fails OR `browser_act` translation hits two consecutive failures, the system falls back to vision grounding. The action either succeeds (LLM returns coordinates, click lands on the intended element) or the agent gracefully reports failure. Costs are bounded.

### Deliverables Checklist

#### A. Vision-LLM caller

- [ ] `src/openhuman/browser_agent/perceive/vision.rs` — `vision_ground(session, action, snapshot_or_screenshot) -> Result<VisionGroundedAction>`.
- [ ] `VisionGroundedAction` enum:
    - `Click { x: f64, y: f64, confidence: f32 }`
    - `Type { x: f64, y: f64, text: String, confidence: f32 }`
    - `Scroll { dy: f64, confidence: f32 }`
    - `Ambiguous { reason: String }`
    - `OffPage { reason: String }` — target not visible
- [ ] Vision-LLM prompt template at `vision_prompt.md`:
    - Bundled via include_str!.
    - Tells the model: "You see a screenshot. The user wants you to: <action>. Return JSON `{action: 'click'|'type'|'scroll'|'ambiguous'|'off_page', x, y, text, confidence}` where (x, y) is in pixel coords on the screenshot. Confidence in [0,1]. If you're not sure, return ambiguous."
    - Includes 3–5 few-shot examples (LLM is much more reliable with examples for coordinate-grounding tasks).

#### B. Trigger logic in `browser_act`

- [ ] F3-3's `browser_act` gains a fallback branch:
    1. Try DOM grounding (existing path).
    2. If DOM returns no actionable elements OR intent-LLM returns `ambiguous`, AND the workflow's `BrowserActionConfig.vision_fallback = true`:
        - Capture screenshot via F3-1.
        - Call `vision_ground()` with the action + screenshot.
        - On `Click(x, y)` or `Type(x, y, text)` → dispatch the F3-1 primitive at those coords.
        - On `Scroll(dy)` → dispatch `scroll(0, dy)`.
        - On `Ambiguous` or `OffPage` → return `{ status: "vision_failed", reason }`.
    3. If `vision_fallback = false`, fall back to today's behavior (`ambiguous`).

#### C. Config + opt-in

- [ ] `BrowserActionConfig.vision_fallback: bool` (default `false`).
- [ ] Workflow validator accepts the field; drafter prompt (`workflow_builder.md`) documents when to set it.
- [ ] WorkflowCard / WorkflowProposalPreview surfaces "Vision fallback: on" in the Settings collapsible.

#### D. Cost gating

- [ ] Per-session counter: `vision_calls_count`. Reset per run.
- [ ] `BrowserActionConfig.max_vision_calls: u32` (default 5).
- [ ] When the counter trips, the next vision call returns `{ status: "vision_quota_exhausted" }` and the agent's safety preamble (F3-6) instructs graceful termination.
- [ ] Vision calls also count against F3-6's `max_llm_tokens` (vision tokens add up FAST — 1500 tokens per screenshot at moderate detail).

#### E. Verification overlay (live-preview integration with F3-5)

- [ ] When vision grounding picks coordinates, F3-7 emits a `DomainEvent::BrowserVisionCoordinatesChosen { x, y, confidence }` so the F3-5 panel can draw a target reticle on the next streamed screenshot frame.
- [ ] Helps the user audit: "did the agent click where I wanted?" visible in real time.

#### F. Tests

- [ ] Unit: `vision_ground` parses model output correctly (mock the LLM call). 5–10 fixture model outputs covering each VisionGroundedAction variant.
- [ ] Unit: `browser_act` correctly falls back when DOM grounding returns empty.
- [ ] Unit: `vision_fallback = false` does NOT trigger vision even when DOM fails (existing behavior preserved).
- [ ] Unit: vision-quota trip returns the right structured error.
- [ ] Integration: a synthetic page with NO actionable HTML elements (just a `<canvas>` with text drawn on it) — `browser_act("click the Save button")` falls back to vision and succeeds. Use a mocked vision model that returns plausible coords; the integration test asserts the dispatch path, not the model's intelligence.

## Architectural Decisions

### Why vision is OPT-IN, not OPT-OUT?

Cost. A typical workflow run with DOM grounding consumes 5–15k tokens. With vision, it can blow past 50k easily. Defaulting on would surprise users with the bill. Opt-in lets users with canvas-heavy targets accept the cost knowingly.

### Why per-workflow max_vision_calls, separate from max_llm_tokens?

`max_llm_tokens` is the cost gate. `max_vision_calls` is an UX gate — even if you have budget, after 5 vision falls you should probably give up rather than keep trying. The cap surfaces the failure mode early. (Vision-fallback that keeps failing usually means the page is genuinely unsuitable for agent automation.)

### Why few-shot examples in the vision prompt?

Coordinate-grounding tasks are sharply better with examples. Anthropic's computer-use docs are emphatic about this; browser-use's research replicates it. 3–5 examples cost ~500 tokens added to every vision call; the accuracy improvement is dramatic.

### What about Anthropic's computer-use API (`computer_20241022` tool type)?

If Anthropic's computer-use is available via OpenHuman's existing provider abstraction, F3-7 should USE IT rather than rolling our own coord-grounding prompt. The Anthropic-native tool is purpose-built and likely more accurate than a hand-rolled prompt. Verify during implementation:

- Is `inference::provider::create_chat_provider("vision", config)` returning a provider that supports computer-use tools?
- If yes — use the native API. F3-7 becomes mostly a thin wrapper.
- If no — ship the hand-rolled prompt. Migrate to native when the provider catches up.

### What about GPT-4o vision or Gemini Vision?

The vision-LLM call should route through the "vision" workload role so users on different inference providers get the best model their provider offers. F3-7 doesn't hard-code Anthropic — though hand-rolled coord-grounding prompts may need provider-specific few-shot tuning.

### Why not vision-by-default with a "DOM only" opt-out?

Two reasons:

1. Vision is 10–100× more expensive per action.
2. Vision is less debuggable when it fails. "Model said click here, page didn't change" is a much harder failure mode than "DOM extractor returned no elements, retry with different selector".

DOM-first is the right default; vision is the exception case.

## Acceptance criteria

- `cargo test --lib browser_agent::perceive::vision` passes.
- `cargo test --test browser_vision_integration` passes — synthetic canvas-only page, mocked vision, end-to-end click dispatch.
- A real workflow with `vision_fallback = true` against a canvas-heavy fixture page (Figma-like) successfully clicks the right element in 4 of 5 runs (Anthropic computer-use baseline; if hand-rolled, accept 3 of 5).
- Vision-quota trip terminates the run cleanly with the right `Failed` status + audit-log entry.
- F3-5 panel renders the target reticle when vision picks coordinates.
- Vision token costs are visible in the audit log per-call.

## Open question for the implementer

Anthropic's computer-use is currently the cleanest off-the-shelf primitive for this. If the OpenHuman backend's provider catalog includes a computer-use-capable Anthropic deployment by the time F3-7 starts, **strongly consider** making F3-7's first version a thin wrapper rather than rolling a custom prompt. Document the decision in the DEVLOG when F3-7 closes.
