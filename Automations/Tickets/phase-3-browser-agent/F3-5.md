## What Is This Ticket?

Build the React side panel that shows users what the browser agent is doing in real time. Streaming screenshots, action log, take-over button, dry-run toggle.

This is the **trust surface** for the entire browser-agent initiative. Without it, users running a workflow that "logs into my brokerage" have no way to know what actually happened — and they won't (and shouldn't) enable browser-agent-driven workflows.

### Components shipped

```text
app/src/components/workflows/browser-preview/
    BrowserPreviewPanel.tsx     // Container — manages live-stream lifecycle
    BrowserScreenshot.tsx       // Renders latest screenshot frame + bounding boxes
    BrowserActionLog.tsx        // Scrolling list of every browser_act / observe / extract call
    BrowserTakeoverButton.tsx   // "I'll take it from here" — pauses agent, gives control to user
    BrowserDryRunToggle.tsx     // When on, agent describes actions instead of executing them
```

The panel docks into the existing run-detail surface (Phase 2 F2-12 adds a per-run history view; F3-5 builds on top).

### Why It Matters

- Browser agent without live preview = scariest possible UX. User clicks "Run", workflow runs, claims success — user has no idea if it logged into their actual brokerage account or some random page that LOOKED right.
- The Phase 3.1 acceptance criteria explicitly includes "live-stream the browser session by default". This is that.
- Take-over is the safety valve for everything the agent can't handle (CAPTCHAs, 2FA prompts, weird UI states). Without it, users hit a dead end and the workflow fails silently.

## What This Ticket Must Accomplish

### Goal

When a `BrowserAction` workflow node is running, the user sees:
- A live screenshot updating every 1–2 seconds (or on every agent action).
- A side-by-side action log: `[10:32:14] browser_observe — found 8 elements`, `[10:32:16] browser_act → click [3] "Login button"`, etc.
- A clearly-visible "Take over" button that pauses the agent and gives the user direct control of the CEF window.
- An optional "Dry run" toggle (set at workflow-create time, not toggleable mid-run).

### Deliverables Checklist

#### A. Rust side — streaming screenshot events

- [ ] `src/openhuman/browser_agent/preview/mod.rs` — `BrowserPreviewBroadcaster` that watches a session and emits compressed screenshots on action boundaries (post-`browser_act`, post-`browser_observe` triggering a new snapshot).
- [ ] New `DomainEvent::BrowserPreviewFrame { run_id, png_bytes_base64, viewport, timestamp }`.
- [ ] Broadcaster respects backpressure — if the frontend isn't consuming, drop intermediate frames (keep the latest).
- [ ] Default cadence: 1 frame post-action. Configurable to "every Ns" for long single-action stalls (e.g., a slow page load).
- [ ] Screenshot compression: PNG with `quality_hint = 60`, max 800×600 (downscale if viewport larger). Per-frame ≤ 50 KB target.
- [ ] Privacy: a redaction pass before broadcast — F3-6's `RedactionPolicy` is consulted (if F3-6 ships in parallel; if not, ship F3-5 with a stub policy that's a no-op).

#### B. Tauri IPC

- [ ] `core_rpc_subscribe_browser_preview(run_id) -> StreamHandle` — establishes a streaming subscription. The frontend gets frames pushed via the existing socket.
- [ ] `core_rpc_release_browser_preview(run_id)` — unsubscribes (called on panel unmount).

#### C. Frontend — the panel itself

- [ ] `app/src/components/workflows/browser-preview/BrowserPreviewPanel.tsx`:
    - Mounts on a run-detail view when the run has at least one `BrowserAction` node.
    - Subscribes via `core_rpc_subscribe_browser_preview` on mount.
    - Renders the screenshot (latest frame) + action log side-by-side. Responsive: side-by-side on `lg+`, stacked on smaller.
    - "Take over" button (functional in F3-5 — wires through to Tauri to surface the CEF window).
    - "Pause" button (separate from take-over — agent pauses but no user input).
- [ ] `BrowserScreenshot.tsx` — `<img>` with base64 src + overlay layer drawing bounding boxes for the "current element of interest" (from the most recent `browser_act`).
- [ ] `BrowserActionLog.tsx` — virtualised list (use existing `react-window` if it's a dep; otherwise plain scroll with cap at 200 entries). Each row: timestamp, tool name, brief argument summary, success/failure pill.
- [ ] `BrowserTakeoverButton.tsx` — fires `core_rpc_browser_takeover(run_id)`. Tauri shell surfaces the (formerly hidden) CEF window. The agent pauses; resume button comes back when the user clicks "Done".
- [ ] `BrowserDryRunToggle.tsx` — read-only display of whether dry-run is on (the toggle itself is at workflow-create time, not mid-run).

#### D. Tauri shell — surface-hidden-window command

- [ ] `app/src-tauri/src/browser_agent_window.rs` — Tauri command `surface_browser_agent_window(target_id)` that finds the CEF window for the run's session and shows it (lifts from hidden state if currently hidden).
- [ ] Companion `hide_browser_agent_window(target_id)` for when the user clicks "Done" after take-over.

#### E. Action-log content

- [ ] The action log subscribes to `DomainEvent::ToolExecutionCompleted` filtered by `session_id == "workflow:<run_id>"` and the tool name being one of `browser_observe` / `browser_act` / `browser_extract`.
- [ ] For each event, renders:
    - Timestamp (relative — "just now" / "5s ago" / "1m ago", in the same style as the WorkflowCard last-run formatting).
    - Tool name.
    - One-line argument summary (e.g., `"click the Save button"` → `"act: click the Save button"`).
    - Success pill (sage) or Failure pill (coral).
- [ ] Click a row → expands to show the full args + result payload.

#### F. Tests

- [ ] Vitest: BrowserPreviewPanel renders without crash with mock streaming hook returning canned frames.
- [ ] Vitest: action log renders the correct sequence + handles burst of events (10 in 100ms — no race).
- [ ] Vitest: take-over button calls the right Tauri command + the panel switches to "user has control" state.
- [ ] Integration (WDIO): run a fixture workflow with a `BrowserAction` node, assert the panel mounts, asserts at least 3 screenshots arrive, assert the action log has the expected entries.

## Architectural Decisions

### Why screenshot-streaming, not "live CEF window embedded in the React app"?

Embedding a CEF webview INSIDE the React app's window (a webview within a webview) is technically possible but architecturally a mess — CEF doesn't compose with itself cleanly, and the security boundary blurs (which user-data-dir? which profile?). Streaming PNGs is the lowest-blast-radius option that gives the user visibility.

The take-over button DOES surface the actual CEF window (separate macOS window) — that's the only time the user interacts with the real session, and the separation is intentional.

### Why post-action screenshots, not 60fps streaming?

Cost. Each PNG is 30–80 KB. At 60 fps that's 2–5 MB/s per active session. Over the Tauri IPC channel that's a non-trivial load and burns user GPU on the encode side too. Post-action gives the user every meaningful state change — the page redraws between consecutive actions, so a single frame post-action captures the new state. Slow-load actions get a configurable timer fallback (default: every 3s during a `wait_for_load`).

### Why bounding boxes via overlay, not bake them into the screenshot?

Two reasons. First, the CDP screenshot is the truth — overlaying lets the user see both the page and the agent's "where I'm clicking" annotation, distinguishable. Second, bounding boxes change frequently (every `browser_act`); re-encoding the PNG for each annotation is wasted CPU.

### Why is the dry-run TOGGLE only at workflow-create time, not mid-run?

Dry-run changes the SEMANTICS of every action — flipping mid-run would leave the page in an undefined intermediate state. Decision happens at workflow-save time; the toggle in the preview is for visibility only.

### What about a Phase 3.2 "replay" feature?

Defer. Phase 3.5 — record-then-replay would be cool but requires persistent screenshot storage + a replay UI. Not Phase 3.1 scope.
