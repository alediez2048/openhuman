# F-18 Primer: MCP server registration is user-isolation safe

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Date drafted:** 2026-05-23
**Phase:** Phase 1.5 (closes a multi-user data-loss gap surfaced during F-17 live testing — same family as F-16's honest-status gate: visible silent failure that misleads the user).
**Estimated time:** 1–1.5 days.

## What Is This Ticket?

The Phase 0 Connections Hub lets a user register MCP servers (and HTTP / API endpoints) through `/connections`. Those registrations live in the per-user `~/.openhuman/users/<user_id>/config.toml` under `[mcp_client]` / `[[mcp_client.servers]]`. The runtime resolves the active user via `~/.openhuman/active_user.toml` on every `Config::load_or_init` call.

**The bug:** the `add_mcp_server` flow can write to one user's config while the runtime reads from another, silently orphaning the MCP credentials. The user sees:

- UI says "Status: connected" (the modal closed without an explicit error).
- Every subsequent MCP tool call fails.
- The chat agent **invents an HTTP 401** error (because the LLM doesn't have a structured "server not registered" signal to surface, so it confabulates a plausible-sounding auth failure).
- The user generates fresh API tokens from the upstream service, re-pastes them, and hits the same error — because the token was never the problem.

Hours wasted, no path to self-diagnose.

This ticket closes the gap by making MCP server registration **user-isolation safe end-to-end**: the write can't land in the wrong user, orphaned entries from prior sessions get a one-click recovery path, and the chat-agent confabulation has structured-error scaffolding to render the real cause.

### How this was discovered (2026-05-23)

During F-17 live verification the user (`active_user_id = "user-123"`) registered a Higgsfield MCP server through the UI. The save persisted at `~/.openhuman/users/6a0b81bc03db3df2223f0e5f/config.toml` (a previous session's user id), not `user-123`'s config. The chat agent's MCP tool calls failed every time because `user-123`'s `[mcp_client].servers = []`. The agent's response told the user "401 from Higgsfield" — a fabrication; no HTTP request was being made. After three rounds of fresh-token + paste-again, a manual `grep` of both config files surfaced the truth: the token was sitting in the wrong user's config.

The active-user flip itself most likely happened via a `test_reset` RPC at some earlier point (the F-17 E2E spec runs this against its own tempdir workspace and shouldn't have touched the main config, but ANY logout / re-auth / dev-mode switch trips the same wire). The root-cause class is "in-flight `Config` handle holds a stale `config_path` that no longer matches the active user resolved from disk."

## What Was Already Done

- **Phase 0 P0-6** wired the MCP add / remove / test RPCs (`openhuman.connections_mcp_add` / `_mcp_remove` / `_mcp_test`) into `src/openhuman/connections/{rpc,ops,schemas}.rs`. The `[mcp_client]` config section + `McpServerConfig` type live in `src/openhuman/config/schema/types.rs`.
- **`Config::save`** at `src/openhuman/config/schema/load.rs:1638` atomically writes the in-memory `Config` to `self.config_path` (the path captured when the `Config` was loaded).
- **`Config::load_or_init`** resolves the active user via `read_active_user_id(default_openhuman_dir)` (reads `active_user.toml`) at `schema/load.rs:309`.
- The MCP-add UI (`app/src/components/connections/sections/McpAddModal.tsx`) wires to `connectionsApi.mcpAdd(req)` which calls `openhuman.connections_mcp_add`.

What's MISSING:

- No validation that `config.config_path` at save-time still matches the currently-resolved active-user path.
- No scan for orphaned per-user MCP configs at boot.
- No structured RPC-error variant the UI / chat agent can detect as "server not registered" vs "auth failure" — they collapse to a generic string the LLM has to interpret.

## What This Ticket Must Accomplish

### Goal

A user can register an MCP server and trust it. If their session is in a state where the registration can't safely land, they get a clear actionable error (not a silent write to the wrong file). If they have orphaned credentials from a previous session, they can recover them with one click — no `sqlite3` / `grep` excursion required.

### Deliverables Checklist

#### Part 1 — Immediate data fix (one-shot, applied at F-18 ramp-up)

- [ ] Copy the orphaned Higgsfield `[[mcp_client.servers]]` block from `~/.openhuman/users/6a0b81bc03db3df2223f0e5f/config.toml` into `~/.openhuman/users/user-123/config.toml`.
- [ ] Switch `servers = []` to the proper `[[mcp_client.servers]]` array-of-tables form.
- [ ] Verify by re-reading the target file + restarting the app + watching the MCP server show as available in `/connections`.
- [ ] Documented in `DEVLOG.md` as a one-off recovery (not a code change). The proper fix is Part 3's migration RPC; this is a "unblock the user TODAY" step.

#### Part 2 — Stale-handle guard in MCP add/remove

- [ ] In `src/openhuman/connections/ops.rs::add_mcp_server`, BEFORE `persisted.save().await`:
  1. Resolve the currently-active config path via the same path used by `Config::load_or_init` (likely a `resolve_active_config_path()` helper extracted from `schema/load.rs`).
  2. If `config.config_path != active_path`, abort with a new structured error `McpAddError::StaleSessionHandle { in_flight_path, active_path }`.
- [ ] Mirror in `remove_mcp_server`.
- [ ] Add the variant to whatever error type the MCP RPCs return; surface it as a stable `code = "stale_session_handle"` in the RPC envelope so the UI can branch.
- [ ] **Frontend:** `McpAddModal.tsx` catches the structured code and renders a clear inline error: "Your session changed since you opened this dialog. Please log out, log back in, and try again." Don't pop a generic toast — the modal stays open with the credentials still typed in, so the user doesn't lose their work.
- [ ] **Tests:**
  - Unit: stale handle (manually rewrite `active_user.toml` between load + save) returns `StaleSessionHandle`.
  - Unit: happy path (no concurrent change) writes successfully.
  - Unit: simulates the failure mode that produced the F-18 bug: load Config under user A → flip `active_user.toml` to user B → save should fail loud, NOT silently write to user A's config.

#### Part 3 — Boot-time orphan MCP scanner + one-click migration

- [ ] **Detection (Rust, runs once at boot after `memory::global::init` finishes):**
  - Walk `~/.openhuman/users/*/config.toml`.
  - For every file where `user_id != active_user_id` AND `[mcp_client].servers.len() > 0`, record the orphans.
  - Cap at first 10 orphan-bearing files to bound work — anyone with more than that has a config sprawl problem we should surface separately.
  - Emit `DomainEvent::McpOrphansDiscovered { count, by_user }` so subscribers (the UI, telemetry) can react.
- [ ] **`openhuman.connections_mcp_orphans_list` RPC** — returns the orphan inventory: `Vec<{ source_user_id, servers: Vec<McpServerConfig> }>`. No secrets in the response — redact `auth.token` to `[redacted]` for the listing; the migration RPC reads the real value server-side.
- [ ] **`openhuman.connections_mcp_orphans_migrate` RPC** — takes `{ source_user_id, server_name }` (one at a time so the UX is "I want THIS server, not all of them"). Reads the orphan's full config including secrets, calls `add_mcp_server` against the currently-active user's config to land it cleanly. Returns the migrated `McpServerConfig`.
- [ ] **Frontend `/connections` surfaces orphan banner** when the list RPC returns ≥1 entry. Banner: "Found N MCP server(s) registered in a previous session: [Higgsfield (from 6a0b81b…)] [Restore] [Dismiss]." Each row gets its own [Restore] button → calls the migrate RPC. Dismiss persists a `mcp_orphans_dismissed_at` timestamp in the user's config so the banner doesn't nag forever.
- [ ] **Never auto-migrate.** Explicit click required — credentials moving across user identities is a security-sensitive operation that should be opt-in.
- [ ] **Tests:**
  - Unit: orphan scanner walks a tempdir with 3 user dirs (one active, two with orphan MCP servers) → returns 2 entries.
  - Unit: migrate copies the server from source to active, leaves the source's config intact (no auto-delete on migration — the source user may want it back).
  - Unit: secret redaction in the list response.

#### Part 4 — Bonus: chat-agent confabulation hardening (optional, scope-aware)

This is the secondary bug surfaced by the F-18 repro. NOT required for shipping F-18 but worth capturing here so it doesn't get lost.

- [ ] When an MCP tool call references a server name that isn't registered, the tool result should be a structured error (`{ kind: "unknown_server", server_name }`) — NOT a free-form string that the LLM has to interpret. The orchestrator's tool-error rendering then knows to surface "MCP server '<name>' isn't registered" rather than letting the LLM invent an HTTP status code.
- [ ] If this scope-creeps the ticket, split into its own F-19 (or punt to Phase 2's polish pass).

## Architectural Decisions

### Why a stale-handle guard, not "force a fresh load before save"?

We considered re-loading Config from disk inside `add_mcp_server`, applying the diff, then saving. Two problems: (1) it silently absorbs ALL other config changes that happened between the user opening the modal and clicking save — the diff is wider than just the MCP server addition, which is surprising and lossy. (2) It hides the "your session changed" signal from the user, who may legitimately want to know.

The guard preserves the user's typed credentials in the modal, surfaces the real reason cleanly, and forces an explicit recovery action. Net: fewer silent failures, more agency.

### Why migrate-on-explicit-click, not auto-migrate at boot?

Credentials moving across user identities is security-sensitive: a Higgsfield API key the user generated while logged in as Account A shouldn't get attached to Account B without the user explicitly saying "yes, that's mine, move it". One-click in the UI keeps the migration auditable. The orphan banner is enough nudge for a user who knows what they're looking at; anyone confused by it leaves the credentials where they are (no data loss).

### Why redact secrets in the `_list` RPC but not `_migrate`?

The `_list` response feeds the UI banner. The UI doesn't need the actual token to render the row — just enough metadata for the user to recognize "yes, that's my Higgsfield setup". `_migrate` reads the real value server-side; the token never crosses the RPC boundary on the read path. Same posture as `connections_list` for Composio (returns connection metadata, never the OAuth refresh token).

### Why is the chat-agent fix optional / out of scope?

The structural fix (registered-server check) is straightforward. But the BROADER pattern — LLMs confabulating error details when tool errors come back as opaque strings — is a Phase 2 polish concern that touches every tool, not just MCP. Fix the structural root cause in F-18 / F-19; the prompt-side discipline (orchestrator system prompt teaching it to NOT invent error details) is its own ticket.

### Migration vs cleanup

Part 3's `_migrate` copies, doesn't move. We never delete from the source config because:
- The source user might log back in tomorrow and expect their MCP servers to still be there.
- A user with two active accounts (work + personal) may want the same MCP server in both.

A separate `_cleanup_orphan` RPC could land later if user feedback shows the duplication is confusing.

## Acceptance criteria

- Adding an MCP server while `active_user.toml` matches the in-flight Config → succeeds, lands in the active user's `config.toml`.
- Adding an MCP server after `active_user.toml` was flipped to a different user → fails loud with `stale_session_handle`, modal stays open with credentials preserved, UI renders "session changed" message.
- Booting the app with orphan MCP entries in `users/*/config.toml` → orphan banner renders in `/connections` listing the orphan servers with source user id.
- Clicking [Restore] on a banner row → server appears in the active user's config + becomes usable immediately (no app restart).
- Dismissing the banner → it doesn't re-render until a NEW orphan appears (existing orphans stay dismissed).
- `cargo test --lib connections::ops_tests::mcp_orphan` passes (≥3 new tests).
- Live: the Higgsfield repro from F-17's session no longer reproduces — registration always lands in the right user, OR fails with a clear actionable error.
