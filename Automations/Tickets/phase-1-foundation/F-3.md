**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Date:** 2026-05-19
**Previous work:** Phase 0 complete; F-2 (`Automations/Tickets/phase-1-foundation/F-2.md`) shipped the workflows CRUD RPCs with an initial-health call site that currently no-ops.

---

## What Is This Ticket?

Implement the real `health::recompute(workflow, connections_snapshot) -> WorkflowHealth` function that F-2 calls on create/update, and wire a `bus.rs` subscriber that listens to `DomainEvent::ConnectionAdded` and `DomainEvent::ConnectionRemoved` and recomputes health for every affected workflow with a bounded `UPDATE … WHERE` clause. Publish `WorkflowHealthChanged` on every transition.

Per ADR-017, `WorkflowHealth` is a **computed-but-persisted** field. F-1 created the JSON column; F-2 stubbed the recompute call site; F-3 makes both real.

### Why It Matters

- Without the subscriber, a user who saves a workflow with `health: NeedsConnections { missing: [linkedin] }` and then sets up LinkedIn in `/connections` would still see ⚠️ until they manually edited the workflow. The subscriber unblocks the canonical user journey from FR-1.2.4.
- The `health` field is the primary list-view filter chip (FR-1.2.7) and the dispatch precondition (FR-1.4.1.3) — every later ticket depends on `health` being accurate.
- Codifies the bounded-work pattern for event subscribers: a single `UPDATE` per event, never an unbounded full-table scan.

## What This Ticket Must Accomplish

### Goal

- A real `health::recompute(workflow: &Workflow, connections: &ConnectionsSnapshot) -> WorkflowHealth` returns `Ready` iff every `ConnectionRef` referenced by any node's `allowed_connections` (or by `required_connections` if a future caller passes one in) is present and `status == Connected` in the snapshot. Otherwise returns `NeedsConnections { missing }` with the precise list.
- A `bus.rs` subscriber registered at core startup that:
  1. Receives `DomainEvent::ConnectionAdded { r#ref }` or `DomainEvent::ConnectionRemoved { r#ref }`.
  2. Queries `workflows` for rows whose `nodes_json` references the changed `ConnectionRef`.
  3. Recomputes `health` for each affected workflow against the fresh connections snapshot.
  4. Persists the new `health` value and, if it changed, publishes `DomainEvent::WorkflowHealthChanged { workflow_id, health }`.
- The subscriber runs in a background tokio task off the event-bus broadcast channel (per `systemsdesign.md §8.1`).
- All work is bounded — a single SQL `UPDATE` per event, never a full-table scan.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| Persisted health column, recomputed on events | ADR-017 | Fast list-view reads + lazy recompute |
| Bounded `UPDATE … WHERE nodes_json LIKE …` per event | systemsdesign §8.1 | Avoid full-table scans on every connection change |
| Only publish `WorkflowHealthChanged` on real transitions | This ticket | Prevents event-storm on no-op recomputes |
| The "LIKE" pre-filter may over-select; final filter in `referenced_connections` | This ticket | Cheap pre-filter using existing string index; cheap secondary filter in code |
| Phase 1 only walks `AgentPrompt.allowed_connections` | systemsdesign §3.3 | Other node kinds arrive in Phase 2; the helper has a clear extension point |

## Suggested Implementation Pattern

**`src/openhuman/workflows/health.rs`:**

```rust
pub fn recompute(workflow: &Workflow, connections: &ConnectionsSnapshot) -> WorkflowHealth {
    let referenced = referenced_connections(workflow);
    let missing = missing(&referenced, connections);
    if missing.is_empty() {
        WorkflowHealth::Ready
    } else {
        WorkflowHealth::NeedsConnections { missing }
    }
}

pub fn referenced_connections(workflow: &Workflow) -> Vec<ConnectionRef> {
    let mut out = Vec::new();
    for node in &workflow.nodes {
        if let NodeConfig::AgentPrompt(cfg) = &node.config {
            out.extend(cfg.allowed_connections.iter().cloned());
        }
        // Phase 2: extend to ToolCall.connection, HttpRequest.connection_id, etc.
    }
    out.sort();
    out.dedup();
    out
}

pub fn missing(refs: &[ConnectionRef], snapshot: &ConnectionsSnapshot) -> Vec<ConnectionRef> {
    refs.iter()
        .filter(|r| !snapshot.is_connected(r))
        .cloned()
        .collect()
}
```

**`src/openhuman/workflows/bus.rs`:**

```rust
pub struct WorkflowHealthRecomputeSubscriber {
    store: Arc<Mutex<WorkflowsStore>>,
    aggregator: Arc<ConnectionsAggregator>,
}

#[async_trait]
impl EventHandler for WorkflowHealthRecomputeSubscriber {
    fn name(&self) -> &str { "workflow::health_recompute" }
    fn domains(&self) -> &[&str] { &["connection"] }

    async fn handle(&self, event: DomainEvent) {
        let r#ref = match &event {
            DomainEvent::ConnectionAdded   { r#ref } => r#ref.clone(),
            DomainEvent::ConnectionRemoved { r#ref } => r#ref.clone(),
            DomainEvent::ConnectionUpdated { r#ref } => r#ref.clone(),
            _ => return,
        };
        tracing::debug!(target: "workflows-bus", "connection event ref={:?}", r#ref);

        let mut store = self.store.lock().await;
        let affected = store.list_workflows_referencing(&r#ref)?;
        let snapshot = self.aggregator.list().await?;
        for wf in affected {
            let new = health::recompute(&wf, &snapshot);
            if new != wf.health {
                store.set_health(&wf.id, &new)?;
                publish_global(DomainEvent::WorkflowHealthChanged {
                    workflow_id: wf.id.clone(),
                    health: new,
                });
            }
        }
    }
}
```

## Edge Cases to Handle

1. **Workflow with no `AgentPrompt` nodes** (e.g., all-`ToolCall` workflows once Phase 2 ships) — `referenced_connections` returns `[]`, `missing` returns `[]`, `recompute` returns `Ready`.
2. **`ConnectionsSnapshot::is_connected` definition.** Verify with the P0-2 aggregator. A `ConnectionView` is "connected" iff `status == Connected` AND, for mechanisms that require verification (Generic HTTP, MCP, Channels), `verification.as_ref().map(|v| v.result == Live).unwrap_or(false)`. For Composio / Webview / Built-in, `verification` is `None` and `status == Connected` is authoritative. `Disabled`, `Error { reason }`, `verification == Failed`, and (for verification-required mechanisms) `verification == None` all count as missing. Reuse the same `requireVerification` truth table the `<ConnectorTile>` frontend uses (Phase 0) so the recompute matches what the Hub displays.
3. **Event arrives during executor's run.** `health` is not recomputed mid-run (FR-1.1.8). The subscriber updates the persisted value but the in-flight run reads from the in-memory workflow snapshot taken at dispatch time. Tests should verify this independence.
4. **Cyclic connection events.** If `set_health` somehow triggered a `ConnectionAdded` cascade (it doesn't today), we'd loop. Document the no-cycle invariant in the subscriber rustdoc; the subscriber only publishes `Workflow*` events, never `Connection*`.
5. **Workflow deleted while subscriber is mid-process.** `store::set_health` returns Ok(0 rows updated); skip silently. Log a `tracing::debug!`.
6. **`ConnectionUpdated` events.** Treat as both add+remove for safety — recompute affected workflows.
7. **Migration race.** The subscriber runs after `WorkflowsStore::open` (which applies migrations). Wire the subscriber registration in the same boot sequence as the store open so ordering is deterministic.
8. **Tracing target.** Use `target: "workflows-bus"` consistently — that's the grep prefix the operator will reach for.
