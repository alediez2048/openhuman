# F-9 Primer: Single-Flight Invariant + Soft Cancel + Orphan Recovery Sweep

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Date:** 2026-05-19
**Previous work:** Phase 0 complete; F-8 (`Automations/Tickets/phase-1-foundation/F-8.md`) shipped the executor pipeline with an unused `in_flight: HashMap<WorkflowId, RunId>` field and a stubbed `cancel_run` returning `Err(CancelError::NotImplemented)`.

---

## What Is This Ticket?

Make the single-flight invariant from ADR-014 real, implement soft cancellation, and add the boot-time orphan-recovery sweep. After F-9:

- At most one in-flight run per workflow. New triggers that arrive while a run is in flight are **dropped**, not queued. Each drop publishes `DomainEvent::WorkflowRunSkipped { workflow_id, reason: AlreadyRunning, attempted_trigger_source }`.
- `workflows_cancel_run(run_id)` sets a `cancelled` flag on the run row. The current node completes naturally; subsequent nodes are skipped; status becomes `Cancelled`; the `in_flight` slot releases.
- On core startup, `executor::orphan_recovery_sweep` marks every `workflow_runs` row with `status = Running` as `Failed { reason: "CoreCrashed" }`. Runs *before* any scheduler dispatch.

### Why It Matters

- Without single-flight, a misconfigured cron (`* * * * *`) or two simultaneous `workflows_run_now` clicks would double-fire side-effecting workflows (e.g., post two LinkedIn updates) — per ADR-014, this is the critical concurrency invariant.
- Soft cancel lets the user halt a long-running run via the UI without aborting the LLM mid-stream (which would corrupt the agent's memory writes). The cooperative pattern is the only safe option.
- Orphan recovery prevents stale `Running` rows from blocking subsequent dispatches forever after a core crash.

## What This Ticket Must Accomplish

### Goal

- **Single-flight enforcement** inside `executor::dispatch_run`:
    1. Acquire the `in_flight` mutex.
    2. If `in_flight.contains_key(&workflow_id)`, publish `WorkflowRunSkipped { workflow_id, reason: AlreadyRunning, attempted_trigger_source }` and return `Err(DispatchError::AlreadyRunning { run_id: existing })`.
    3. Otherwise create the `workflow_runs` row, insert into `in_flight`, drop the mutex, spawn the task.
- **Slot release** on `execute_inner` exit (every code path — success, failure, timeout, cancel, panic — must remove the `workflow_id` from `in_flight`).
- **Soft cancellation:**
    - `workflows_cancel_run(run_id)` RPC calls `executor::cancel_run(run_id)`.
    - `cancel_run` looks up the run, sets `workflow_runs.cancelled = 1` (and `set_cancelled_flag` in `store.rs`), publishes a `WorkflowRunStepCompleted { status: Cancelled }`-shaped event for the active step if there is one.
    - The executor's node loop checks `store::is_cancelled(run_id)` between nodes; if true, marks the run `Cancelled` and exits. (Phase 1 has one node so the check effectively fires after the first node completes — there's nothing to skip — but the loop structure is in place for Phase 2.)
    - The current node's LLM call is **not** aborted (per ADR-014).
- **Orphan recovery sweep:**
    - `executor::orphan_recovery_sweep(ctx) -> Result<usize>`:
        ```sql
        UPDATE workflow_runs
        SET status = 'failed', error = 'CoreCrashed',
            completed_at = ?
        WHERE status = 'running'
        ```
        Returns the count of rows touched.
    - Called at core startup, **before** any `scheduler::reconcile_at_startup` dispatch. Place in the boot sequence so it runs after `WorkflowsStore::open` but before cron-tick handlers wake.
    - Logs: `[workflows-run] orphan_recovery_sweep marked N runs as Failed{CoreCrashed}`.
    - For every recovered run, publishes `WorkflowRunCompleted { workflow_id, run_id, status: Failed }`.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| Drop overlapping triggers, don't queue | ADR-014 | One simple invariant, queue semantics deferred |
| Soft cancel — current node completes | ADR-014 | LLM mid-stream abort would corrupt memory writes |
| Orphan sweep runs at boot, before scheduler | FR-1.6.10 | Stale Running rows would otherwise block new dispatches |
| `Drop`-based slot release in `execute_inner` | This ticket | Panic-safe; works on every exit path |
| Cancellation check between nodes, not within | FR-1.6.9 | Cooperative pattern; in-node LLM calls are not aborted |

## Suggested Implementation Pattern

**`src/openhuman/workflows/executor.rs` — in-flight slot guard:**

```rust
struct InFlightSlot {
    state: Arc<ExecutorState>,
    workflow_id: WorkflowId,
}

impl Drop for InFlightSlot {
    fn drop(&mut self) {
        let mut in_flight = self.state.in_flight.lock();
        in_flight.remove(&self.workflow_id);
        tracing::debug!(target: "workflows-run",
            "in_flight slot released wf={}", self.workflow_id);
    }
}

pub async fn dispatch_run(/* ... */) -> Result<RunId, DispatchError> {
    // ... health check, workflow lookup ...

    let mut in_flight = state.in_flight.lock();
    if let Some(existing) = in_flight.get(&workflow_id).cloned() {
        drop(in_flight);  // release lock before publish
        publish_global(DomainEvent::WorkflowRunSkipped {
            workflow_id: workflow_id.clone(),
            reason: SkippedReason::AlreadyRunning,
            attempted_trigger_source: trigger_source,
        });
        tracing::info!(target: "workflows-skip",
            "wf={} already running (run={})", workflow_id, existing);
        return Err(DispatchError::AlreadyRunning { run_id: existing });
    }
    let run_id = uuid::Uuid::now_v7().to_string();
    in_flight.insert(workflow_id.clone(), run_id.clone());
    drop(in_flight);

    // Insert workflow_runs row, publish WorkflowRunStarted, spawn task...
    let slot = InFlightSlot { state: state.clone(), workflow_id: workflow_id.clone() };
    tokio::spawn(async move {
        let _slot_guard = slot; // moved in; Drop releases on any exit
        execute_inner(ctx, workflow_id, run_id).await;
    });
    Ok(run_id)
}
```

**Cancellation check in `execute_inner`:**

```rust
for node in &workflow.nodes {
    if ctx.workflows_store().is_cancelled(&run_id)? {
        tracing::info!(target: "workflows-run",
            "wf={} run={} cancelled mid-execution; skipping remaining nodes",
            workflow_id, run_id);
        return RunOutcome::Cancelled;
    }
    // ... execute node ...
}
```

## Edge Cases to Handle

1. **Slot leak on panic.** The `InFlightSlot` Drop impl handles this — tokio's `JoinHandle` will trigger Drop on the captured slot guard. Test this with a deliberately-panicking node.
2. **Cancel during in-flight check.** The user clicks cancel before `dispatch_run` even starts. Handle gracefully — return `Err(NotFound)` since no run row exists yet. The cancel-run RPC handler should accept this and surface "Already complete or never started" to the UI.
3. **Cancel of a run that just completed.** Race — the executor marked status=Succeeded a millisecond before the cancel arrived. `cancel_run` returns `Err(NotRunning { current_status: Succeeded })`. The UI shows a transient toast.
4. **`is_cancelled` lookup error.** If the DB query fails, treat as "not cancelled" but log a warning. Don't crash the run.
5. **Orphan sweep on an empty DB.** No-op; returns `Ok(0)`.
6. **Orphan sweep timing.** It runs BEFORE `reconcile_at_startup`. If it ran after, the scheduler would already have started ticking and could dispatch into a workflow whose previous run is still listed as Running. Document this ordering explicitly with a comment.
7. **`workflow_runs.error` column.** When marking Failed{CoreCrashed}, write `"CoreCrashed"` as the error string — keep it short and grep-friendly.
8. **Concurrent dispatch + cancel.** User triggers Run-Now and then immediately Cancel. `dispatch_run` may not have inserted the row yet. Result: cancel returns `NotFound`, then the run proceeds. Acceptable — document this race and recommend the UI debounces / disables Cancel until the dispatch confirms.
9. **`WorkflowRunSkipped` events in the run-history view.** Per NFR-2.4.5, these should appear as a distinct "Skipped (already running)" row. The UI work belongs to F-15 / a follow-up; F-9 ensures the events are published so the data is available.
