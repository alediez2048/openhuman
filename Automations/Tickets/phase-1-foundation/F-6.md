## What Is This Ticket?

Build the frontend Starter workflows catalog section on `/workflows`. Renders the response of `workflows_list_starter_templates` as catalog cards beneath the "Your workflows" section. Each card has [Add to my workflows] (calls `workflows_create` with template payload + `origin = Seed`) and [Add & Enable] (create then enable). Once added, the card disappears from the catalog automatically because of F-5's server-side dedup (FR-1.8.2/FR-1.8.3).

The empty state from F-4 promotes this section prominently when the user has zero workflows.

### Why It Matters

- This is the second user-visible creation path (after chat — F-12..F-14), and the easier one to ship. It validates the full create → enable → dedup → list-rerender loop without needing the agent runtime.
- Confirms F-5's dedup semantics work in production (catalog vanishes after [Add]; reappears after the user deletes the resulting workflow).
- Lands the per-card "missing connections" rendering — the same UX building block F-14's `<WorkflowProposalPreview>` will reuse for chat-driven proposals.

## What This Ticket Must Accomplish

### Goal

- `StarterWorkflowsSection.tsx` mounts under "Your workflows" on `/workflows` (replacing F-4's placeholder).
- On mount, dispatches `fetchStarterTemplates` to populate Redux.
- Renders one `StarterWorkflowCard` per response entry.
- Each card shows: name · description · `trigger_summary` · required-connection pills (missing ones get the amber `⚠` prefix) · [Add to my workflows] · [Add & Enable].
- Both Add buttons call `workflows_create` with `origin: { type: "seed", template_id }` and the `raw_payload` from the RPC response. [Add & Enable] follows with `workflows_enable`.
- After successful create, the catalog re-fetches automatically (the dedup is on the server, so the just-added template won't appear in the next response).
- The "Your workflows" list also re-fetches so the new row appears.
- Empty-state behavior on `/workflows`: when `workflows.length === 0`, the Starter section sits prominently *above* the chat CTA. When `workflows.length > 0`, the section sits below the user's list (FR-1.2.6).
- Hide-starter preference: a small "Hide starter workflows" link in the section header writes `hideStarterSection: true` to Redux (F-4 already declared the field). Hidden users can re-show via a Settings toggle (this ticket adds the Settings toggle).

## Suggested Implementation Pattern

**`app/src/components/workflows/StarterWorkflowCard.tsx` (extract):**

```tsx
export function StarterWorkflowCard({ template }: { template: StarterTemplateView }) {
  const t = useT();
  const dispatch = useAppDispatch();
  const [loading, setLoading] = useState<'add' | 'add-enable' | null>(null);
  const triggerSummary = useCronHumanizer(template.raw_payload.trigger);

  const handleAdd = async (enable: boolean) => {
    setLoading(enable ? 'add-enable' : 'add');
    try {
      await dispatch(addStarterTemplate({ template, enableImmediately: enable })).unwrap();
    } finally {
      setLoading(null);
    }
  };

  return (
    <div className="bg-white rounded-2xl shadow-soft border border-ocean-100 p-4">
      <h4 className="font-semibold text-stone-900">{template.name}</h4>
      <p className="text-sm text-stone-500 mt-1">{template.description}</p>
      <div className="flex flex-wrap gap-1 mt-2">
        {template.required_connections.map((ref, i) => (
          <ConnectionPill
            key={i}
            connectionRef={ref}
            missing={template.missing_connections.includes(ref)}
          />
        ))}
      </div>
      <div className="text-xs text-stone-500 mt-2">⏰ {triggerSummary}</div>
      <div className="flex gap-2 mt-4">
        <button
          className="btn-ghost"
          onClick={() => handleAdd(false)}
          disabled={loading !== null}
        >
          {loading === 'add' ? t('workflows.adding') : t('workflows.add_to_my_workflows')}
        </button>
        <button
          className="btn-primary"
          onClick={() => handleAdd(true)}
          disabled={loading !== null}
        >
          {loading === 'add-enable' ? t('workflows.adding') : t('workflows.add_and_enable')}
        </button>
      </div>
    </div>
  );
}
```

## Edge Cases to Handle

1. **`workflows_list_starter_templates` fails (offline / RPC error).** Show a section-level error state with a Retry button. Don't crash the page.
2. **[Add] succeeds but [enable] fails on [Add & Enable].** The workflow exists (paused) but the enable failed — surface a toast: "Added, but enable failed: <reason>. Toggle it manually from /workflows." Re-fetch both lists.
3. **`workflows_create` rejects on a malformed payload (shouldn't happen — bundled JSON is validated).** Surface a generic error toast; log to console; this is a bug, log a Sentry breadcrumb.
4. **Double-click on [Add] button.** Disabled state + `loading` guard prevents the second call.
5. **`useCronHumanizer` for `manual` triggers.** Return "Run on demand" rather than empty string.
6. **Missing-connection pill click.** Optional — wire it to navigate to `/connections` with a focus filter on the missing provider. Out of scope if it adds time; document as a follow-up.
7. **Catalog shows `[]` because user added all four.** Render `t('workflows.starter_empty_catalog')` once, not a blank section.
8. **`hideStarterSection` + zero workflows.** Per FR-1.2.6, ignore the preference — the catalog still shows. Test this explicitly.
9. **Catalog re-fetch race.** If `fetchWorkflows` and `fetchStarterTemplates` both kicked off and the catalog response arrives first with stale dedup data, the user sees the just-added template for ~100ms. Acceptable; document.
