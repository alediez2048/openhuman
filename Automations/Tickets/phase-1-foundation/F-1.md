## What Is This Ticket?

Create the `src/openhuman/workflows/` Rust domain — the foundation for the Phase 1 Workflows feature. This ticket establishes the module layout, the full type universe from `systemsdesign.md §2.2/§2.3` (`Workflow`, `Trigger`, `Node`, `Edge`, `Run`, `RunStep`, `WorkflowHealth`, `WorkflowOrigin`, `NodeKind`, `NodeConfig`, `AgentPromptConfig`, `RunStatus`, `TriggerSource`, `OnErrorPolicy`, plus the proposal types), the SQLite store at `${OPENHUMAN_WORKSPACE}/workflows.db` with three migrations, and the empty stubs for every subsequent file in the domain.

**Crucially: no RPC handlers, no scheduler, no executor, no UI** — only scaffolding, types, migrations, and the `DomainEvent` enum extension. F-2 through F-15 will fill in the empty stubs one at a time.

### Why It Matters

- Every subsequent Phase 1 ticket (F-2 through F-15) depends on this scaffolding existing.
- The two-database architecture (`connections.db` from Phase 0 + `workflows.db` from this ticket) is locked by ADR-003. Creating `workflows.db` correctly here means the rest of Phase 1 doesn't have to retrofit the file boundary.
- Locks the full type vocabulary into the codebase in one commit. F-2 onwards can build on a stable shape without redefining types.
- Mirrors the precedent set by `src/openhuman/connections/` from P0-1 — same module layout, same migration pattern.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| Separate `workflows.db` from `connections.db` | ADR-003 | Clean domain ownership; rollback friendly |
| `WorkflowHealth` is persisted, recomputed on connection events | ADR-017 | Fast list-view reads; recompute is event-driven (F-3) |
| `WorkflowOrigin` discriminator with `Seed { template_id }` carrying the id | ADR-018 | Enables catalog dedup query in F-5 without inference |
| `ProposalValidationError` is a structured enum (not opaque string) | ADR-019 | Telemetry per variant + surgical retry prompts (F-11) |
| Single-flight invariant + `cancelled` flag on `workflow_runs` | ADR-014 | Schema column ready for F-9 to enforce |
| All 9 `NodeKind` variants declared from day one | systemsdesign §2.2 | Forward compat for Phase 2/3 without schema bumps |

    JsonParse           { reason: String },
    UnknownConnection   { r#ref: ConnectionRef, candidates: Vec<ConnectionRef> },

            Self::InvalidCron { .. }           => "invalid_cron",
            Self::EdgeIntegrity { .. }         => "edge_integrity",
            Self::MissingRequiredField { .. }  => "missing_required_field",

## Edge Cases to Handle

1. **Workspace dir doesn't exist.** `WorkflowsStore::open` creates parent directories or returns a clear error. Match the connections store's behavior.
2. **Partial migration failure.** Wrap each migration in its own transaction; a failed migration leaves the DB unchanged so retry on next startup works.
3. **`schema_migrations` table doesn't exist yet on first boot.** `apply_migrations` must `CREATE TABLE IF NOT EXISTS schema_migrations` before checking for applied versions.
4. **Foreign-key cascades.** SQLite disables foreign keys by default. Set `PRAGMA foreign_keys = ON` on each connection so `workflow_runs` → `workflows` and `workflow_run_steps` → `workflow_runs` cascades work.
5. **Concurrent core startups against the same workspace.** SQLite file-level locking handles this; the open call may block briefly. Acceptable.
6. **Tests on Windows.** Use `tempfile::TempDir` and a `Path` rather than hard-coded `/tmp/…` paths.
7. **`ConnectionRef` import cycle.** `workflows/types.rs` imports `ConnectionRef` from `connections/types.rs` — that's fine, but `connections/` must NOT import from `workflows/` (no back-edge). Verify with `cargo check`.
8. **`Trigger::Webhook { tunnel_uuid: Uuid }` and the `uuid` crate.** Use the existing workspace `uuid` dep with the `serde` feature already enabled.
9. **JSON column shape.** `health TEXT NOT NULL` stores the JSON-encoded `WorkflowHealth`. The schema indexes on `health` work because SQLite indexes whole-column text values — the discriminator-key-prefix sort suffices for `WHERE health LIKE '%"needs_connections"%'`-style queries later.
