---

## What Is This Ticket?

Fill the empty `src/openhuman/workflows/executor.rs` with the full Phase 1 execution pipeline:

1. `dispatch_run(workflow_id, trigger_source)` creates a `workflow_runs` row, returns its `RunId`, and spawns the run on a tokio task.
2. `execute_inner` walks the node graph (Phase 1: exactly one `agent_prompt` node), awaits `scheduler_gate`, calls the per-node executor.
3. `agent_prompt` executor builds an `AgentDefinition` via `build_node_agent_definition` (the allowlist function from ADR-016 — baseline + `allowed_connections` + 4 read-only workflow tools). Runs `agent::run_subagent`. Persists the result as a `RunStep` (output truncated to 64 KiB per NFR-2.3.5). Marks the run `Succeeded`, `Failed`, or `TimedOut`.
4. Publishes `WorkflowRunStarted`, `WorkflowRunStepStarted`, `WorkflowRunStepCompleted`, `WorkflowRunCompleted` events at the right moments.

**Not in this ticket:** the single-flight `in_flight` HashMap + orphan recovery sweep + soft-cancel (F-9). F-8 calls into a stubbed `ExecutorState` and F-9 makes the registry real. **Not in this ticket:** the agent-tool allowlist *enforcement test* — that goes in F-10 (which also defines the read-only workflow tools the allowlist references).

### Why It Matters

- Without an executor, a cron tick fires a workflow but nothing actually happens. F-8 is the moment "Workflows" go from a CRUD database to a working feature.
- The `build_node_agent_definition` function is the security boundary from ADR-016 / NFR-2.3.7. F-8 implements it; F-10 adds the unit test asserting the allowlist is correct.
- `scheduler_gate` integration ensures workflow runs respect the same cooperative-shutdown semantics as every other long-running task in OpenHuman.

## What This Ticket Must Accomplish

### Goal

- `WorkflowsStore` extended with run-row CRUD: `insert_run`, `update_run_status`, `insert_run_step`, `update_run_step`, `mark_run_terminal`.
- `executor::dispatch_run(ctx, workflow_id, trigger_source) -> Result<RunId, DispatchError>`:
    1. Look up workflow; reject `NotFound`.
    2. Reject if `health != Ready` with `DispatchError::HealthBlocked { health }`.
    3. (F-9 will add single-flight check here.)
    4. Create `workflow_runs` row with `status = Running`, `started_at = now`, `trigger_source`.
    5. Publish `WorkflowRunStarted { workflow_id, run_id }`.
    6. Spawn `execute_inner` on a tokio task.
    7. Return `RunId` immediately.
- `executor::execute_inner(ctx, workflow_id, run_id) -> RunOutcome`:
    1. Await `scheduler_gate::wait_ready().await`.
    2. Apply the per-workflow wall-clock timeout (default 5 min, max 1 hour per FR-1.6.5) via `tokio::time::timeout`.
    3. Walk `workflow.nodes` in order. Phase 1: exactly one node, hard-fail with `Err` if more than one node is present (link to F-15's capability-catalog DEVLOG explaining the Phase 1 limitation).
    4. For each node, call the per-kind executor. Phase 1 only supports `NodeKind::AgentPrompt`.
    5. Mark run terminal: `Succeeded`, `Failed { error_summary }`, or `TimedOut`. Publish `WorkflowRunCompleted`.
- `executor::execute_agent_prompt(ctx, run_id, node, config)`:
    1. Publish `WorkflowRunStepStarted { run_id, node_id }`.
    2. Insert `workflow_run_steps` row with `status = Running`.
    3. Build `AgentDefinition` via `build_node_agent_definition(allowed_connections)`.
    4. Call `agent::run_subagent(definition, &config.prompt, parent_context)`.
    5. On success: truncate output to 64 KiB; persist via `update_run_step`; publish `WorkflowRunStepCompleted { status: Succeeded }`.
    6. On error: persist error string; publish `WorkflowRunStepCompleted { status: Failed }`. Per FR-1.6.4 (Phase 1 hard-codes `on_error = Halt`), abort the run.
- `executor::build_node_agent_definition(allowed_connections) -> AgentDefinition` returns an `AgentDefinition` whose `allowed_tools` is exactly:
    - **Baseline tools** — memory, web_search, time, etc. (the same baseline `agent::run_subagent` uses elsewhere; pull from the canonical baseline list defined in `src/openhuman/agent/`).
    - **Resolved connection tools** for each `ConnectionRef` in `allowed_connections` (resolve via the `tool_registry` per ConnectionRef → tool name mapping that already exists for the main agent).
    - **The 4 read-only workflow tools:** `workflow_list`, `workflow_get`, `workflows_list_runs`, `workflows_get_run`. (F-10 implements these tools and registers them. F-8 references them by name — coordinate with F-10 so the names match.)
    - **Excluded:** every `workflow_propose_*`, every mutating workflow tool, every non-baseline agent tool.
- Define `executor::ExecutorState { /* in_flight stub for F-9, store handle, runtime handles */ }`. The `in_flight: HashMap<WorkflowId, RunId>` field exists but F-8 does not consult it; F-9 makes it functional.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| `agent_prompt` is the only Phase 1 node kind | FR-1.5.1.1 | Other kinds reserved for Phase 2; executor rejects them with clear errors |
| Allowlist constructed in `build_node_agent_definition` | ADR-016 / NFR-2.3.7 | Single function = single test surface in F-10 |
| 64 KiB truncation on step output | NFR-2.3.5 | UTF-8-safe truncation; document the exact boundary rule |
| 5-min default timeout, 1-hr max | FR-1.6.5 | Read from `workflow.settings.timeout_secs`; clamp to [1s, 3600s] |
| Phase 1 hard-codes `on_error = Halt` | FR-1.6.4 | `OnErrorPolicy::Continue` declared but ignored; document |
| Run runs through `scheduler_gate::wait_ready` | FR-1.6.6 | Same gate as agent's main loop |
| Single-flight + orphan sweep are F-9 | ADR-014 | F-8 leaves the `in_flight` field unused but present |

## Suggested Implementation Pattern

**`src/openhuman/workflows/executor.rs` (extract):**

```rust
pub async fn dispatch_run(
    ctx: &Ctx,
    workflow_id: WorkflowId,
    trigger_source: TriggerSource,
) -> Result<RunId, DispatchError> {
    let workflow = ctx.workflows_store().get_workflow(&workflow_id)?
        .ok_or(DispatchError::NotFound)?;
    if !matches!(workflow.health, WorkflowHealth::Ready) {
        return Err(DispatchError::HealthBlocked { health: workflow.health });
    }
    // F-9: single-flight check goes here.

    let run_id = uuid::Uuid::now_v7().to_string();
    let run = Run {
        id: run_id.clone(),
        workflow_id: workflow_id.clone(),
        trigger_source: trigger_source.clone(),
        status: RunStatus::Running,
        started_at: chrono::Utc::now(),
        completed_at: None,
        error: None,
        cancelled: false,
    };
    ctx.workflows_store().insert_run(&run)?;
    publish_global(DomainEvent::WorkflowRunStarted {
        workflow_id: workflow_id.clone(),
        run_id: run_id.clone(),
    });
    tracing::info!(target: "workflows-run",
        "dispatch_run wf={workflow_id} run={run_id} source={trigger_source:?}");

    let ctx2 = ctx.clone();
    tokio::spawn(execute_inner(ctx2, workflow_id, run_id.clone()));
    Ok(run_id)
}

pub fn build_node_agent_definition(allowed_connections: &[ConnectionRef]) -> AgentDefinition {
    let mut tools = baseline_tool_names().to_vec();
    tools.extend(allowed_connections.iter().filter_map(resolve_connection_tools).flatten());
    tools.extend([
        "workflow_list",
        "workflow_get",
        "workflows_list_runs",
        "workflows_get_run",
    ]);
    AgentDefinition {
        allowed_tools: tools,
        // … other fields from existing agent definition shape
    }
}
```

**Truncation helper:**

```rust
const MAX_OUTPUT_BYTES: usize = 64 * 1024;

fn truncate_output(s: String) -> String {
    if s.len() <= MAX_OUTPUT_BYTES { return s; }
    let mut cut = MAX_OUTPUT_BYTES;
    while !s.is_char_boundary(cut) { cut -= 1; }
    let mut out = s[..cut].to_string();
    out.push_str("\n…[truncated]");
    out
}
```

## Edge Cases to Handle

1. **Workflow with multiple nodes.** Phase 1 supports exactly one node. If `workflow.nodes.len() != 1`, `execute_inner` returns `Err(NodeError::PhaseConstraint { phase: 1 })`. The validator (F-11) prevents this at create time; F-8 belts-and-suspenders the runtime check.
2. **`agent::run_subagent` panics.** Tokio's `JoinError::Panic` surfaces — mark the run `Failed { reason: "Agent task panicked: …" }`. Catch via `tokio::spawn(...).await`-with-`?` pattern.
3. **`scheduler_gate::wait_ready` is cancelled.** Treat as `Cancelled` status; release any held resources.
4. **Timeout fires mid-LLM-call.** Per FR-1.6.5, hard-kill. `tokio::time::timeout` wrapping `execute_inner` does this — the inner future is dropped, which cancels the agent task.
5. **Empty output from the agent.** Persist empty string — not NULL; downstream tooling (run history view) treats null as "still pending."
6. **64 KiB cap.** UTF-8 boundary check is critical — splitting in the middle of a multibyte char produces invalid UTF-8 that SQLite will reject.
7. **`allowed_connections` references a connection the user just disconnected.** The dispatch precondition (`health == Ready`) catches this. But there's a race: connection removed between health-check and `build_node_agent_definition`. The agent tool call fails at runtime — record the error in the step and fail the run.
8. **Run id collision** (UUIDv7 — extremely rare). Treat as a hard error; surface clearly.
9. **Concurrent `dispatch_run` calls for different workflows.** Allowed — single-flight is per-workflow (F-9). Test this with two simultaneous dispatches.
10. **`workflows_get_run` on a still-Running run.** Return the run with `Vec<RunStep>` containing partial step rows (status = Running). The UI can poll until terminal.
