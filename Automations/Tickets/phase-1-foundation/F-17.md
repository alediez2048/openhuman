## What Is This Ticket?

OpenHuman's product spine is **"memory + doer"** — a personal AI agent that learns from you, grows with you, and uses local-first Memory Tree as the substrate of intelligence. Phase 1 shipped the **doer** half of that spine (the Workflows runtime) but **not the memory loop**. Every workflow run today is stateless: the executor doesn't read prior-run context before invoking the agent, and the agent has no instinct to persist its outcomes when it's done. F-17 closes that gap.

Without F-17 a daily 8am "Morning email digest" workflow is just as smart on day 30 as on day 1 — it doesn't remember what it sent yesterday, doesn't know which categories you've consistently asked to skip, doesn't accumulate the cross-run context that would make it actually feel like a learning system. With F-17, every workflow run is **(read prior context) → (do the work) → (write a structured outcome)** so the next run starts with everything the prior runs already knew.

### Concrete behavior change

**Before F-17:**
```text
[cron tick] → executor.dispatch_run → workflow_node agent (with empty context) → tools fire → run ends, no memory write
[same workflow tomorrow] → same empty context → agent has zero awareness of yesterday's run
```

**After F-17:**
```text
[cron tick] → executor.dispatch_run → memory_recall("workflow:<id>", scope=last_3_runs + tagged) → workflow_node agent
   (system prompt includes "## Prior runs of this workflow" section)
   → tools fire → memory_store(structured summary keyed by workflow_id + run_id) → run ends
[same workflow tomorrow] → memory_recall returns yesterday's summary → agent skips already-sent items, applies learned filters, builds on prior state
```

### Why It Matters

This is the most under-emphasized gap in the whole initiative. Three failure modes it fixes:

1. **Duplicate work.** Today's "Meeting email → Calendar + Slack" workflow would happily re-create a Calendar event for the same email thread tomorrow because the runtime never recorded that it already handled it. Memory-loop fixes this.
2. **No learned preferences.** A user says "stop including UnitedHealthcare promos in the digest" in chat — the next workflow run doesn't know. F-17's memory-store/recall is the channel for that feedback to flow back into the workflow.
3. **No compounding intelligence.** OpenHuman's positioning ("the memory and doer for everything you do") implies the doer half compounds via memory. Without F-17, workflows are an n8n clone with a chat front-end. With F-17, they're the spine of the memory-doer feedback loop.

The user explicitly agreed during the 2026-05-22 grill that workflow memory is key for OpenHuman's performance. F-17 is the concrete answer to that.

## What Was Already Done

- **F-16 (`3b572f71`)** wired `memory_recall` + `memory_store` into the workflow_node's `BASELINE_TOOL_NAMES`. Tools are available to the agent. **But nothing forces the agent to use them**, and the executor doesn't auto-recall before the agent starts.
- **`crate::openhuman::memory::Memory` trait** (`src/openhuman/memory/traits.rs`) exposes `async fn store(...)` and `async fn recall(...)`. The executor can call these directly without going through the LLM-driven tool path.
- **Memory tree structure** — local-first SQLite, organised as per-source / per-topic / per-day chunks. F-17 introduces workflow-keyed memory as a new addressable slice.
- **F-8's `event_context = "workflow:<run_id>"`** tag — already used by F-16's tool-failure counter and (planned) F4-5's live-preview streaming. F-17 uses the same scope.

## What This Ticket Must Accomplish

### Goal

Two automatic memory hooks in the workflow executor + prompt-level reinforcement so the agent collaborates with the auto-injection rather than working around it.

### Deliverables Checklist

#### A. Memory schema for workflow runs

- [ ] Define a stable memory address pattern: `workflow:{workflow_id}` is the **topic**; each run writes a chunk tagged `run:{run_id}` + timestamp.
- [ ] `WorkflowRunMemory` struct in `src/openhuman/workflows/memory.rs`. **Ground-truth-first split** (the agent's text is one signal, the runtime's tool-call trace is the truth; future runs see both, trust the latter):
  ```rust
  pub struct WorkflowRunMemory {
      pub workflow_id: WorkflowId,
      pub run_id: RunId,
      pub triggered_at: DateTime<Utc>,
      pub trigger_source: TriggerSource,
      pub status: RunStatus,                 // F-16 honest-status; the trace-derived terminal status

      // ── Ground truth (built from the F-16 event-bus tap, NOT from the agent's text) ──
      pub actual: ActualOutcome,             // structured, runtime-authored, source-of-truth

      // ── Narrative (the agent's story about what happened) ──
      pub narrative: String,                 // 1–3 sentence agent-authored final response

      // ── Cross-check ──
      pub narrative_matches_actual: bool,    // true iff narrative's claims line up with `actual`
      pub narrative_drift: Vec<String>,      // when false, list the specific claims that diverged
                                             // e.g. "Narrative claims Slack message sent;
                                             //       actual shows composio_execute SLACK_SEND_MESSAGE success=false"

      // ── Entity tagging (forward-compat for Phase 5 structured business entities) ──
      pub entity_tags: Vec<String>,          // e.g. ["entity:source:stripe",
                                             //       "entity:lead:acme-corp",
                                             //       "entity:deal:acme-q3-2026"]
                                             // Runtime auto-tags by connection (`entity:source:<toolkit>`);
                                             // agent can add domain-specific tags via the prompt convention.
                                             // Lets Phase 5 build a structured query layer on top of
                                             // memory chunks without committing to schemas today.
  }

  pub struct ActualOutcome {
      pub tool_calls: Vec<ToolCallTrace>,    // every call from the F-16 event-bus tap,
                                             // with success/failure as observed by the harness
      pub side_effects_confirmed: Vec<String>,  // outcomes the trace PROVES happened
                                                // ("Sent Slack DM via SLACK_SEND_MESSAGE — message_id ts_1234")
      pub side_effects_claimed_unverified: Vec<String>,  // outcomes the agent claimed but the trace can't confirm
                                                         // (kept for the cross-check, NOT promoted to "happened")
      pub anomalies: Vec<String>,            // errors, denials, cost-cap trips, dry-runs
  }

  pub struct ToolCallTrace {
      pub tool_name: String,
      pub redacted_args: serde_json::Value,
      pub success: bool,
      pub elapsed_ms: u64,
      pub inner_status: Option<String>,      // F-16's inner-status payload (e.g. "cost_cap_exceeded")
  }
  ```
- [ ] `ToolCallTrace` redaction reuses the F4-6-planned redaction policy when it exists; for Phase 1.5 the redaction is a simple regex pass (no password-field detection; that ships with F4-6).
- [ ] **Cross-check semantics:** the runtime builds `actual` from the F-16 event-bus tap (extended in deliverable C to record per-call detail). The narrative-vs-actual diff is computed by a simple heuristic: extract verb-phrases from the agent's text ("sent", "created", "skipped"); for each, check whether a matching successful tool call exists in `actual.tool_calls`. Mismatches become `narrative_drift` entries. The full implementation is intentionally simple (regex + substring); a smarter version is a future ticket.
- [ ] **`status` comes from F-16's honest-status gate, not the narrative.** A run where the agent claims "Done!" but `tool_failure_count > 0` is Failed.
- [ ] `Memory::store` call uses key shape `workflow:{workflow_id}` + chunk text serialised from `WorkflowRunMemory` as canonical Markdown (matches the existing memory pipeline's per-source/per-topic structure).

#### B. Pre-run memory recall (executor)

- [ ] `executor::run_agent_prompt` (or its caller `execute_agent_prompt_node`) calls `Memory::recall("workflow:{workflow_id}", limit=3, recency=true)` BEFORE building the workflow_node agent.
- [ ] Results are formatted as a Markdown block. Ground-truth-first: the **Actual** line is always rendered from `actual`; the agent's narrative is rendered as a separate, clearly-labelled `Narrative` line ONLY when it doesn't drift from actual. When `narrative_matches_actual = false`, the narrative is REPLACED with the explicit drift annotation so the next-run agent sees the prior failure honestly instead of inheriting confabulation.
  ```markdown
  ## Prior runs of this workflow

  - **2026-05-21 08:00** (Succeeded) — Actual: fetched 12 unread Gmails (GMAIL_FETCH_EMAILS success); sent SLACK_SEND_MESSAGE success ts_8821. Skipped 8 emails (newsletters/promos) per agent's classification.
  - **2026-05-20 08:00** (Succeeded) — Actual: fetched 8; sent SLACK_SEND_MESSAGE success ts_8740.
  - **2026-05-19 08:00** (Failed) — Actual: fetched 8; SLACK_SEND_MESSAGE FAILED with rate_limit_exceeded. ⚠ Narrative drift: agent's text claimed "digest sent" but the tool call didn't succeed — DO NOT assume yesterday's digest reached the user.
  ```
- [ ] The Markdown block is prepended to the `agent_prompt_config.prompt` (user-message preamble; doesn't touch system prompt to keep the prefix cache stable).
- [ ] When the recall returns empty (first-ever run of this workflow), prepend a single line: "## No prior runs — this is the first execution." Helps the LLM understand it's not missing context.
- [ ] Token budget: cap the recall block at 1200 tokens (~ same tier as F4-2's "compact" snapshot). Drop oldest entries if over.
- [ ] Recall is best-effort — if the memory call fails, log a warn + proceed without memory; do NOT fail the run.

#### C0. Entity-tagging convention (forward-compat Phase 5 hook)

- [ ] **Auto-tagging from connections.** Executor inspects the workflow's `allowed_connections` and emits a tag per source: `ConnectionRef::Composio { toolkit_id }` → `entity:source:{toolkit_id}` (e.g. `entity:source:stripe`); `ConnectionRef::Channel { provider }` → `entity:source:{provider}`. Always-included; no opt-out.
- [ ] **Agent-authored tags.** The workflow_node prompt (deliverable D) teaches the agent to optionally emit a structured `## Entities touched` section at the end of its final response, lines like `- entity:lead:acme-corp` / `- entity:deal:acme-q3-2026` / `- entity:proposal:p-2026-018`. The runtime parses that section into `entity_tags` and merges with auto-tags. **Parsing is best-effort + tolerant** — a missing/malformed section just means no agent-authored tags, not a failure.
- [ ] **No schema validation yet.** Tags are free-form `entity:<kind>:<id>` strings. Phase 5 will eventually enforce a vocabulary; F-17 ships the convention without the enforcement so we can observe what tags actually emerge in real use before committing to schemas. The convention is documented in the workflow_node prompt + the drafter prompt so it's discoverable.
- [ ] **Memory storage uses `entity_tags` as Memory-Tree topic tags** so Phase 5's structured query layer can filter chunks by entity without scanning every workflow run. Concretely: a chunk's storage metadata includes the tag list; future `Memory::query_by_tag(...)` (Phase 5 ticket) reads them.

## Architectural Decisions

### Why ground-truth-first (`actual` from the event-bus tap, `narrative` as a separate field)?

The agent's final text response is its STORY about what happened, not WHAT happened. The runtime knows the truth: every tool call goes through F-16's event-bus tap with a hard `success: bool`. If the agent says "I sent the Slack message" but the trace shows the send failed, storing the agent's text as the run summary would make every subsequent run inherit the confabulation. F-17 splits the two: `actual` is built from the trace (source of truth); `narrative` is the agent's text (signal for human-readable summary + cross-check). The cross-check (`narrative_drift`) is what surfaces lying to the next run.

This costs ~0.5 day extra over the naive "agent text IS the summary" version. The cost is paid once; the system stops compounding lies forever.

### Why a heuristic drift detector, not a smarter classifier?

A regex/substring drift detector misses subtle cases ("sent" vs "shared", "created" vs "made") but catches the high-value lies cheaply. A smarter classifier would need another LLM call per run, adding latency + cost. Phase 1.5 keeps the detector simple; a future ticket can add semantic similarity if the heuristic misses too much.

### Why automatic runtime recall, not let the LLM choose?

Two reasons. First, the LLM doesn't reliably remember to call `memory_recall` first — F-16's lessons taught us the agent's discipline isn't enough. Build the discipline into the runtime. Second, prepending to the user-message keeps the system-prompt prefix-cache stable. If we made memory_recall a tool call, every prompt would shift mid-conversation and we'd lose the cache hit.

### Why pre-prepend to user-message, not the system prompt?

System prompt is per-archetype (the workflow_node TOML). Per-run dynamic context is per-user-message. Mixing them risks (a) cache misses, (b) confused identity ("am I a workflow_node or am I a thing-that-knows-about-runs?"). The workflow_node prompt teaches the AGENT how to use the recall; the recall content itself lives in the user message.

### Why automatic runtime store, not LLM-driven memory_store?

The agent's final response IS the summary the next run needs to see. Asking the LLM to ALSO call memory_store doubles the work. The runtime takes the response, structures it, writes it. The LLM is told (in the prompt update) that it doesn't need to call `memory_store` for run summaries — only for cross-workflow learning (e.g. discovered user preferences that should apply broadly).

### What about per-workflow vs cross-workflow memory?

F-17 ships **per-workflow** memory (keyed by `workflow:{workflow_id}`). Cross-workflow memory (e.g. "this user prefers concise digests" applying across every digest-style workflow) is left to the agent to opt into via explicit `memory_store` calls. The runtime doesn't auto-aggregate cross-workflow because we don't yet know the right aggregation granularity — observation period, then design.

### How does this interact with the F-16 honest-status counter?

The post-run store reads from F-16's event-bus tap to extract tool-call results (success/failure, name, redacted args). F-16's counter today only counts; F-17 extends the counter to ALSO record a small per-call summary that gets fed into the memory chunk. Net effect: F-16's gate stays unchanged; F-17 reuses the same event subscription for richer telemetry.

### Privacy

Workflow memory includes summaries of what happened — possibly with PII (email senders, Slack channel names, etc.). The Phase 1.5 redaction is a simple regex pass; F4-6's richer redaction policy (planned) replaces this when it lands. Memory is local-first per OpenHuman's design — never sent to a third party — so the bar is "auditable", not "encrypted at rest beyond the user's filesystem".
