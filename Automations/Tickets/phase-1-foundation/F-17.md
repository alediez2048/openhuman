## What Is This Ticket?

OpenHuman's product spine is **"memory + doer"** — a personal AI agent that learns from you, grows with you, and uses local-first Memory Tree as the substrate of intelligence. Phase 1 shipped the **doer** half of that spine (the Workflows runtime) but **not the memory loop**. Every workflow run today is stateless: the executor doesn't read prior-run context before invoking the agent, and the agent has no instinct to persist its outcomes when it's done. F-17 closes that gap.

Without F-17 a daily 8am "Morning email digest" workflow is just as smart on day 30 as on day 1 — it doesn't remember what it sent yesterday, doesn't know which categories you've consistently asked to skip, doesn't accumulate the cross-run context that would make it actually feel like a learning system. With F-17, every workflow run is **(read prior context) → (do the work) → (write a structured outcome)** so the next run starts with everything the prior runs already knew.

### Concrete behavior change

**Before F-17:**
```text
[cron tick] → executor.dispatch_run → workflow_node agent (with empty context) → tools fire → run ends, no memory write
[same workflow tomorrow] → same empty context → agent has zero awareness of yesterday's run
```

**After F-17:**
```text
[cron tick] → executor.dispatch_run → memory_recall("workflow:<id>", scope=last_3_runs + tagged) → workflow_node agent
   (system prompt includes "## Prior runs of this workflow" section)
   → tools fire → memory_store(structured summary keyed by workflow_id + run_id) → run ends
[same workflow tomorrow] → memory_recall returns yesterday's summary → agent skips already-sent items, applies learned filters, builds on prior state
```

### Why It Matters

This is the most under-emphasized gap in the whole initiative. Three failure modes it fixes:

1. **Duplicate work.** Today's "Meeting email → Calendar + Slack" workflow would happily re-create a Calendar event for the same email thread tomorrow because the runtime never recorded that it already handled it. Memory-loop fixes this.
2. **No learned preferences.** A user says "stop including UnitedHealthcare promos in the digest" in chat — the next workflow run doesn't know. F-17's memory-store/recall is the channel for that feedback to flow back into the workflow.
3. **No compounding intelligence.** OpenHuman's positioning ("the memory and doer for everything you do") implies the doer half compounds via memory. Without F-17, workflows are an n8n clone with a chat front-end. With F-17, they're the spine of the memory-doer feedback loop.

The user explicitly agreed during the 2026-05-22 grill that workflow memory is key for OpenHuman's performance. F-17 is the concrete answer to that.

## What Was Already Done

- **F-16 (`3b572f71`)** wired `memory_recall` + `memory_store` into the workflow_node's `BASELINE_TOOL_NAMES`. Tools are available to the agent. **But nothing forces the agent to use them**, and the executor doesn't auto-recall before the agent starts.
- **`crate::openhuman::memory::Memory` trait** (`src/openhuman/memory/traits.rs`) exposes `async fn store(...)` and `async fn recall(...)`. The executor can call these directly without going through the LLM-driven tool path.
- **Memory tree structure** — local-first SQLite, organised as per-source / per-topic / per-day chunks. F-17 introduces workflow-keyed memory as a new addressable slice.
- **F-8's `event_context = "workflow:<run_id>"`** tag — already used by F-16's tool-failure counter and (planned) F4-5's live-preview streaming. F-17 uses the same scope.

## Architectural Decisions

### Why automatic runtime recall, not let the LLM choose?

Two reasons. First, the LLM doesn't reliably remember to call `memory_recall` first — F-16's lessons taught us the agent's discipline isn't enough. Build the discipline into the runtime. Second, prepending to the user-message keeps the system-prompt prefix-cache stable. If we made memory_recall a tool call, every prompt would shift mid-conversation and we'd lose the cache hit.

### Why pre-prepend to user-message, not the system prompt?

System prompt is per-archetype (the workflow_node TOML). Per-run dynamic context is per-user-message. Mixing them risks (a) cache misses, (b) confused identity ("am I a workflow_node or am I a thing-that-knows-about-runs?"). The workflow_node prompt teaches the AGENT how to use the recall; the recall content itself lives in the user message.

### Why automatic runtime store, not LLM-driven memory_store?

The agent's final response IS the summary the next run needs to see. Asking the LLM to ALSO call memory_store doubles the work. The runtime takes the response, structures it, writes it. The LLM is told (in the prompt update) that it doesn't need to call `memory_store` for run summaries — only for cross-workflow learning (e.g. discovered user preferences that should apply broadly).

### What about per-workflow vs cross-workflow memory?

F-17 ships **per-workflow** memory (keyed by `workflow:{workflow_id}`). Cross-workflow memory (e.g. "this user prefers concise digests" applying across every digest-style workflow) is left to the agent to opt into via explicit `memory_store` calls. The runtime doesn't auto-aggregate cross-workflow because we don't yet know the right aggregation granularity — observation period, then design.

### How does this interact with the F-16 honest-status counter?

The post-run store reads from F-16's event-bus tap to extract tool-call results (success/failure, name, redacted args). F-16's counter today only counts; F-17 extends the counter to ALSO record a small per-call summary that gets fed into the memory chunk. Net effect: F-16's gate stays unchanged; F-17 reuses the same event subscription for richer telemetry.

### Privacy

Workflow memory includes summaries of what happened — possibly with PII (email senders, Slack channel names, etc.). The Phase 1.5 redaction is a simple regex pass; F4-6's richer redaction policy (planned) replaces this when it lands. Memory is local-first per OpenHuman's design — never sent to a third party — so the bar is "auditable", not "encrypted at rest beyond the user's filesystem".
