## What Is This Ticket?

F-18 fixed the silent-wrong-user-write bug that caused two days of "fresh API key → paste → fail → fresh API key" loops with Higgsfield. **The day F-18 landed, the user immediately hit a NEW MCP failure mode**: same surface symptom ("Higgsfield doesn't work"), totally different root cause (endpoint URL pointed at the server's root instead of `/mcp`).

What made it brutal:

- **The chat agent invented errors.** Repeated user-visible string: "POST / returns 404 from Higgsfield". This was a **fabrication** — the runtime hadn't reloaded the config since the previous boot, so the agent had no actual probe result to report. The LLM pattern-matched on the diagnostic conversation and generated plausible-sounding 404 errors.
- **The user had no way to know the endpoint was wrong** until manually `curl`-ing every plausible path. The MCP-add modal happily saved any URL without ever testing it.
- **The "Add MCP server" UX assumes the user knows the exact MCP endpoint path** — but every server is different: Higgsfield is `/mcp`, GitBook is `/~gitbook/mcp`, Linear is `/sse`, others are `/messages`. No standard.

F-19 closes both failure modes:

1. **MCP tool errors are structured.** The orchestrator no longer interprets opaque strings into invented HTTP status codes.
2. **MCP add auto-probes** common endpoint paths and saves the one that actually responds to `initialize`. The user pastes the root URL, we figure out the rest.

Part 3 (catalog) is the stretch goal — it's polish, not pain relief.

### Repro that drove this ticket

During F-17 live verification + F-18 fix-up:

Total time burned on what was a one-line config typo: ~3 hours across two sessions.

## What Was Already Done

- **F-18** landed the stale-session-handle guard + orphan migration RPCs. MCP server registration can no longer silently land in the wrong user dir.
- **MCP client at `src/openhuman/mcp_client/`** wraps streamable-HTTP transport against registered servers; the failure modes (DNS error, HTTP 401, HTTP 404, transport mismatch) currently collapse to free-form `anyhow::Error` strings on the tool-result path.
- **`add_mcp_server` at `src/openhuman/connections/ops.rs:429`** persists the user's endpoint verbatim — no probe, no validation beyond `starts_with("http")`.
- **Orchestrator prompt** at `src/openhuman/agent/agents/orchestrator/prompt.md` has no discipline against confabulating tool-failure details — the LLM sees a free-form string and writes plausible-sounding prose.

## What This Ticket Must Accomplish

### Goal

A user pastes a URL into "Add MCP server" and it either works or fails with a clear actionable error. Tool failures during chat surface the real cause — not the LLM's hallucinated guess.

### Deliverables Checklist

#### Part 1 — Structured MCP tool errors end-to-end (HIGHEST priority)

The pain: the chat agent invents HTTP status codes when MCP tool calls fail. Fix the structural cause: tool errors must carry a structured `kind` the orchestrator surfaces verbatim instead of feeding the LLM a string to interpret.

- [ ] Add `McpToolError` enum in `src/openhuman/mcp_client/` (or wherever the MCP tool dispatch lives):
  ```rust
  pub enum McpToolError {
      UnknownServer { server_name: String },       // server not in registry
      EndpointUnreachable { url: String, detail: String }, // DNS / TLS / connection refused
      EndpointReturned404 { url: String },         // wrong path
      AuthFailed { server: String, status: u16 },  // 401 / 403
      TransportMismatch { server: String, detail: String }, // HTTP vs SSE vs stdio mismatch
      ToolNotFound { server: String, tool_name: String },
      ServerError { server: String, status: u16, body_preview: String }, // 5xx
      Timeout { server: String, elapsed_ms: u64 },
      ProtocolError { server: String, detail: String }, // valid HTTP, bogus JSON-RPC
  }
  ```
- [ ] Where the MCP client surfaces a result, classify it into one of the variants (HTTP 404 → `EndpointReturned404 { url }`, connection refused → `EndpointUnreachable`, etc.).
- [ ] The `mcp_call_tool` agent tool's result shape returns the structured error in a stable JSON shape: `{ "kind": "endpoint_returned_404", "detail": "...", "human_readable": "Higgsfield MCP endpoint at https://mcp.higgsfield.ai returned 404. Try the path /mcp." }`.
- [ ] **Orchestrator prompt update** (`src/openhuman/agent/agents/orchestrator/prompt.md`) — add a section:
  > **MCP tool failures.** When an MCP tool call returns `{ "kind": "...", "human_readable": "..." }`, surface the `human_readable` verbatim. NEVER invent HTTP status codes, error messages, or root causes. If you're not sure what an MCP error means, say so plainly and tell the user to inspect their server's config.
- [ ] Tests: each enum variant has a unit test mapping a representative MCP client failure to the correct `McpToolError`. The `human_readable` rendering is pinned per variant.

#### Part 2 — Endpoint auto-probe on MCP add

The pain: every MCP server uses a different path; users have no way to know.

- [ ] In `connections::ops::add_mcp_server`, BEFORE save:
  1. Probe a sequence: user-provided endpoint, `<base>/mcp`, `<base>/sse`, `<base>/messages`, `<base>/` (in that order; stop on first success).
  2. "Success" = HTTP 200 OR 202 from `POST <path>` with a valid JSON-RPC `initialize` response (parse the body, look for `result.protocolVersion`).
  3. Use a 5s per-path timeout; bound total probe budget at 15s.
  4. If user-provided endpoint already includes a path (`/foo`), still probe it first but don't auto-append `/mcp` etc.; the user is clearly being explicit.
  5. If a probe succeeds, persist the corrected endpoint. Log `[mcp-probe] auto-corrected endpoint from <user-input> to <discovered>` in `RpcOutcome.logs` so the UI can render a "we auto-detected the right path" toast.
  6. If NO path responds, return a structured `McpProbeError { tried_paths: Vec<(String, ProbeOutcome)> }` so the user sees what was tried.
- [ ] **Frontend `McpAddModal.tsx`** renders the auto-correction notice when present: small inline "ℹ Auto-corrected endpoint to `https://mcp.higgsfield.ai/mcp`" below the URL input on save success.
- [ ] If the probe finds the server but auth fails (401), surface that clearly: "Endpoint at <path> exists but rejected your token — check the bearer value or generate a fresh one."
- [ ] Tests:
  - Probe sequence picks `/mcp` when the user supplies the root URL and `/mcp` is the one that responds.
  - Probe respects an explicit user-supplied path — doesn't auto-append.
  - All paths 404 → structured `McpProbeError` with the tried list.
  - Auth-failed probe surfaces a `401` not a generic 404.

#### Part 3 — Curated MCP catalog (STRETCH; defer if P1+P2 burn the budget)

- [ ] Bundle `src/openhuman/connections/data/featured_mcp_catalog.json` with ~10 servers:
  - Higgsfield (`https://mcp.higgsfield.ai/mcp`, bearer)
  - Linear (`https://mcp.linear.app/sse`, bearer)
  - GitBook (`https://tinyhumans.gitbook.io/openhuman/~gitbook/mcp`, bearer)
  - Sentry (`https://mcp.sentry.dev/mcp`, bearer)
  - Notion (`https://mcp.notion.com/mcp`, bearer)
  - …+ 4–5 more from the public MCP server list
- [ ] Frontend renders a "Featured MCP servers" tile row in `McpServersSection.tsx` above the [+ Add custom] tile. Click → pre-fills the McpAddModal with name + endpoint + auth-kind defaults; user only types the bearer token.
- [ ] Catalog entries carry `verified_at: <date>` so we can age out stale endpoints later.

## Architectural Decisions

### Why structured tool errors over LLM-prompted discipline?

We considered just adding "don't invent HTTP status codes" to the orchestrator prompt. The LLM ignores it under pressure. Structural fix means even a careless or model-swap-degraded orchestrator can't lie because the failure data has a deterministic shape that's rendered verbatim, not interpreted.

### Why auto-probe vs require explicit paths?

Requiring explicit paths is what we have today. It costs the user a `curl` excursion every time they add an MCP server because there's no standard path. Auto-probe pays a 5–15s probe cost on add (acceptable for a one-time setup) in exchange for never having to know the path. The probe IS the test — if no path responds, the user knows immediately rather than discovering later via tool failure.

### Why isn't this just "test on save"?

A pure "test on save" would test the user's exact URL and fail on 404 with a generic error. The user still has to guess paths. Auto-probe + test on save is the same code path — probe IS the test, with auto-correction.

### Why defer the catalog (Part 3)?

The acute pain is the wrong-path + confabulated-error pair. The catalog is great UX but not pain relief. Ship Parts 1+2 first to actually un-burn the user; catalog as polish.

### How does this interact with the F-18 orphan migration?

`migrate_mcp_orphan` calls into `add_mcp_server`, so Part 2's auto-probe also runs on migration. Side benefit: an orphaned MCP server with a wrong path gets auto-corrected when the user clicks Restore.

## Acceptance criteria

- Adding `https://mcp.higgsfield.ai` (root URL, no path) — auto-probes `/mcp`, succeeds, persists `https://mcp.higgsfield.ai/mcp`, UI shows "auto-corrected" notice.
- Adding `https://example.invalid/wrong` — probe fails with structured `McpProbeError`; UI shows the list of paths tried.
- An MCP tool call against a 404-returning endpoint surfaces the `EndpointReturned404` kind verbatim — chat agent does NOT invent "401" / generic errors.
- Confabulation regression: if the test harness injects a generic MCP failure, the orchestrator's response to the user MUST contain the `human_readable` substring from the structured error — not a paraphrase, not a LLM-generated guess. Pin via an integration test.
- `cargo test --lib connections::ops_tests::mcp_probe` ≥4 tests green.
- `cargo test --lib mcp_client::tests::mcp_tool_error_classification` ≥7 tests green (one per `McpToolError` variant).
