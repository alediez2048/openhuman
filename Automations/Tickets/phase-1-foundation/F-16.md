# F-16 Primer: Enforce `NodeAgentDefinition.allowed_tools` in workflow execution + honest step status

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Date:** 2026-05-22
**Previous work:** F-1..F-15 + Phase 1.5 polish landed on `main`. Chat-driven create → preview → Save → list-row works end-to-end (verified live 2026-05-21). Manual Run now dispatches, executor fires, run terminates `Succeeded` — but **no actual external action happens** on the Slack half. F-16 closes the gap.

---

## What Is This Ticket?

Fix the runtime executor so an `agent_prompt` node ACTUALLY runs inside the constrained sub-agent its [`NodeAgentDefinition`] declares — instead of secretly running inside the orchestrator with the full delegation tree.

Three coupled defects, one root cause:

1. **The allowlist is fiction.** `executor::run_agent_prompt` builds `def: NodeAgentDefinition { allowed_tools: [...] }` (per ADR-016), logs `allowed_tools=11`, then calls `Agent::from_config(config)?` — the default **orchestrator** — and never applies `def`. The `def` parameter is read once for the log line and discarded.
2. **Identity is wrong.** Because the agent runs as orchestrator, its system prompt says "delegate to specialised sub-agents for external services". So when the workflow's `agent_prompt.prompt` says "send a Slack message", the LLM picks `delegate_to_integrations_agent` — a tool that's not in `def.allowed_tools` and shouldn't be reachable. The `composio_execute` tool (which IS in `def.allowed_tools`) goes unused.
3. **Step status lies.** `RunStep` records `Succeeded` whenever `agent.run_single` returns non-empty text. A run where the agent says "I tried to send to Slack but the tool wasn't available" counts as Succeeded — same as a run where the Slack message actually went out. Run history is unreliable.

Symptom that surfaced this (logs, 2026-05-21 22:13–22:16):

```
22:13:11 workflows-run dispatch_run wf=... run=...
22:13:11 workflows-run run_agent_prompt invoking agent ... allowed_tools=11
22:13:18 [agent] delegating to integrations_agent via delegate_to_integrations_agent (skill_filter=gmail)
22:13:20 [subagent_runner] tool not in allowlist ... tool=GMAIL_GET_MESSAGES
22:13:31 [agent] delegating to integrations_agent ... (skill_filter=slack)
22:13:32 [subagent_runner:typed] fuzzy filter thin; falling back to full toolkit toolkit=slack total=16 filter_hits=2
22:13:35 [subagent_runner] tool not in allowlist ... tool=slack_slack_chat_postMessage
22:13:44 workflows-run step terminal ... node=n1 status=Succeeded     ← LIE
22:13:44 workflows-run run terminal ... status=Succeeded               ← LIE
```

`allowed_tools=11` was logged but `delegate_to_integrations_agent` (not in the 11) was called. That tool isn't blocked. The orchestrator's tool surface is what's actually visible. ADR-016 isn't enforced.

### Why It Matters

- ADR-016 is the documented Phase 1 security boundary for the agent_prompt sub-agent. As long as `run_agent_prompt` ignores `def`, ADR-016 is unenforced. Any future "agent_prompt with restricted scope" workflow (e.g. read-only summary nodes, no-network nodes) can't be trusted — the constraint just doesn't apply.
- The Phase 1 acceptance criteria includes "Composio-routed workflows run end-to-end". Tonight's live test proves they don't — Gmail half worked (by accident, via lazy_resolver inside integrations_agent), Slack half didn't. `composio_execute` is the deliberate per-connection tool; the executor should be making it the obvious choice for the LLM, not hiding it behind orchestrator routing.
- Honest `RunStep.status` is the foundation for everything downstream — run-history detail view, alert/retry policies in Phase 2 (F2-7..F2-10), debugging when a real user reports "the workflow ran but nothing happened".

- `composio_execute` — already a registered tool. The `connection_tool_name` mapping at `executor.rs:164` already returns `"composio_execute"` for `ConnectionRef::Composio`.

## What This Ticket Must Accomplish

### Goal

1. **A new agent definition `workflow_node`** lives in `src/openhuman/agent/agents/workflow_node/`, registered in `loader.rs::BUILTINS`. It has:
   - `omit_identity = true` (no orchestrator persona)
   - `omit_safety_preamble = false` (still inherits global safety rules)
   - `omit_memory_context = true` (workflow nodes don't get the user's chat memory — they get the prompt verbatim)
   - `omit_skills_catalog = true`
   - `temperature` = 0.3 (deterministic-leaning; the prompt is the spec)
   - `[tools].named = [<empty>]` — the BASE allowlist is empty; the **dynamic per-run allowlist override** is what supplies the real tools.
   - `[model].hint = "agentic"`
   - System prompt body (`prompt.md`) tells the agent: "You are an automated workflow step. Follow the prompt below verbatim. Use ONLY the tools listed in your tool surface. Do NOT delegate, do NOT ask the user clarifying questions, do NOT echo the prompt. When done, emit a one-line summary of what you did."

2. **A per-instance allowlist override on the `Agent` builder.** Either:
   - **Option A (lighter):** New builder `Agent::from_config_for_agent_with_tool_override(config, agent_id, allowed_tools)` that overrides `definition.tools` to `ToolScope::Named(allowed_tools)` before constructing. Implementation: clone the definition, mutate `.tools`, call `build_session_agent_inner` with the override. Tests: assert the override wins over the TOML.
   - **Option B (heavier, deferred):** Make `ToolScope::Named` lazy-resolved from a closure passed to the agent builder. Reject for Phase 1 — Option A is simpler and matches existing patterns.
   Choose A.

3. **`executor::run_agent_prompt` rewritten** to actually use `def`:
   ```rust
   let mut agent = Agent::from_config_for_agent_with_tool_override(
       config,
       "workflow_node",
       def.allowed_tools.clone(),  // already built per ADR-016
   )?;
   agent.set_event_context(format!("workflow:{run_id}"), "workflow");
   // Honour model_tier when set — Composio-only workflows might prefer fast,
   // multi-tool ones might prefer reasoning. Until per-tier override lands,
   // log when model_tier is set and ignored.
   let text = agent.run_single(&agent_prompt_config.prompt).await?;
   ```
   The orchestrator path (`Agent::from_config(config)`) is removed from this function entirely. `def: &NodeAgentDefinition` is now load-bearing.

4. **Honest step status.** `RunStep.status` becomes a function of "did the agent loop terminate cleanly AND were there no `tool not in allowlist` rejections AND no `tool execution returned is_error=true` results". Three options for how to measure this:
   - **Sub-option a:** Add an `event_bus` subscriber on `[subagent_runner] tool not in allowlist` warnings keyed by `workflow:<run_id>` event_context, mark step `Failed` with `error_summary = "agent tried to call tool '{}' which wasn't in the allowlist"` when fired during a run.
   - **Sub-option b:** Have the agent harness return structured tool-call telemetry (count of denied calls, count of `is_error=true` calls) from `run_single`. Bigger surface change.
   - **Sub-option c:** Force the LLM to terminate with a sentinel structured output like `{"status":"ok"|"failed", "summary":"..."}` that the executor parses.
   Choose **(a)** for Phase 1 — minimal harness change, leverages existing event bus + `event_context` tagging. (b) is the right long-term shape and lands in Phase 2 as part of multi-node orchestration.

## Architectural Decisions

### Why a dedicated `workflow_node` agent definition (not just `from_config_for_agent_with_tool_override(..., "code_executor", ...)`)?

- Identity should be honest: a workflow_node is NOT a code executor, NOT an orchestrator, NOT a planner. It's "a constrained automated step that follows the prompt and uses exactly these tools". The prompt body, `omit_*` flags, and temperature should reflect that — and they don't on any existing agent.
- Registry-resolved identity means future tooling (telemetry filters, prompt audits, sub-agent allowlist checks) can pattern-match on `agent_id="workflow_node"` cleanly.

### Why per-instance allowlist override (not just hard-coding the full set in `workflow_node/agent.toml`)?

- The allowed tools are a function of the workflow's `allowed_connections`, which is per-workflow. A static TOML allowlist can't capture this — it would either (a) list every connection-resolved tool (defeating the security boundary) or (b) be empty (and the executor would always need an override anyway).
- Per-instance override is what the harness already supports via `ToolScope::Named`; F-16 just exposes it on the builder so the executor can hand it the list it computed.

### Why event-bus-based step status (not a return-value contract on `run_single`)?

- Smaller surface change. The agent harness already publishes the relevant warnings (`tool not in allowlist`, `tool finish success=false`). The event bus already supports `event_context` tagging — we set `"workflow:<run_id>"` on the agent and filter on it. No `run_single` signature change required.
- Phase 2 will need richer tool telemetry anyway (for retry policies, partial-success counting in multi-node chains). Whatever shape that takes can replace the event-tap with no change to F-16's external contract.

### Why not fix integrations_agent's Composio Slack tool naming?

- That's a separate bug class (toolkit-action-name discovery for the LLM, lives in `agent/harness/subagent_runner/ops.rs`). It also affects non-workflow chat ("Send a Slack message to X" from the regular chat surface). Fixing it tonight wouldn't help workflows because the workflow shouldn't be going through integrations_agent at all — that's the whole point of F-16.
- The integrations_agent Slack-naming bug is real and worth its own ticket. **Not F-16's scope.**
