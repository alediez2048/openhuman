# F-12 Primer: Propose-Only Agent Tools (`workflow_propose_*`)

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Date:** 2026-05-19
**Previous work:** Phase 0 complete; F-11 (`Automations/Tickets/phase-1-foundation/F-11.md`) shipped `proposer::draft_with_retries` + the validator. F-10 shipped the four read-only agent tools and the allowlist test (which today asserts zero propose tools).

---

## What Is This Ticket?

Add the six propose-only agent tools to `src/openhuman/workflows/agent_tools.rs`:
- `workflow_propose_create(description)` → returns a `WorkflowProposal`. Calls `proposer::draft_with_retries`.
- `workflow_propose_update(workflow_id, instructions)` → returns a `WorkflowEditProposal` with `diff_summary`.
- `workflow_propose_delete(workflow_id)` → returns a `WorkflowDeletePreview`.
- `workflow_propose_enable(workflow_id)` / `_disable(workflow_id)` → returns a `WorkflowStateProposal` for the enable/disable action.
- `workflow_propose_run_now(workflow_id)` → returns a `WorkflowStateProposal` with `action = RunNow`.

**All six RETURN PROPOSAL PAYLOADS — NO MUTATION.** Per ADR-012, the agent literally has zero mutating tools. The UI button click on the preview component (F-14) is the only thing that calls the mutating RPCs.

Update F-10's `agent_tools_tests.rs` negative-assertion test to allow the six propose tools while still forbidding any mutating tool.

### Why It Matters

- This ticket closes the agent-side of the chat-driven creation flow. After F-12, the chat agent can call `workflow_propose_create` and get back a payload to render. F-14 renders that payload as `<WorkflowProposalPreview>`; F-13 ships the system prompt that teaches the chat agent how to call this tool.
- Confirms ADR-012's structural-security claim: a unit test in this ticket asserts that every workflow-domain agent tool is read or propose — zero mutations.
- Locks the propose-tool payload shape, which both the agent runtime and the UI component (F-14) consume.

## What This Ticket Must Accomplish

### Goal

- Six new pub-fn handlers in `agent_tools.rs`, all registered via `register_all`:
    - `handle_workflow_propose_create(ctx, args { description }) -> serde_json::Value` — calls `proposer::draft_with_retries(description, snapshot, current_phase, max_attempts=3)`. Returns the `WorkflowProposal` JSON. On `DraftFailure`, returns a structured `{ "error": …, "kind_label": … }` payload so the chat agent can surface it gracefully.
    - `handle_workflow_propose_update(ctx, args { workflow_id, instructions })`:
        1. Fetches the current `Workflow`.
        2. Calls `proposer::draft_with_retries_for_update(ctx, instructions, current_workflow, snapshot, phase, max_attempts)` — a new variant in `proposer.rs` that runs the drafting sub-agent with the current workflow JSON inlined into the system prompt and the user's instruction as the description.
        3. Diffs `current` vs `proposed` to produce `diff_summary: Vec<String>` (bullet list of changes; see "diff format" below).
        4. Returns a `WorkflowEditProposal { workflow_id, current, proposed, diff_summary, rationale }`.
    - `handle_workflow_propose_delete(ctx, args { workflow_id })`:
        1. Fetches the current `Workflow`.
        2. Queries `workflow_runs` count via `ops::count_runs(workflow_id)`.
        3. Returns `WorkflowDeletePreview { workflow_id, name, run_count, retention_days: 30 }`.
    - `handle_workflow_propose_enable(ctx, args { workflow_id })` → `WorkflowStateProposal { workflow_id, action: Enable, rationale }`. The `rationale` is generated server-side (e.g., "Currently disabled. Enabling will resume cron firing."). No LLM call needed for this one.
    - `handle_workflow_propose_disable(ctx, args { workflow_id })` → `WorkflowStateProposal { action: Disable, rationale }`.
    - `handle_workflow_propose_run_now(ctx, args { workflow_id })`:
        1. Fetches workflow + asserts `health == Ready`.
        2. Returns `WorkflowStateProposal { action: RunNow, rationale: ["Estimated time: <duration estimate from history>"] }`. (Use the median of the last 5 successful runs as the estimate; fall back to "Unknown" if none.)
        3. If `health != Ready`, return a payload with `rationale = ["Cannot run: <missing connections>"]` and an `enabled: false` flag the UI can use to disable the button.
- Update `register_all` in `agent_tools.rs` to register all 10 tools (4 read + 6 propose).
- Update the diff-summary helper for `_propose_update`:
    - Bullet-by-bullet description of changes between `current` and `proposed`. Examples:
        - "Renamed from `Founder morning digest` to `Founder daily brief`."
        - "Changed cron schedule from `0 8 * * 1-5` to `0 9 * * 1-5`."
        - "Added `slack` to allowed connections on step 1."
    - Implement in `src/openhuman/workflows/diff.rs` (new file): `pub fn workflow_diff(current: &Workflow, proposed: &Workflow) -> Vec<String>`.
- Update F-10's allowlist tests:
    - `agent_tools_tests.rs`: positive assertions for all 10 names (4 read + 6 propose). Negative assertion: zero mutating names — *exact list of forbidden names* (`workflows_create`, `workflows_update`, …) unchanged.
    - `executor_tests.rs`'s `build_node_agent_definition` test: zero `workflow_propose_*` names appear (the propose tools must NOT leak into `agent_prompt` node sub-agents). This is the critical post-F-12 invariant.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| Six propose tools, zero mutating tools | ADR-012 | Single security boundary: UI click triggers mutation |
| Propose-update uses a sibling drafting fn with current workflow inlined | This ticket | Reusing `draft_with_retries` would lose context |
| Diff is server-side, returned as `Vec<String>` bullets | systemsdesign §4.3 | UI doesn't recompute; the diff is the same regardless of who looks at it |
| `_propose_enable/disable/run_now` skip the LLM (static rationale) | This ticket | Cheap, deterministic, low-latency for state toggles |
| Propose tools must NOT leak into `agent_prompt` sub-agent | ADR-016 | Regression-tested in `executor_tests.rs` |
| `retention_days: 30` hard-coded in delete preview | FR-1.3.4 | Lean default per OQ-5 |

## Suggested Implementation Pattern

**`src/openhuman/workflows/agent_tools.rs` (extract — new handlers):**

```rust
const TOOL_PROPOSE_CREATE:   &str = "workflow_propose_create";
const TOOL_PROPOSE_UPDATE:   &str = "workflow_propose_update";
const TOOL_PROPOSE_DELETE:   &str = "workflow_propose_delete";
const TOOL_PROPOSE_ENABLE:   &str = "workflow_propose_enable";
const TOOL_PROPOSE_DISABLE:  &str = "workflow_propose_disable";
const TOOL_PROPOSE_RUN_NOW:  &str = "workflow_propose_run_now";

pub fn propose_tool_names() -> &'static [&'static str] {
    &[
        TOOL_PROPOSE_CREATE,
        TOOL_PROPOSE_UPDATE,
        TOOL_PROPOSE_DELETE,
        TOOL_PROPOSE_ENABLE,
        TOOL_PROPOSE_DISABLE,
        TOOL_PROPOSE_RUN_NOW,
    ]
}

pub async fn handle_workflow_propose_create(
    ctx: &Ctx,
    args: WorkflowProposeCreateArgs,
) -> serde_json::Value {
    tracing::info!(target: "workflows-agent",
        "tool=workflow_propose_create desc_len={}", args.description.len());
    let snapshot = ctx.connections_aggregator().list().await.unwrap_or_default();
    match proposer::draft_with_retries(ctx, &args.description, &snapshot, 1, 3).await {
        Ok(proposal) => serde_json::to_value(proposal).unwrap(),
        Err(DraftFailure::ValidationFailedAfterRetries { attempts, last_error }) => {
            serde_json::json!({
                "error": "validation_failed_after_retries",
                "attempts": attempts,
                "kind_label": last_error.kind_label(),
                "details": format!("{:?}", last_error),
            })
        }
        Err(DraftFailure::RunFailure { reason }) => {
            serde_json::json!({ "error": "drafting_failed", "reason": reason })
        }
    }
}
```

**`src/openhuman/workflows/diff.rs` (extract):**

```rust
pub fn workflow_diff(current: &Workflow, proposed: &Workflow) -> Vec<String> {
    let mut bullets = Vec::new();
    if current.name != proposed.name {
        bullets.push(format!("Renamed from “{}” to “{}”.", current.name, proposed.name));
    }
    if let (Trigger::Cron { expr: a, .. }, Trigger::Cron { expr: b, .. })
        = (&current.trigger, &proposed.trigger)
    {
        if a != b {
            bullets.push(format!("Changed cron schedule from `{a}` to `{b}`."));
        }
    }
    // ... node-level diffs, settings diffs ...
    bullets
}
```

## Edge Cases to Handle

1. **`workflow_propose_create` with empty description.** Return an error payload (`{ "error": "empty_description" }`); the agent should ask the user for more detail.
2. **`workflow_propose_update` on a deleted workflow.** Return `{ "error": "not_found" }`.
3. **`workflow_propose_delete` with zero runs.** `run_count = 0` — fine; UI still shows the preview.
4. **`workflow_propose_run_now` on a `Manual` workflow.** Same payload shape as cron — `RunNow` is valid for both. Test.
5. **Diff for identical workflows** (the agent's proposal didn't actually change anything). `workflow_diff` returns `[]`; the agent should detect this and tell the user "no changes needed." Handle by surfacing an empty `diff_summary` and a `rationale = ["No changes to apply."]` so the UI can decide whether to render or hide the preview.
6. **`draft_with_retries_for_update` produces an invalid edit.** Same retry budget as create (3 attempts). Document this in `proposer.rs`.
7. **Propose tools in the `build_node_agent_definition` allowlist.** Must NOT appear. The regression test catches this.
8. **Estimating run-now duration with no historical runs.** Return rationale `"Estimated time: unknown (no past runs)."`.
9. **Large diffs** (workflow with many node changes). Cap `diff_summary` at 20 bullets; if more, append `"... and N more changes."`.
10. **Concurrent edits.** F-12 doesn't add ETags; if the user clicks Apply on a stale preview, `workflows_update` succeeds with the proposed value (last-writer-wins). Phase 1 acceptable.

### Phase 0 deferred follow-up that may bite this ticket

- **Generic HTTP "saved connection" agent tool is NOT shipped.** Phase 0 left this in the STATE.md deferred-follow-ups list: the agent has raw `http_request` but no awareness of saved `GenericHttp` connections by id. If F-12's `workflow_propose_create` proposes a workflow whose `agent_prompt.allowed_connections` includes `GenericHttp { id: ... }`, the resulting sub-agent at run time (F-8) will have no per-id tool to call — it will fall back to raw `http_request`. **Acceptable for Phase 1** (the validator + executor still gate the connection's existence via `health == Ready`, and the secret stays server-side via `secret_ref` resolution in the HTTP probe path). If Phase 2 needs a richer `http_call(connection_id=…)` agent tool, it should be a P2 follow-up — not F-12's scope. Document this trade-off in the F-12 commit body so reviewers don't push back.
