## What Is This Ticket?

Fill the empty `src/openhuman/workflows/scheduler.rs` stub with cron trigger registration (via the existing `cron::JobType::WorkflowTrigger { workflow_id }` soft-id reference per ADR-003) and manual trigger handling (new `workflows_run_now` RPC). Boot-time scan registers every enabled cron-triggered workflow; `workflows_create` / `_update` / `_enable` / `_disable` keep the cron registration in sync.

**Not in this ticket:** the actual execution body (F-8 fills `executor.rs`). This ticket calls `executor::dispatch_run` but `dispatch_run` itself can be a stub returning `Ok(run_id)` without doing any work yet — F-8 makes it real.

### Why It Matters

- The scheduler is the link between user-configured triggers and the executor. Without it, no cron-triggered workflow ever fires automatically; users would have to click [Run now] every time.
- The cron-domain reuse decision (per `techstack.md §4`'s internal decision) is locked here: F-7 uses `cron::JobType::WorkflowTrigger` rather than running a sibling scheduler loop. This means `workflows.db` and `cron.db` stay separate (ADR-003) but the trigger registration ergonomics are shared.
- Sets up the `workflows_run_now` RPC that the overflow menu (F-4), the catalog [Add & Enable] flow (F-6), and the chat-driven proposals (F-14) all eventually call.

## What This Ticket Must Accomplish

### Goal

- New `cron::JobType` variant: `WorkflowTrigger { workflow_id: WorkflowId }`. (One-line extension of `src/openhuman/cron/types.rs`.)
- Cron-domain dispatch: when the cron scheduler fires a `WorkflowTrigger` job, it routes the trigger to `workflows::executor::dispatch_run(workflow_id, TriggerSource::Cron)`. Use the existing event-bus pattern (`request_native_global` or `publish_global`) — pick the one that matches how cron currently dispatches its other job types and document the choice.
- `workflows::scheduler::register(workflow)` — given a `Workflow` with `Trigger::Cron`, registers it with the cron domain. No-op for `Trigger::Manual`.
- `workflows::scheduler::deregister(workflow_id)` — removes the registration. Called on disable, delete, and when a trigger config changes.
- `workflows::scheduler::reconcile_at_startup()` — boot-time scan that:
    1. Queries all workflows where `enabled = true` AND `trigger.type = cron`.
    2. Registers each via the cron domain.
    3. Logs the count: `[workflows] reconciled N cron-triggered workflows at startup`.
- `workflows_run_now(workflow_id)` RPC — calls `executor::dispatch_run(workflow_id, TriggerSource::Manual { initiator: …, … })` synchronously to the run-row-creation point only; the actual execution runs async (FR-1.6.7). Returns the `RunId` immediately. Refuses when `health != Ready` (returns `RpcOutcome::Err { code: "health_blocked" }`).
- F-2 wiring update: `workflows_create` calls `scheduler::register` after persistence; `workflows_update` calls `deregister` then `register` (only if `enabled && trigger.is_cron()`); `workflows_enable` calls `register`; `workflows_disable` calls `deregister`; `workflows_delete` calls `deregister` before the cascade delete.

## Suggested Implementation Pattern

**`src/openhuman/workflows/scheduler.rs` (extract):**

```rust
pub async fn register(ctx: &Ctx, workflow: &Workflow) -> Result<()> {
    if !workflow.enabled {
        return Ok(());
    }
    let Trigger::Cron { expr, tz, .. } = &workflow.trigger else {
        return Ok(());
    };
    tracing::info!(
        target: "workflows-scheduler",
        "register wf={} cron='{}' tz={:?}", workflow.id, expr, tz
    );
    cron::ops::add_job(
        ctx,
        cron::CronJob {
            job_type: cron::JobType::WorkflowTrigger { workflow_id: workflow.id.clone() },
            schedule: cron::Schedule::Expression { expr: expr.clone(), tz: tz.clone() },
            // … other CronJob fields
        },
    ).await
}

pub async fn handle_run_now(
    ctx: &Ctx,
    workflow_id: WorkflowId,
    initiator: ManualInitiator,
) -> Result<RunId, RunNowError> {
    let workflow = ctx.workflows_store().get_workflow(&workflow_id)?
        .ok_or(RunNowError::NotFound)?;
    if !matches!(workflow.health, WorkflowHealth::Ready) {
        return Err(RunNowError::HealthBlocked { health: workflow.health });
    }
    executor::dispatch_run(ctx, workflow_id, TriggerSource::Manual { initiator })
        .await
        .map_err(RunNowError::Dispatch)
}
```

**`src/openhuman/cron/scheduler.rs` dispatch (extract):**

```rust
match job.job_type {
```

## Edge Cases to Handle

1. **Cron expression edited via `workflows_update`.** Old job stays in `cron.db` if you only call `register` (which is a no-op when a duplicate exists in some impls). Always `deregister` first, then `register`.
2. **Workflow deleted while a cron tick was already in flight.** Cron's tick handler reads `workflow_id`; the `executor::dispatch_run` should handle the "workflow not found" case gracefully (return Ok, log `[workflows] cron tick for deleted workflow ignored`).
3. **`reconcile_at_startup` runs before cron's own startup.** Order matters — wire it explicitly AFTER cron initialization. Test by setting `OPENHUMAN_WORKSPACE` to a directory with a pre-existing workflows row and observing the startup log.
4. **`workflows_run_now` on a manual-triggered workflow.** Same code path; the trigger-type doesn't matter for run-now. Test this.
5. **`workflows_run_now` on a disabled workflow.** Per FR-1.4.1.2, Manual triggers fire regardless of `enabled`. Re-read that FR carefully — the gate is `health == Ready`, not `enabled`. Implement accordingly.
6. **Cron registration races with health changes.** F-3's subscriber doesn't currently call `scheduler::register` on health transitions; the next cron tick observes the latest workflow state when it fires. Document this — eventual consistency is fine.
7. **TimeZone handling.** Phase 1 cron with `tz = null` defaults to UTC per FR-1.4.1.1. Pass through to the cron crate; do not interpret here.
8. **Active-hours window** (`Trigger::Cron { active_hours }`). Phase 1 stub: store the field, don't gate dispatch on it yet — leave a TODO referencing a Phase 2 enhancement.
9. **Deduplication of job-types in `cron.db`.** `cron::add_job` may already dedupe by some key; if it doesn't, document that `register` is idempotent in the workflows-side rustdoc.
