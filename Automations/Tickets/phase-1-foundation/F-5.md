## What Is This Ticket?

Bundle the Phase 1 starter templates (RU-1 through RU-4) as JSON files under `src/openhuman/workflows/templates/`, embed them via `include_str!`, and expose them through a stateless `workflows_list_starter_templates` RPC that filters by `min_phase` and dedupes against the user's existing `Workflow.origin = Seed { template_id }` rows.

Per ADR-008, the catalog is **read-only at runtime** — not auto-inserted into the user's workflows table. Per ADR-004, templates ship as in-repo JSON via `include_str!` (OQ-8).

### Why It Matters

- The catalog is the empty-state's secondary path (FR-1.2.6) — without it, the only way to get a workflow into the list is the chat-driven hero flow (F-12..F-14), which is much heavier to build. F-5 lands a working alternative on its own.
- Locks the template-JSON shape that the chat-driven proposer (F-11) and the catalog UI (F-6) both consume — both code paths use the same `WorkflowProposal`-shaped JSON.
- Validates F-1's `WorkflowOrigin::Seed { template_id }` discriminator end-to-end: the catalog dedup query (FR-1.8.2) is the first real consumer of the `template_id` payload.

## Architectural Decisions

| Decision | Reference | Summary |
|---|---|---|
| Templates ship in-repo as JSON via `include_str!` | ADR-004 | No backend fetch; ships with the binary |
| Catalog is read-only at runtime, dedup via `origin.Seed.template_id` | ADR-008 / ADR-018 | Each [Add] click promotes a template into the user's table |
| `StarterTemplateView` includes `missing_connections` server-side | This ticket | UI doesn't recompute; the connections snapshot is server-authoritative |
| RU-2's `await_human_approval` step is downgraded to "queue in Telegram" for Phase 1 | This ticket | Phase 3 adds the real node kind; documented in template `rationale_at_seed` |
| `min_phase` filter is server-side, not client | systemsdesign §5.2 | Avoids leaking Phase 2 templates into the UI before their node kinds work |

## Suggested Implementation Pattern

**`src/openhuman/workflows/templates/mod.rs`:**

```rust
pub const RU_1_JSON: &str = include_str!("ru-1-founder-morning-digest.json");
pub const RU_2_JSON: &str = include_str!("ru-2-linkedin-engagement-queue.json");
pub const RU_3_JSON: &str = include_str!("ru-3-spotify-friday-five.json");
pub const RU_4_JSON: &str = include_str!("ru-4-jira-sprint-retro.json");

pub fn all_bundled() -> Vec<StarterTemplate> {
    [RU_1_JSON, RU_2_JSON, RU_3_JSON, RU_4_JSON]
        .iter()
        .filter_map(|raw| {
            serde_json::from_str::<StarterTemplate>(raw)
                .map_err(|e| {
                    tracing::error!(target: "workflows", "starter template parse failed: {e}");
                    e
                })
                .ok()
        })
        .collect()
}
```

**`src/openhuman/workflows/ops.rs` — `list_starter_templates`:**

```rust
pub async fn list_starter_templates(
    ctx: &Ctx,
    phase: u32,
) -> RpcOutcome<Vec<StarterTemplateView>> {
    let bundled = templates::all_bundled();
    let user_seeded: HashSet<String> = ctx
        .workflows_store()
        .list_seed_origins()?
        .into_iter()
        .collect();
    let connections = ctx.connections_aggregator().list().await?;

    let views: Vec<_> = bundled
        .into_iter()
        .filter(|t| t.min_phase <= phase)
        .filter(|t| !user_seeded.contains(&t.template_id))
        .map(|t| StarterTemplateView::from_template(t, &connections))
        .collect();

    RpcOutcome::ok(views)
}
```

## Edge Cases to Handle

1. **Malformed template JSON.** Boot the binary anyway; log a clear error. `all_bundled()` skips broken files. The unit test ensures all four parse — so production never sees a malformed file.
2. **Template references a `ConnectionRef` shape that doesn't exist** (e.g., an extra discriminator variant added in Phase 2 by mistake). Serde's typed parsing rejects it at boot; the test catches it.
3. **User has already added all 4 templates.** Catalog returns `[]`; UI (F-6) renders an empty catalog with copy "All starter workflows added — you can still build more in chat."
4. **`phase` argument from the RPC.** Read the current phase from the about_app capability catalog or a const (F-15 ensures this is exposed). For F-5, hard-code `current_phase = 1` and add a TODO referencing F-15.
5. **Cron expression in a template is invalid.** The parse test in `templates_tests.rs` fails — fix before merge. The validator (F-11) catches the same in chat-driven flows.
6. **Phase 1 RU-2 (no `await_human_approval`).** Document the Phase 1 degradation in the template's `rationale_at_seed` so the user understands.
7. **`template_id` collision with a future template added by the same release.** `templates_tests.rs` asserts every `template_id` is unique across `all_bundled()`.
8. **Missing-connection labels in `StarterTemplateView`.** Pull human labels from `ConnectionRef::display_label()` (defined in `connections/types.rs` via P0-2/P0-3) — never the raw enum string.
9. **`rationale_at_seed` field — does the runtime use it?** Phase 1 uses it as the `rationale` field when the catalog promotes a template into a workflow; it's identical content. Document this mapping.
