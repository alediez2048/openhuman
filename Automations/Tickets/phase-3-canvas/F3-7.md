# F3-7 Primer: `await_human_approval` Node Kind + Approval UI

**For:** New coding-agent session
**Project:** OpenHuman — Workflows & Automations
**Previous work:** F2-7 ships the persistent-resume executor pattern for delays. F3-7 reuses it for approvals.

---

## What Is This Ticket?

Pauses a run waiting for the user to approve or reject. Resume infra mirrors F2-7's `delay` node — persisted state, no in-memory await. Per OQ-19 the user gets both an OS notification and an in-app banner.

### Why It Matters

- The forcing function for the "draft a tweet, let me approve, then post" pattern called out in `prd.md`. Without `await_human_approval`, every approval flow has to bounce through chat — slower and easier to lose.

---

## What This Ticket Must Accomplish

### Goal

- `AwaitHumanApprovalConfig`:
    ```rust
    pub struct AwaitHumanApprovalConfig {
        pub prompt_label: String,        // "Approve tweet?"
        pub timeout_secs: u32,           // default 3600; max 7 days
        pub timeout_action: TimeoutAction, // ApproveImplicit | RejectImplicit | Halt
    }
    ```
- Executor body:
    1. Persist `workflow_runs.status = AwaitingApproval` + `awaiting_step_id`.
    2. Release the in-flight slot.
    3. Publish `WorkflowAwaitingApproval { run_id, node_id, prompt_label }`.
- UI surfaces (frontend):
    - OS native notification via the existing `native_notifications` Tauri command.
    - In-app banner across the app top showing all pending approvals (new `<PendingApprovalsBanner>` component).
    - Click → modal with [Approve] / [Reject] buttons + the prompt context.
- New RPCs `workflows_approve_run(run_id)` + `workflows_reject_run(run_id)`:
    - Validates the run is `AwaitingApproval`.
    - Persists the decision in a new `workflow_run_approvals` table.
    - Re-dispatches the run from the next node (Approve) or marks it Cancelled (Reject).
- Resume sweep (extend F2-7's): runs whose `awaiting_until <= now` honor `timeout_action`.
